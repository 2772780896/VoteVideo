## 一、起点：移动端浏览器宿主选型

大厂阉割了手机端 Chrome 的插件生态。为了夺回控制权，横向评测后选定 **Via 浏览器**：

| 宿主 | 评价 |
|------|------|
| Chrome 手机版 | 官方禁止扩展，出局 |
| Edge 手机版 | 支持精选扩展，但限制冷门插件 |
| Kiwi / 狐猴 | 完整 Chromium 内核，兼容 Chrome 商店 |
| **Via（胜出）** | < 2MB，系统 WebView 秒开，无广告，支持油猴脚本注入 |

Via 就像拆光座椅的超轻量赛车，将所有主导权交还用户。通过注入沉浸式翻译油猴脚本，验证了客户端代码本地接管网页的可行性。

## 二、架构演进：四个阶段的技术博弈

### 2.1 阶段一：如何建立用户诉求与 CSS 修改的桥梁？

**直觉猜想**：在 Prompt 里塞满 CSS 样式，或定义"CSS 类名与功能对照字典"。

**证伪**：死胡同。面对千变万化的类名和口语需求，字典无法穷尽，且会榨干 AI 上下文窗口。

**架构突破**：让 AI 扮演现场写代码的"前端翻译官"：

```
脚本捕获用户选中的局部 HTML
    → 局部 HTML + 用户口语需求 发给 AI
    → AI 现场输出纯 CSS（带 !important）
    → 脚本动态创建 <style> 标签注入网页
```

### 2.2 阶段二：数据持久化与渲染闪烁（FOUC）

**直觉猜想**：缓存到 localStorage，提高优先级。

**纠正**：localStorage 本身没有执行优先级。

**架构突破**：利用脚本生命周期"抢跑"——设置 `@run-at document-start`：

```
浏览器解析 HTML 前 → 脚本从 localStorage 读出 AI CSS
    → 注入到 <html> 根节点
    → 后续 DOM 渲染时，AI 样式已在内存中 + !important → 无闪烁
```

### 2.3 阶段三：Tailwind CSS 与盲猜问题

**Tailwind 破解**：AI 不需要懂 Tailwind，利用 DOM 阶层关系定位，输出原生纯 CSS，优先级吊打 Tailwind。

**盲猜破解（Element Picker）**：拒绝盲猜，引导用户显式选择目标元素：

```js
// 捕获阶段拦截 mouseover → 动态红框反馈
document.addEventListener('mouseover', highlight, { capture: true });

// 捕获阶段拦截 click → 锁定元素，提取 outerHTML
document.addEventListener('click', (e) => {
    e.preventDefault();  // 锁死页面跳转
    const html = e.target.outerHTML;  // 精准提取
    // 发送给 AI...
}, { capture: true });
```

### 2.4 阶段四：全局模糊路径修改的稳定性灾难

**宏大设想**：选完并保存后，跳转到同站点其他页面时自动识别并应用修改——"一句话颠覆全站样式"。

**方案推演**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 全局 CSS 变量劫持 (`:root`) | 极轻量 | 只对使用了全局变量的网站有效 |
| XPath/CSS 绝对路径 | 不需 AI，纯前端算法 | React/Vue 动态路由下 DOM 微调即断裂，可拓展性毁灭 |
| 视觉智能体大模型 | 彻底抛弃 DOM，向量匹配 | 手机端算力/延迟完全不切实际 |

## 三、终极决断：回归"当前页面拾取"

放弃稳定性极差的"全站多页面盲匹配"，回归确定性最高的方案：

```
[用户跨页面跳转] → [悬浮球始终置顶跟随] → [点击悬浮球唤醒画笔]
                                                    │
[注入当前页面] ← [AI 现场生成纯 CSS] ← [输入口语需求] ← [点击红框锁定 DOM]
     │
[LocalStorage 缓存 (URL 为 Key)] → [下次打开该 URL 时 document-start 抢跑]
```

**为什么这是最优解**：
- **Bug 归零**：只对眼前锁定的 DOM 节点负责
- **极简**：几十行原生 JS 事件监听，不引入笨重模型
- **掌控感**：所见即所得，用户亲手"指"给 AI 看

## 四、跨标签页通信与分布式状态

### 4.1 核心概念厘清

- **SPA 本质**：跳转 = 内存中的组件替换，不白屏，同一标签页共享连续内存
- **标签页隔离**：新建标签页 = 全新进程 + 全新内存，与原标签页无任何关联
- **全局状态边界**：Vuex/Redux/Pinia 只统治单标签页，跨标签页瞬间蒸发

### 4.2 全局状态 vs LocalStorage 的辩证

| | 全局状态（State） | LocalStorage |
|---|---|---|
| 响应式 | 自动触发视图刷新 | 不触发刷新，需手动监听 |
| 速度 | 纳秒级（RAM） | 毫秒级（ROM），高频操作卡顿 |
| 跨标签 | 不能 | 能（同源共享） |
| 持久化 | 不能（刷新即丢） | 能 |

**结论**：不是二选一，而是各司其职。

### 4.3 终极武器库：四种通信介质

| 武器 | 战场 | 介质 | 特点 |
|------|------|------|------|
| 全局状态 | 单标签页内 | 内存 | 极速 + 响应式，但不过标签页 |
| BroadcastChannel | 多标签页间 | 浏览器内部通道 | 内存级速度，需手写发送/监听 |
| LocalStorage | 跨标签/跨时间 | 闪存 | 自带广播 + 持久化，但慢且无响应式 |
| WebSocket | 跨设备 | TCP/IP | 万物互联，但耗电耗带宽 |

### 4.4 悬浮球 AI 拾取器的跨标签联动

```
【标签页 A：AI 对话窗】                        【标签页 B：浏览网页】
用户点击"画笔"                                       监听中
    │                                                   │
    └── BroadcastChannel 广播 ──────────────────→ 收到信号
                                                    │
                                            isPicking = true（全局状态）
                                                    │
                                            界面觉醒红框拾取
                                                    │
                                            用户点击目标组件
                                                    │
    ←────── 发送组件 HTML ←──────────────── 锁定拦截并打包

对话窗不刷新，自动装填 DOM → 发送给 AI → 生成 CSS → 注入标签页 B
```
