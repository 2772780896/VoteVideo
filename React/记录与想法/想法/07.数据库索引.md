# 数据库索引原理与设计 — 学习笔记

> B+ 树聚簇索引与复合索引的物理结构、最左前缀原则、索引字段选择（区分度/类型/长度）、写入开销、user_like 表设计实战、数据库设计决策树。

## 关键词
数据库, 索引, B+树, 聚簇索引, 复合索引, 最左前缀, 区分度, 回表, filesort, user_like

---

## 一、索引的物理本质

索引是**一棵独立的 B+ 树**，与主表数据分离存储。

主表数据按聚簇索引（主键 ID）组织成 B+ 树。复合索引是另一棵独立的 B+ 树，维护另一套排序体系。

**两棵树互不干扰，共享主键 ID 作为回表桥梁。**

---

## 二、聚簇索引（Clustered Index）

- 主表数据按主键 ID 的顺序排列叶子节点，形成一棵 B+ 树
- 叶子节点直接存储完整行数据
- 每张表只有**一棵**聚簇索引树

```sql
CREATE TABLE user_like (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,  -- 这就是聚簇索引的排序字段
  userId BIGINT NOT NULL,
  ...
);
```

---

## 三、复合索引（Composite Index）

```sql
CREATE INDEX idx_like_user_target ON user_like (userId, targetType, targetId);
```

这棵 B+ 树的排序规则：**先按 userId 排序，userId 相同则按 targetType 排序，targetType 相同则按 targetId 排序**。

叶子节点存储的不是完整行数据，而是：

```
[userId, targetType, targetId, 主键ID]
```

查询时通过主键 ID 回表（Bookmark Lookup）获取完整数据。

---

## 四、最左前缀原则（Leftmost Prefix）

复合索引 `(A, B, C)` 支持的查询组合：

| 查询条件 | 是否走索引 |
|---------|-----------|
| `WHERE A = x` | ✅ |
| `WHERE A = x AND B = y` | ✅ |
| `WHERE A = x AND B = y AND C = z` | ✅ |
| `WHERE B = y` | ❌（跳过 A，无法导航） |
| `WHERE B = y AND C = z` | ❌ |

字段顺序决定查询边界，不能随意调换。

---

## 五、为什么字段顺序不能乱放

复合索引的 B+ 树按顺序分层组织：

```
第一层：按 userId 分组（所有 userId=5 的记录在一起）
第二层：在 userId=5 的组内，按 targetType 分组
第三层：在 targetType='video' 的组内，按 targetId 排序
```

如果查询跳过了第一层，数据库无从下手，只能全表扫描。

---

## 六、索引的存储开销

假设 `user_like` 表 1000 万行：

- 索引叶子节点只存 `userId + targetType + targetId + 主键ID`（4 个 BIGINT ≈ 32 字节）
- 完整行数据可能有数十个字段
- 复合索引体积 ≈ 表体积的 **30%~50%**

**不能给所有字段建索引**，空间和写入成本会翻倍。

---

## 七、写入时的索引维护

每插入一行数据：

```sql
INSERT INTO user_like (userId, targetType, targetId, createdAt) VALUES (5, 'video', 123, NOW());
```

数据库执行：
1. 将完整行数据插入聚簇索引树 ✅
2. 将 `(userId, targetType, targetId, 主键ID)` 插入复合索引树 ✅

**写入变慢的原因**：多一次 B+ 树插入操作。索引越多，写入成本越高。

---

## 八、复合索引字段选择原则

### 区分度（Selectivity）

区分度 = 不同值数量 / 总行数。**越高越好**。

- `userId` 区分度极高（用户量大）
- `targetType` 只有 4 个值，区分度低
- `userId + targetType + targetId` 组合后几乎唯一

### 字段类型与长度

- 优先用 INT/BIGINT 而非 VARCHAR（比较快，索引体积小）
- 尽量用固定长度类型

### 与查询条件的匹配

高频查询 `WHERE userId = ? AND targetType = ?` 决定了 `(userId, targetType, targetId)` 的顺序。

---

## 九、createdAt 为何不加入复合索引

查询场景通常不需要按 `createdAt` 过滤条件本身，而是用于**排序**：

```sql
ORDER BY createdAt DESC  -- 文件排序（filesort），不回表时用不上索引
```

如果 `createdAt` 需要作为范围查询条件（如"查询最近7天的点赞"），可以加在索引末尾，但要注意范围查询会**阻断后续字段的索引命中**：

```sql
INDEX (userId, targetType, createdAt)
WHERE userId = 5 AND targetType = 'video' AND createdAt > '2025-01-01'
-- createdAt 是范围查询，targetId 无法利用索引
```

## 十、user_like 表设计

```sql
CREATE TABLE user_like (
  id          BIGINT PRIMARY KEY AUTO_INCREMENT,
  userId      BIGINT NOT NULL,
  targetType  VARCHAR(20) NOT NULL,
  targetId    BIGINT NOT NULL,
  createdAt   DATETIME NOT NULL,

  INDEX idx_like_user_target (userId, targetType, targetId)
);
```

**为什么这样设计**：

- `userId` 在前 → 查"某用户的所有点赞"
- `targetType` 其次 → 同一用户内按类型分组
- `targetId` 最后 → 精准定位具体对象
- `createdAt` 不进索引 → 它只用于排序，单独用 filesort 处理

---

## 十一、数据库设计决策树

```
需要建索引吗？
├── 查询频率高吗？否 → 不建
├── 字段区分度低（< 10%）？是 → 不建（收益小）
├── 是高频查询的 WHERE 条件？是 → 考虑加入
│   └── 是范围查询？是 → 放索引末尾
└── 是 ORDER BY 专用？是 → 不进索引，用 filesort
```
