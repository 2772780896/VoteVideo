# Storybook — 学习笔记

> 组件视觉开发工具（不是测试框架）：语法与工作原理、文件约定、Controls 面板；VoteVideo 项目实战经验（Main 页面故事因 useData 耦合导致代码重复的解决方案）。

## 关键词
Storybook, stories, args, argTypes, Controls, 视觉测试, 组件隔离, decorators, mockData

---

## 一、它算什么？

Storybook **不是测试框架**，但它是测试流程的**前置环节**：

```
开发阶段              测试阶段              发布阶段
┌──────────┐    ┌──────────────┐    ┌──────────────┐
│ Storybook │ →  │ 单元测试 Jest │ →  │ E2E Playwright│
│ 视觉开发   │    │ 逻辑正确性    │    │ 用户操作流程   │
└──────────┘    └──────────────┘    └──────────────┘
```

**一句话**：Storybook 管"长得好不好看"，Jest 管"算得对不对"，Playwright 管"用户能不能走通"。

## 二、实际开发中的样子

### 典型工作流

```
① 设计师给 Figma 设计稿
② 前端在 Storybook 里写组件，对照设计稿调样式
③ 故事就是活的文档，PM/设计师打开就能验收
④ 改业务逻辑时，先在 Storybook 改组件，确认没问题再嵌入页面
⑤ CI 跑 Storybook 构建 + Chromatic 视觉回归测试
```

### 个人项目 vs 正规军

| | 个人项目 | 正规项目 |
|---|---|---|
| 调 VideoCard 样式 | 进首页 → 等请求 → 滚动到 → 看效果 | 打开 Storybook → 点 VideoCard → 秒看 |
| 检查超长标题 | 去数据库改一条数据 | 点 LongTitle 故事 |
| 改了组件怕影响其他页面 | 手动进 10 个页面各看一眼 | Storybook 一条龙看完所有状态 |
| 组件文档 | 没有 | Storybook 本身就是活文档 |

## 三、语法和工作原理

### 3.1 文件约定

`.storybook/main.js` 中有扫描规则：

```js
stories: ['../src/**/*.stories.@(js|jsx|ts|tsx)']
```

创建 `*.stories.jsx` 文件后，Storybook 启动时自动扫描。

**动哪些文件**：

| 文件 | 频率 | 内容 |
|------|------|------|
| `.storybook/main.js` | 一次性 | 扫描规则、框架、插件 |
| `.storybook/preview.js` | 一次性 | 全局装饰器、import CSS |
| `package.json` | 加一行 | `"storybook": "storybook dev -p 6006"` |
| `XXX.stories.jsx` | **每次新建** | 每个组件的测试故事 |

故事文件和组件文件放同一目录：

```
src/components/common/DataCard/VideoCard.jsx
src/components/common/DataCard/VideoCard.stories.jsx
```

### 3.2 故事文件结构

```jsx
import VideoCard from './VideoCard'

// ① 默认导出：组件元信息
export default {
    title: 'Common/VideoCard',    // 侧栏分层路径
    component: VideoCard,         // 要渲染的组件
}

// ② 命名导出：每个 = 一个故事 = 侧栏一个可点击项
export const Default = {
    args: {
        video: {
            vid: 1,
            coverUrl: 'https://picsum.photos/300/170',
            title: '正常标题',
            viewCount: '12.3万',
            messageCount: '456',
            duration: '10:30',
            uploader: { userName: '张三' },
            date: '2024-01-01',
        },
    },
}

export const LongTitle = {
    args: {
        video: {
            ...Default.args.video,     // 复用基础数据
            title: '超长标题'.repeat(20),
        },
    },
}

export const NoUploader = {
    args: {
        video: {
            ...Default.args.video,
            uploader: null,            // 测试可选链 ?.
        },
    },
}
```

### 3.3 工作原理

```
① npm run storybook → 读 .storybook/main.js
② 扫描所有 *.stories.jsx → 解析 export default / export const
③ export default 的 title → 生成侧栏目录树
④ export const 的名字 → 生成侧栏可点击项
⑤ 点击故事 → React.createElement(Component, args)
⑥ 组件在隔离的 iframe 中渲染
⑦ 切换故事 → args 变了 → React 重新渲染
```

**核心**：`args` 对象直接变成组件的 props。Storybook 只是传话筒。

### 3.4 Controls 面板

```jsx
export default {
    title: 'Common/VideoCard',
    component: VideoCard,
    argTypes: {
        title: { control: 'text' },          // 文本框
        viewCount: { control: 'text' },      // 文本框
        uploader: { control: 'object' },     // JSON 编辑器
    },
}
```

界面底部出现 Controls 面板，可拖滑块、改文字，组件实时更新。

TypeScript 项目可自动从 `interface` 生成 Controls。

## 四、VoteVideo 实战：Main 页面故事

### 4.1 做了什么

侧栏出现两套故事：

```
Components/VideoCard
  ├── Default / LongTitle / NoUploader / ShortTitle / LargeNumbers
  │   来自 VideoCard.stories.jsx，测单个组件

Main/完整页面
  ├── 正常 / 加载中 / 加载失败 / 视频列表为空 / 轮播正常_视频加载中
  │   来自 Main.stories.jsx，测完整页面布局
```

### 4.2 为什么 Main.stories.jsx 有大段重复代码

**因为 Main.jsx 的渲染逻辑嵌在 `useData` 调用里，不从外部接收 props。**

Storybook 要模拟"页面 loading 中"就必须传 `carouselLoading=true`，但 HomePage 不接收这个 prop——它内部自己调用 `useData` 拿到的。

**两种解法**：

| 解法 | 做法 | 代价 |
|------|------|------|
| A. 改架构 | 拆 Container + View，View 接收 props | 改了源码 ❌ |
| B. 复刻 | 故事文件里写结构一样的 `MainDemo` | 代码重复 ⚠ |

选了 B（不碰源码优先），代价是 80 行重复 JSX。

### 4.3 这现实吗？

**不现实，正规军不会这么写。** 改了 Main.jsx 布局，故事里的 MainDemo 不会自动更新。

**实际项目的成熟方案**：

**方案 1：Mock 整个 Hook**（不改架构，不重复代码）

```jsx
import HomePage from './Main'
import * as useDataModule from '@/hooks/useData'

export const 加载中 = {
    parameters: {
        mockData: [
            [useDataModule, { data: null, loading: true, error: null }],
        ],
    },
    render: () => <HomePage />,  // 直接用真组件
}
```

需要 `storybook-addon-mock` 或 `msw-storybook-addon`。

**方案 2：组件天然接收 props**（架构本身就支持）

如果组件设计成接收 `carouselLoading`、`carouselError` 等状态，Storybook 直接传就行。这需要架构意识，不是为 Storybook 改，而是架构本身就该这么做。

## 五、设计思路总结

> 传统开发：组件嵌在页面里调 → 改一次看一次，10 个状态要手动进 10 个页面
>
> Storybook：把组件"升维"——脱离页面独立渲染，每种 props 组合就是一个"故事"，一键切换

**本质**：把 UI 组件当成纯函数来测试——给定 props → 渲染结果 → 视觉校验。跟 `expect(add(1,2)).toBe(3)` 是同一个思路，只是结果不是数字，是像素。
