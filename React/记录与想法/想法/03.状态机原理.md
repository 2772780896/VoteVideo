# 状态机原理 — 学习笔记

> 有限状态机（FSM）的本质：用"硬性规则"组成的逻辑控制器，消灭布尔地狱；声明式 vs 隐式两种实现方案的对比；Promise/Redux 的状态机本质；影响前端代码设计的算法思维。

## 关键词
状态机, FSM, 布尔地狱, 声明式, 隐式, Promise, Redux, 查表法, 哈希表, 空间换时间, TypeScript, XState

---

## 一、为什么需要状态机？

假设一个 AI 口语模考页面有 4 个状态：空闲、录音中、AI 思考中、播放反馈。

**布尔地狱（不用状态机）**：

```js
// 一堆布尔值互相打架
if (!isThinking && !isPlaying && !isRecording) {
    isRecording = true;
    startRecording();
}
```

功能变多后，布尔值互相冲突（不小心同时把 `isRecording` 和 `isThinking` 都设为 `true`），页面卡死变 Bug。

**状态机（优雅逻辑）**：

```
[ 空闲 ] ──(点击录音)──► [ 录音中 ] ──(松开)──► [ AI思考中 ] ──(收到结果)──► [ 播放反馈 ]
   ▲                                                                            │
   └───────────────────────────(播放完毕)──────────────────────────────────────┘
```

当前处于 [AI思考中] 时，用户把录音按钮点烂，状态机也直接无视——路线图上没有这条线。

## 二、两种实现方案

### 2.1 隐式状态机（逻辑断言式）— 推荐日常使用

零配置，用 `||` 直接锁死合法路径：

```ts
type State = 'IDLE' | 'RECORDING' | 'THINKING' | 'PLAYING';

class SimpleMachine {
    private current: State = 'IDLE';

    public transitionTo(next: State): void {
        const isValid =
            (this.current === 'IDLE'      && next === 'RECORDING') ||
            (this.current === 'RECORDING' && next === 'THINKING')  ||
            (this.current === 'THINKING'  && next === 'PLAYING')   ||
            (this.current === 'PLAYING'   && next === 'IDLE');

        if (isValid) {
            this.current = next;
        } else {
            console.warn(`非法操作：不能从 ${this.current} 跳到 ${next}`);
        }
    }

    public get is() { return this.current; }
}
```

使用：

```ts
const m = new SimpleMachine();
m.transitionTo('RECORDING'); // ✅ 成功
m.transitionTo('RECORDING'); // ❌ 拦截！不能跳到自身
m.transitionTo('THINKING');  // ✅ 成功

// TS 自动提示 4 个状态供选择，打错字直接编译报错
```

**优点**：零认知开销、极速落地、改动直观（加一行 `||` 即可）。

**缺点**：状态超过 7 个时 `isValid` 变成 `||` 墙；缺乏副作用绑定（状态改变后自动触发函数）。

### 2.2 声明式状态机（配置化）— 大型项目专用

用 JSON 对象定义所有规则，"查表"定位下一步：

```ts
const config = {
    IDLE:      { on: { CLICK_RECORD:   'RECORDING' } },
    RECORDING: { on: { RELEASE_BUTTON: 'THINKING'  } },
    THINKING:  { on: { AI_SUCCESS:     'PLAYING'   } },
    PLAYING:   { on: { AUDIO_END:      'IDLE'      } }
};

// 核心调用就一行——不需要 if-else，对象属性查找 O(1)
const nextState = config[currentState].on[event];
```

**调用方式**：`machine.send('CLICK_RECORD')`——传的是事件（动作），不是目的地。

**设计原因**：
- **嵌套对象结构** = 哈希表，计算机两次属性查找即得结果（空间换时间）
- **send 事件而非直接改状态** = 安全隔离，业务代码只管汇报"用户干了什么"，规则集中管理

**优点**：确定性 100%、适合团队协作、支持副作用绑定（状态切换时自动触发函数）。

**缺点**：初始配置繁琐，TS 类型约束（映射类型 `[S in ExamState]`）理解成本高。

### 2.3 方案对比

| 维度 | 隐式（逻辑断言） | 声明式（配置化） |
|------|-----------------|-----------------|
| 设计思维 | 关卡保安：符合规矩就放行 | 地图思维：先画完整路线图 |
| 实现方式 | 代码驱动（if/`||`） | 数据驱动（JSON 查表） |
| 适合场景 | 中小型业务、3-7 个状态 | 大型低代码画布、音视频编辑器 |
| 代表库 | 无（手写） | XState |

**选择建议**：90% 前端需求用隐式就够，不必上 XState 重型武器。

## 三、状态机在前端的无处不在

### 3.1 Promise = 三状态微型状态机

```
PENDING ──► FULFILLED   (不可逆)
PENDING ──► REJECTED    (不可逆)
```

`.then(callback)` = 在状态机上注册副作用，状态变为 FULFILLED 时自动执行。

### 3.2 Redux = 全局状态机

| 状态机概念 | Redux 术语 |
|-----------|-----------|
| State（状态） | `store` 里的 JSON 树 |
| Event（事件） | `dispatch({ type: 'ADD_TODO' })` |
| Transition（规则） | Reducer 的 `switch-case` |

```js
// Reducer 本质就是状态机的路径规则
function todoReducer(state, action) {
    switch (action.type) {
        case 'FETCH_START':   return { status: 'LOADING' };
        case 'FETCH_SUCCESS': return { status: 'SUCCESS', data: action.payload };
        default:              return state;
    }
}
```

### 3.3 数据驱动 vs 动作驱动

| | 动作驱动（jQuery 时代） | 数据/状态驱动（React/Redux） |
|---|---|---|
| 思维 | "用户点了按钮" → 改页面 → 发请求 | "用户点了按钮" → 改状态 → UI 自动响应 |
| 问题 | 代码散落各角落，易冲突 | 单一数据源，单向数据流 |

## 四、查表法：消灭 if-else 的算法思维

声明式状态机的嵌套对象结构，本质是**哈希表 / 查找表**——用空间换时间。

**垃圾代码（if-else 遍历查找）**：

```js
function getRoleName(roleType) {
    if (roleType === 'admin')   return '超级管理员';
    if (roleType === 'editor')  return '文章编辑';
    if (roleType === 'visitor') return '访客';
    return '未知角色';
}
```

**查表法（O(1) 定位）**：

```js
const ROLE_MAP = {
    admin: '超级管理员',
    editor: '文章编辑',
    visitor: '访客'
};

function getRoleName(roleType) {
    return ROLE_MAP[roleType] || '未知角色';
}
```

不管字典有 3 个还是 3 万个角色，查找速度一样（O(1)）。这就是项目里常见的 `SUBTYPE_MAP`、`ACTION_MAP` 等映射表的算法根基。

## 五、影响前端代码设计的四大算法思维

### 5.1 缓存思维（空间换时间）

算过的结果存起来，下次直接用，绝不重复计算。

**前端应用**：`useMemo`、`useCallback`、Vue `computed`。

### 5.2 双指针与快慢指针

两个指针各司其职，一次遍历完成两件事。

**前端应用**：虚拟列表（`startIndex` / `endIndex` 只渲染可视区域）、防抖节流。

### 5.3 分治法与递归

大问题切成相同的小问题，递归解决。

**前端应用**：虚拟 DOM Diff（树的 DFS/BFS）、Vite/Webpack 依赖图遍历。

### 5.4 状态机与不可变数据

逻辑抽象为状态和事件，且不允许直接修改原数据。

**前端应用**：React `setState`、Redux（`oldState !== newState` 引用对比决定是否刷新）。

### 5.5 补充：动态规划

把大问题拆成阶段性小问题，记住每阶段最优解。

**前端应用**：Diff 算法（最长公共子序列 LCS），求解树节点移动最短路径。

### 5.6 补充：拓扑排序

"必须 A 完成 B 才能开始"的先后顺序链。

**前端应用**：Vite/Webpack 编译依赖图、pnpm/npm 包安装顺序。

### 5.7 补充：四叉树 / 八叉树

空间分割，只计算当前可见区域。

**前端应用**：3D 地图、可视化大屏（高德地图放大/缩小/拖拽时毫秒级响应）。
