## 一、为什么必须写单元测试？

手写 `console.log()` 手动测试的五大灾难：

| 灾难 | 说明 |
|------|------|
| 环境污染 | 临时测试代码混入业务，上线前删、下次改又写回来 |
| 肉眼极限 | 1000 个测试案例靠盯控制台，极易看漏 |
| 连带崩溃 | 改底层函数 A，天边模块 B 隐蔽塌方，手动无法穷举 |
| 无法融入 CI/CD | 远程服务器没有显示器，机器看不懂 print |
| 无法统计覆盖率 | 无法评估有没有漏掉 if-else 分支 |

---

## 二、测试库的本质

测试库（Vitest/Jest/pytest）不知道你的业务逻辑，只扮演两个角色：
- **自动化跑腿**：拿测试单元，传参给业务代码运行
- **自动化对账**：`expect` 断言比对"实际结果"和"预期结果"，一致绿灯，不一致红灯

---

## 三、单元测试该测什么

| 方向 | 说明 |
|------|------|
| 快乐路径 | 正常情况，核心功能通畅 |
| 边界条件 | 悬崖边缘（如 17/18/19 岁），90% 的 Bug 在此 |
| 异常处理 | 输入错误 / 环境崩塌，系统优雅报错而非闪退 |

### 防漏兜底机制

- **代码覆盖率**：工具统计哪些行被测试跑过，大厂硬性要求 80%~90%
- **等价类划分法**：按数学分类（有效/负数/超范围/非数字），每类挑一个代表
- **TDD（测试驱动开发）**：先写测试（全红），再写业务代码直到全绿

---

## 四、Mock（模拟）

**核心原则**：单元测试只测你写的逻辑本身，绝不连接外部网络和真实数据库。

**本质**：在测试环境中，把外部库（如 axios）原地调包成"假方法"，不联网，直接秒回假数据，将被测函数与外部世界隔离。

---

## 五、Vitest / Jest 实战规范

### 5.1 文件命名

- 测试文件：`*.test.js` 或 `*.spec.js`（如 `user.test.js`）
- 推荐**镜像并列**：测试代码和业务代码放同一目录

```
src/utils/
├── math.js          ← 业务代码
└── math.test.js     ← 测试代码（紧挨着）
```

### 5.2 基本语法

```js
import { test, expect } from 'vitest'
import { add } from './math'

test('add(1, 2) 应该返回 3', () => {
    expect(add(1, 2)).toBe(3)
})
```

### 5.3 常用断言

| 断言 | 含义 |
|------|------|
| `expect(x).toBe(y)` | 严格相等 `===` |
| `expect(x).toEqual(obj)` | 深度相等（对象/数组） |
| `expect(arr).toContain(item)` | 数组包含 |
| `expect(fn).toThrow()` | 函数抛异常 |
| `expect(promise).rejects.toThrow()` | 异步 reject |

### 5.4 Mock 实战（含异步）

```js
import { vi, test, expect } from 'vitest'
import axios from 'axios'
import { checkUserStatus } from './user'

vi.mock('axios')  // 拦截整个 axios 库

test('积分 150 → VIP', async () => {
    axios.get.mockResolvedValue({ data: { score: 150 } })  // 假数据秒回
    const result = await checkUserStatus('user_123')
    expect(result).toBe('VIP')
})
```

## 六、CI 完整流水线

```
push 代码
  │
  ├─ ① Format + Lint       1 min   静态检查（不运行代码）
  │     ├─ Prettier：格式（缩进、引号、分号）
  │     └─ ESLint：坑位（未使用变量、useEffect 缺依赖）
  │
  ├─ ② 类型检查             1 min   TypeScript 项目专用
  │     └─ tsc --noEmit
  │
  ├─ ③ 构建                 2 min   npm run build
  │
  ├─ ④ 单元测试（Jest）      3 min   逻辑对不对
  ├─ ⑤ 视觉测试（Storybook） 5 min   长相变没变
  ├─ ⑥ E2E（Playwright）    10 min  流程走不走得通
  │
  └─ ⑦ 部署
```

**原则**：越靠前越快、越靠后越接近用户视角。前置不通过就不跑后面。

### 五层防线总览

| 层级 | 工具 | 测什么 | 速度 | 个人项目 |
|------|------|--------|------|----------|
| ① Lint | ESLint + Prettier | 代码规范、潜在地雷 | 秒级 | 必须 |
| ② 类型 | TypeScript | 类型正确性 | 秒级 | 引入 TS 后配 |
| ③ 单元 | Jest/Vitest | 函数/组件逻辑 | 分钟 | 必须 |
| ④ 视觉 | Storybook + 截图 | 组件外观 | 分钟 | 推荐 |
| ⑤ E2E | Playwright | 核心业务链路 | 分钟 | 只写 1-3 条 |

## 七、Playwright — E2E 端到端测试

### 7.1 原理

通过 Chrome DevTools Protocol 远程控制真实 Chrome 浏览器：

```
Node.js 测试脚本 → WebSocket（CDP）→ 真实 Chrome → 执行操作 → 返回快照/截图
```

JS 模拟 DOM（Jest 用 jsdom）做不到的事——CSS 渲染、真实鼠标点击、视频录制——Playwright 全能做。

### 7.2 选择器

```js
page.locator('text=登录')                     // 文本
page.locator('.btn-submit')                   // CSS class
page.locator('#login-form')                   // id
page.locator('button[type="submit"]')         // HTML 属性
page.locator('role=button', { name: '登录' }) // 无障碍语义
page.locator('[data-testid="login-btn"]')     // 测试标记（最稳定）
```

**稳定度**：`test-id > role > CSS > text`

### 7.3 基本操作 + 断言

```js
const { test, expect } = require('@playwright/test')

test('用户搜索视频', async ({ page }) => {
    await page.goto('http://localhost:5173')
    await page.locator('input[placeholder="搜索"]').fill('React')
    await page.keyboard.press('Enter')

    await expect(page).toHaveURL(/\/search/)
    await expect(page.locator('text=搜索结果')).toBeVisible()
})
```

| 断言 | 含义 |
|------|------|
| `expect(el).toBeVisible()` | 元素可见 |
| `expect(el).toHaveText('xxx')` | 文字完全匹配 |
| `expect(el).toContainText('部分')` | 包含子串 |
| `expect(page).toHaveURL('/path')` | URL 正确 |
| `expect(el).toHaveScreenshot()` | 截图对比基准图 |

### 7.4 结果输出

```
终端：        ✓ 首页加载正常 (1.2s)    ✗ 搜索功能 (3.0s)
浏览器报告：  npx playwright show-report
失败截图：    test-results/ 目录
失败视频：    test-results/*.webm
```

## 八、Chromatic — 视觉回归测试（付费云）

Storybook 团队云服务，CI 里自动截图逐像素对比：

```
改 CSS → CI 构建 Storybook → Chromatic 截图 → 对比上次
  ├─ 没变化 → 通过
  └─ 有变化 → PR 标注差异，等人审批
```

免费额度每月 5000 次截图。替代方案：Storybook + Playwright 截图存 Git 仓库自行对比。

## 九、测试金字塔

```
          ╱╲
         ╱ E2E ╲          最少 ← 3-5 条核心链路
        ╱────────╲
       ╱ 集成测试  ╲        中等
      ╱────────────╲
     ╱   单元测试    ╲      最多 ← 每个函数都要测
    ╱────────────────╲
```

**规矩**：越底层写越多、跑越快；越上层写越少、跑越慢。

E2E 只测核心链路：
```
✅ 登录 → 首页加载 → 点视频 → 播放 → 评论
✅ 搜索 → 选标签 → 看结果 → 翻页
❌ 轮播图小圆点颜色（Storybook 看）
❌ VideoCard 标题折行（Storybook 看）
❌ 格式化函数（单元测试测）
```

## 十、AI 辅助测试的正确姿势

写测试单元中"调函数、手写传参、伪造 Mock 数据"是机械体力活，正是 AI 的主场。

**Prompt 模板**：

```
我是 JS 开发者，使用 Vitest 测试框架。
这是我刚写好的业务函数：
【贴代码】

请帮我编写 .test.js 文件：
1. 快乐路径测试
2. 至少 3 个边界值/极端情况
3. 有外部请求请用 vi.mock 拦截
```

## 十一、个人项目建议优先级

```
① ESLint + Prettier          零成本
② 单元测试 (Vitest/Jest)     收益最大
③ Storybook                  组件多了再装
④ Playwright E2E             项目稳定后写 1-3 条核心链路
⑤ TypeScript + Chromatic     锦上添花
```

**不要过早写 E2E**：代码没定型先写死测试 = UI 频繁改动 = 修测试比修业务代码还累。
