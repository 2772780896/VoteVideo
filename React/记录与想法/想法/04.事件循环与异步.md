## 一、Promise / async / await 底层模型

### 1.1 Promise 的本质

一个带两个口袋的占位盒子：
- **value 口袋**：装成功数据
- **error 口袋**：装失败报错

```js
const myBox = new Promise((resolve, reject) => {
    // 这里的代码是【绝对同步】当场执行的
    // resolve: 成功遥控器 → 往 value 口袋塞数据
    // reject:  失败遥控器 → 往 error 口袋塞报错
});
```

### 1.2 await 的本质

一个冷酷的"时间静止保安"：
- 面对 Promise 盒子时，如果两个口袋都空 → **冻结整个 async 函数**
- 直到某个口袋被塞进东西 → 掏出数据，解除冻结
- **限制**：只能管到它所在的 async 函数，塞进普通回调就失效

### 1.3 完整生命周期

```
① 当场造盒（同步）：new Promise 立即执行，去后台登记闹钟
② 保安站岗（异步死等）：await 发现口袋空，冻结流水线
③ 闹钟响起（状态变化）：resolve(数据) → 盒子状态变为成功
④ 微任务通知（收货放行）：状态改变触发微任务 → 保安收到数据 → 代码继续
```

## 二、try/catch 为什么抓不到 setTimeout 的错误？

```js
try {
    setTimeout(() => { throw new Error() }, 2000);
} catch (err) {
    // 永远不会执行到这里
}
```

**根因不是宏任务/微任务，而是时间差**：
1. `try` 块执行：告诉浏览器"2秒后把这个函数放进宏任务队列"
2. `try` 块执行完毕，从调用栈消失
3. 2秒后，宏任务被拿出来执行，触发报错
4. 此时 try/catch 早就走完了——"抓贼的保安已经下班了"

**用 Promise + await 修复**：

```js
const myBox = new Promise((resolve, reject) => {
    setTimeout(() => {
        reject("报错啦");  // 手动把错误塞进 error 口袋
    }, 2000);
});

try {
    await myBox;  // await 保安原地死等 2 秒
} catch (err) {
    // 2秒后，error 口袋被塞进东西 → await 引爆 → catch 接住
}
```

**原理**：不是靠宏/微任务的魔法，而是靠 Promise 盒子作为媒介，用 await 强行拉长了 try/catch 的值班寿命。

## 三、Axios 错误传递与拦截器吞错误

### 3.1 错误流转路线

```
浏览器网络组件报错
    → Axios 内部小盒子
    → 响应拦截器
    → 页面 await 面前
    → 抛出异常
    → catch(err) 捕获
```

### 3.2 拦截器"吞错误"大坑

**现象**：拦截器里写了 `alert(error)`，导致页面 try/catch 抓不到错误。

**原因**：拦截器处理完错误后，如果没有返回值，JS 默认返回 `undefined`（相当于成功）。Axios 据此制造"成功口袋装了空"的新盒子，错误被假装成功地消灭了。

**正确解法**：

```js
// 拦截器里处理完错误后，必须继续往上抛
return Promise.reject(error);  // 做个新盒子，把错误塞进 error 口袋
```

## 四、宏任务与微任务

### 4.1 它们到底是什么？

内存中两个普通的待办事项队列：

| 队列 | 来源 | 内容 |
|------|------|------|
| **宏任务队列** | 浏览器外部环境 | 定时器到期、鼠标点击、网络请求完毕 |
| **微任务队列** | JS 代码内部衍生 | Promise 状态变化后的通知 |

### 4.2 事件循环（Event Loop）铁律

```
① 执行当前同步代码
② 同步代码干完 → 必须优先清空微任务队列
③ 微任务清空 → 从宏任务队列拿【一个】出来执行
④ 执行完一个宏任务 → 再次检查微任务 → 循环
```

### 4.3 经典执行顺序

```js
setTimeout(() => { console.log("3. 宏任务") }, 0);

Promise.resolve().then(() => { console.log("2. 微任务") });

console.log("1. 同步代码");

// 输出顺序：1 → 2 → 3
```

**哪怕宏任务在代码里写在最前面，只要没有 await 拦截，执行顺序上它必须等微任务全部干完。**

### 4.4 微任务的无限插队权

微任务执行时如果又产生新微任务，继续插队：

```js
setTimeout(() => { console.log("宏任务") }, 0);

Promise.resolve()
    .then(() => {
        console.log("微任务1");
        Promise.resolve().then(() => { console.log("微任务2") });
    });

// 输出顺序：微任务1 → 微任务2 → 宏任务
```

**结论**：微任务只要不断流，宏任务就会被永远饿死。

### 4.5 微任务里包着宏任务

```js
Promise.resolve().then(() => {
    setTimeout(() => { console.log("3. 宏任务") }, 0);  // 只是去排队
    console.log("1. 微任务里的同步代码");                   // 立刻执行
});
console.log("2. 最外层同步代码");

// 输出顺序：2 → 1 → 3
```

微任务内部的同步代码一口气干完，里面的宏任务老老实实去最后面排队。

### 4.6 用 await 扭转顺序

```js
await new Promise((resolve) => {
    setTimeout(() => {
        console.log("1. 宏任务，被 await 保护");
        resolve();
    }, 2000);
});

Promise.resolve().then(() => {
    console.log("2. 微任务，今天没法插队了");
});

// await 冻结了 2 秒，后面的微任务连进队列的机会都没有
```

## 五、总结

| 场景 | 执行规则 |
|------|----------|
| 无 await | 同步 → 微任务无限插队 → 宏任务收尾 |
| 有 await | await 化身时间静止器，后面全部冻结直到 Promise 放行 |
| try/catch + setTimeout | 抓不到，因为 try 早就走完了（时间差问题） |
| try/catch + await Promise | 能抓到，await 拉长了 catch 的值班寿命 |
| Axios 拦截器错误 | 必须 `return Promise.reject(error)` 否则被吞 |
