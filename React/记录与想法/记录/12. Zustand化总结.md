# Zustand 化总结文档

## 一、需求背景

### 1. 原有问题

在引入 Zustand 之前，项目中的登录状态管理存在以下问题：

| 问题 | 说明 | 影响 |
|------|------|------|
| 状态分散 | `isLoggedIn` 在 `TopMenu` 本地，`LoginModal` 的 `open` 也在本地 | 无法跨组件控制 |
| 无法跨组件触发登录弹窗 | 其他页面（如点赞、评论）无法打开登录弹窗 | 用户体验差 |
| 登出需要刷新页面 | `UserMenu` 使用 `window.location.reload()` 重置状态 | 页面闪烁，体验差 |
| 重复读取 Cookie | 多个组件各自读取 `Cookies.get('token')` | 性能浪费 |

### 2. 目标

- ✅ 统一管理登录状态
- ✅ 任何组件都能打开登录弹窗
- ✅ 登出时不需要刷新页面
- ✅ 减少 Cookie 读取次数
- ✅ 应用启动时验证 token 有效性
- ✅ 为需要登录的路由添加路由守卫
- ✅ 未登录用户操作时展示登录框

---

## 二、方案选择

### 对比分析

| 方案 | 优点 | 缺点 | 适用性 |
|------|------|------|--------|
| **Context + useReducer** | 内置 API，无额外依赖 | Provider 嵌套，性能优化复杂 | 中小型项目 |
| **Zustand** | 轻量，API 简洁，无需 Provider | 第三方依赖 | 中小型项目（推荐） |
| **Redux Toolkit** | 生态丰富，DevTools 强大 | Boilerplate 多，学习成本高 | 大型项目 |
| **直接操作 Cookie** | 无额外依赖 | 需要手动触发事件，容易出错 | 不推荐 |

### 最终选择：Zustand

**选择理由**：
1. 轻量级（~1KB），适合本项目规模
2. API 简洁，学习成本低
3. 无需 Provider 包裹，使用灵活
4. 社区主流推荐

## 三、实现方案

### 1. 创建 Auth Store

**文件**：`src/stores/authStore.js`

**设计思路**：
- 将登录状态、用户信息、弹窗状态集中管理
- 提供统一的登录、登出、弹窗控制方法
- 在 store 内部处理 Cookie 的读写
- **新增**：应用启动时验证 token 有效性

**状态设计**：
```js
{
  isLoggedIn: boolean,      // 是否登录
  user: object | null,     // 用户信息（可选缓存）
  loginModalOpen: boolean   // 登录弹窗是否显示
}
```

**方法设计**：
```js
{
  login(token, uid, userInfo?),    // 登录成功后的处理
  logout(),                       // 退出登录
  openLoginModal(),               // 打开登录弹窗
  closeLoginModal(),              // 关闭登录弹窗
  initAuth(),                     // 初始化认证状态（验证 token）
  setUser(userInfo)               // 更新用户信息
}
```

**initAuth() 实现**：
- 检查 Cookie 中是否有 token
- 如果有，调 `getMyProfile()` 接口验证 token 是否有效
- 如果有效，设置 `isLoggedIn = true` 并缓存用户信息
- 如果无效，清除 Cookie 和状态

### 2. 修改组件

#### (1) `main.jsx` - 应用入口

**改动**：
- 引入 `useAuthStore`
- 在渲染前调用 `initAuth()` 初始化认证状态

**目的**：
- 确保应用启动时验证 token 有效性
- 如果 token 无效，自动退出登录

#### (2) `TopMenu.jsx` - 导航栏

**改动**：
- 移除：直接读取 `Cookies.get('token')`
- 移除：本地 `isLoggedIn` 状态
- 移除：`LoginModal` 的 `onLoginSuccess` 回调
- 新增：从 Zustand 获取 `isLoggedIn` 状态

**目的**：
- 使用全局状态控制登录/用户菜单的显示
- 简化组件逻辑

#### (3) `LoginModal.jsx` - 登录弹窗

**改动**：
- 移除：本地 `open` 状态
- 移除：`onLoginSuccess` 回调
- 新增：从 Zustand 获取 `loginModalOpen` 状态
- 新增：登录成功后调用 Zustand 的 `login()` 方法

**目的**：
- 弹窗状态全局可控
- 其他组件也能打开登录弹窗

#### (4) `UserMenu.jsx` - 用户菜单

**改动**：
- 移除：直接调用 `Cookies.remove()`
- 移除：`window.location.reload()`
- 新增：调用 Zustand 的 `logout()` 方法

**目的**：
- 退出登录时不需要刷新页面
- 状态更新后组件自动重新渲染

#### (5) `InteractionBar.jsx` - 交互操作条

**改动**：
- 新增：引入 Zustand，获取 `isLoggedIn` 和 `openLoginModal`
- 新增：`checkLogin()` 方法，在操作前检查登录状态
- 修改：`handleLike`、`handleFavourite`、`handleReshare` 在操作前调用 `checkLogin()`

**目的**：
- 未登录用户点击点赞/收藏/转发时，打开登录弹窗
- 已登录用户正常执行操作

#### (6) `User.jsx` - 用户主页

**改动**：
- 新增：引入 Zustand，获取 `isLoggedIn` 和 `openLoginModal`
- 新增：`checkLogin()` 方法，在操作前检查登录状态
- 修改：`handleFollow`、`handleMessage` 在操作前调用 `checkLogin()`

**目的**：
- 未登录用户点击关注/发消息时，打开登录弹窗
- 已登录用户正常执行操作

### 3. 创建路由守卫组件

**文件**：`src/components/common/RequireAuth.jsx`

**功能**：
- 检查用户是否登录
- 如果未登录，打开登录弹窗并重定向到首页
- 如果已登录，渲染受保护的组件

**使用方式**：
```jsx
<RequireAuth>
  <ProtectedComponent />
</RequireAuth>
```

### 4. 应用路由守卫

**文件**：`src/router/index.jsx`

**受保护的路由**：
- `/upload` - 上传页面
- `/user/profile` - 个人中心
- `/dynamic` - 动态页面

**实现方式**：
- 在路由配置中，用 `<RequireAuth>` 包裹需要保护的页面组件
- 未登录用户访问这些路由时，会自动打开登录弹窗并跳转到首页

## 四、使用说明

### 1. 在组件中判断登录状态

```js
import useAuthStore from '@/stores/authStore'

const MyComponent = () => {
  const isLoggedIn = useAuthStore(state => state.isLoggedIn)
  
  if (!isLoggedIn) {
    return <div>请先登录</div>
  }
  
  return <div>已登录内容</div>
}
```

### 2. 触发登录弹窗

```js
import useAuthStore from '@/stores/authStore'

const MyComponent = () => {
  const isLoggedIn = useAuthStore(state => state.isLoggedIn)
  const openLoginModal = useAuthStore(state => state.openLoginModal)

  const handleLike = () => {
    if (!isLoggedIn) {
      openLoginModal()  // 打开登录弹窗
      return
    }
    // 执行点赞逻辑
  }

  return <button onClick={handleLike}>点赞</button>
}
```

### 3. 退出登录

```js
import useAuthStore from '@/stores/authStore'

const UserMenu = () => {
  const logout = useAuthStore(state => state.logout)
  const navigate = useNavigate()

  const handleLogout = () => {
    logout()              // 清除 Cookie 和状态
    navigate('/')         // 跳转到首页
    // 不需要刷新页面
  }
}
```

### 4. 使用路由守卫

```jsx
// 在 router/index.jsx 中
{
  path: '/upload',
  element: (
    <RequireAuth>
      <UploadPage />
    </RequireAuth>
  ),
}
```

## 五、验证 token 有效性

### 实现方式

在 `authStore.js` 的 `initAuth()` 方法中，调接口验证 token 是否有效：

```js
initAuth: async () => {
  const token = Cookies.get('token')
  
  if (!token) {
    set({ isLoggedIn: false, user: null })
    return
  }

  try {
    // 调接口验证 token 是否有效
    const res = await getMyProfile()
    
    // 接口返回成功，token 有效
    set({ 
      isLoggedIn: true, 
      user: res.data 
    })
  } catch (err) {
    // token 无效或接口调用失败，清除登录状态
    Cookies.remove('token')
    Cookies.remove('uid')
    set({ isLoggedIn: false, user: null })
  }
}
```

### 验证接口

使用 `GET /api/user/profile/` 接口（需要认证）：
- 如果返回 200，说明 token 有效
- 如果返回 401，说明 token 无效，需要重新登录

## 七、待优化项

### 1. 其他需要登录状态的组件

以下组件可能还需要判断登录状态：

| 组件 | 场景 |
|------|------|
| `UploadVideo.jsx` 等 | 上传前判断（已通过路由守卫保护） |
| `Profile.jsx` | 访问个人中心前判断（已通过路由守卫保护） |
| `Video.jsx` / `Essay.jsx` / `Post.jsx` | 交互功能需登录（已通过 InteractionBar 保护） |
| `CommentCard.jsx` | 评论点赞/踩前判断 |

### 2. Token 过期处理

当前实现：
- 应用启动时验证 token 有效性
- 如果 token 过期，需要重新登录

建议优化：
- 在请求拦截器中统一处理 401 错误
- 自动打开登录弹窗，而不是跳转到首页

### 3. 用户信息缓存

当前实现：
- `initAuth()` 中会缓存用户信息到 Zustand
- 其他组件可以直接从 Zustand 获取用户信息

建议优化：
- 在 Zustand 中添加 `updateUser()` 方法
- 在用户修改资料后，更新 Zustand 中的用户信息
