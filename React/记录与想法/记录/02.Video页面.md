## 一、页面职责

| 任务 | 说明 |
|------|------|
| 视频播放 | 原生 `<video>` 标签渲染，封面图+视频流 |
| 标签展示 | 视频关联的标签，支持展开/折叠 |

接口：`fetchItem('video', vid)` 获取详情，`fetchItemList('comment', ...)` 获取评论，`fetchItemList('video', { subType: 'related', vid, ... })` 获取推荐。字段定义见 [前端接口分析.md](./13.%20VoteVideo-前端接口分析.md) 第二、四节。

## 二、开发历程

### 2.1 初始搭建：播放器 + 标签 + 评论区

从路由参数 `useParams()` 取 vid → `fetchItem('video', vid)` 请求详情 → 页面组件树搭建：

```
VideoPage（初始版）
├── TopMenuApp
├── VideoPlayerApp      — <video poster={coverUrl}><source src={videoUrl}/></video>
├── TagRow              — map TagPopover，支持展开/折叠
├── SortDropdown        — 排序下拉
└── CommentList         — CommentCard × N
```

**VideoPlayer 两个关键细节**：
- `aspectRatio: '16/9'` 保持宽高比
- `key={playVideo?.vid}` — vid 变化时销毁重建 `<video>`，解决切换视频后播放失败（旧播放器不会自动重新加载新 src）

**TagRow 用 useMemo 缓存**：`useMemo(() => tagList?.map(...), [tagList])`，tagList 不变时不重新创建 JSX 数组。

### 2.2 DataList 通用列表组件设计

项目中多处需要列表+分页+排序，于是抽象了 DataList：

```jsx
<DataList
  request={getCommentList}     // API 函数
  sort={sort}                  // 排序参数
  renderItem={(item) => <CommentCard key={item.cid} comment={item} />}
  pageSize={16}                // 每页条数
  gridClassName="flex flex-col gap-4"  // 默认竖向列表
/>
```

> DataList 的完整设计、Props 接口、骨架屏实现见 [通用组件与Hook.md](./10.通用组件与Hook.md) 第一节。

### 2.3 SortDropdown 受控组件模式

**放弃 Antd Dropdown 的原因**：只用了一个组件却引入整个 Antd 包，且 Antd 的 `open` 状态与 React 受控模式冲突。

**两个 sort 的职责分离**：

```
VideoPage                          SortDropdown
  │  const [sort, setSort]         │  const [sort, setSort]
  │       ↑                        │       ↑
  │  传给评论区重新请求              │  用在 Dropdown 显示 label
  │                                │
  │  <SortDropdown pushSort={setSort} />
  └──────── pushSort() ←──────────┘
```

子组件内部 sort 通过 `useEffect(() => pushSort(sort), [sort])` 单向同步给父组件。两个 sort 互不干扰——SortDropdown 不知道父组件拿 sort 干嘛，VideoPage 不知道内部怎么存的。

**纯 Tailwind 实现要点**：`useRef` + `mousedown` 点击外部关闭（展开时才注册监听防泄漏）、`transition-transform rotate-180` 三角旋转动画、选中后自动 `setOpen(false)`。

> SortDropdown 完整 Props 和代码见 [通用组件与Hook.md](./10.通用组件与Hook.md) 第二节。

### 2.4 CommentCard 递归子评论

```jsx
<CommentCard comment={comment} depth={0} maxDepth={2} />
  → 渲染评论内容
  → {showSub && depth < maxDepth && comment.subCommentList.map(i =>
       <CommentCard comment={i} depth={depth+1} maxDepth={maxDepth} />
     )}
```

**递归终止条件**：`subCommentCount === 0` 时 subCommentList 为空，不再递归。`depth < maxDepth` 防止无限递归爆栈。

**布尔值状态 vs 字符串状态**：用 `useState(false)` 而非 `useState('展开')`。字符串做状态容易拼错、无法做反向逻辑、改文案要改 4 处。用布尔值 + 三元表达式，改文案只需改一处。

### 2.5 大重构：DataList 重写 + VideoPlayer 三态 + useData refresh

**DataList 五个核心修复**：

① **错误处理逻辑完全重写**：原来用 `data.code !== 200 && data.code !== undefined` 判断，但 data=null 时 `null.code=undefined`，`undefined !== 200` 为 true → 页面加载瞬间就跳走。改为使用 useData 返回的 `error` 状态判断。

② **骨架屏**：`Array(pageSize).fill(0).map()` 占位，animate-pulse 闪烁效果，数量与 pageSize 一致保持 CLS=0。

③ **navigate(0) → refresh**：删掉 useNavigate 依赖，错误态重试按钮直接调 useData 的 `refresh()`，只重新发请求不影响其他组件。

④ **默认 gridClassName**：改为 `flex flex-col gap-4`（竖向列表），评论区直接用。

⑤ **data.data 嵌套**：`listData?.data ?? []`，listData 已经是 `{ data:[], total:N }`。

**VideoPlayer 三态 UI**：loading → 骨架屏，error → 错误+重试按钮，success → `<video>`。三个状态由组件内部判断，对外只暴露 `playVideo/loading/error/onRefresh` 四个 prop。

**Video.jsx 数据流修正**：

```jsx
// 之前（错误）
const video = useData(getVideo, vid)?.data

// 之后（正确）
const { data: videoData, loading, error, refresh } = useData(fetchItem, 'video', vid)
const video = videoData?.data

<VideoPlayerApp playVideo={video} loading={loading} error={error} onRefresh={refresh} />
```

### 2.6 CommentCard 深度改造：6 个 Bug + 拍平方案 + replyTo

**6 个 Bug 修复**：

| Bug | 问题 | 修复 |
|-----|------|------|
| 1 | `handelShowSub` 拼写错误 | → `handleToggleSub` |
| 2 | `useState('展开')` 存中文字符串 | → `useState(false)` 布尔值 |
| 3 | `map` 渲染缺少 `key` | → `key={comment.cid}` |
| 4 | 点踩/回复是 `<span>` 不可点击 | → `<button>` + `onClick` |
| 5 | 固定 `width:50px/80px` 响应式灾难 | → Tailwind 响应式类 |
| 6 | 递归无深度限制可能爆栈 | → `depth` + `maxDepth=2` |

**拍平方案（已放弃）**：尝试取消缩进、用竖线颜色区分深度。结论是灾难性失败——没有缩进后父子关系完全靠颜色区分，评论一多无法分辨。回到缩进方案，竖线统一 `border-gray-300`。

**replyTo 字段**：修改 `createComment.js`，子评论携带 `replyTo`（父评论的 uploader），显示"回复 @XXX"。签名变为 `createComment(number, parentUploader = null)`。

### 2.7 SideVideoFlex → DataList 直接集成

**架构决策**：不保留 SideVideoFlex 包装组件，直接在 Video.jsx 右侧栏用 DataList。

```jsx
{video && (
  <DataList
    request={(sort, page, pageSize) =>
      fetchItemList('video', { subType: 'related', vid: video.vid, sort, page, element: pageSize })
    }
    sort={null}
    pageSize={5}
    renderItem={(v) => <VideoCard video={v} />}
    gridClassName="flex flex-col gap-3"
  />
)}
```

`{video && (...)}` 防止 `video` 加载中时 `video.vid` 为 undefined。

**传 vid 而非传 category**：后端推荐是标准做法（YouTube/B站），灵活性远比省一次查询更重要。

**Mock 预填充问题**：`getRelatedVideos` 从 publicState 取数据，但 publicState 的 video 列表只在 `getVideoList` 被调用时才填充。用户直接访问详情页时数据为空。修复：mock 文件加载时预填充 48 条视频。

### 2.8 UploaderCard Tailwind 化

保持独立封装（职责单一）。修复 Avatar 永远显示默认图标：使用真实 `profilePictureUrl`。

### 2.9 交互体系重构

**request.js 拦截器修复**：原来所有 `needToken:true` 的请求都被改成 POST，DELETE 也失效。修复后 DELETE 保持 method 不变，token 通过 data 字段传。同时 `config.data` 改为展开合并而非替换，防止覆盖上传请求的数据。

**统一 interact() 函数**：取消语义化包装（likeVideo、favouriteVideo 等），只导出 `interact(mediaType, action, mediaId)`。6 个 action × 3 个 mediaType = 18 个函数太多了，`interact('video', 'like', 123)` 语义自文档化。

> interact 的完整 ACTION_MAP 见 [前端接口分析.md](./13.%20VoteVideo-前端接口分析.md) 第四节。

**状态下沉原则**：谁用谁管状态，不从父组件透传。

| 组件 | 自身管理 | 调用方式 |
|------|---------|---------|
| InteractionBar | 6 个状态（liked/favourited/reshared 各 is+count） | `interact('video', action, vid)` |
| CommentCard | 3 个状态（isLiked/likeCount/isDisliked） | `interact('comment', action, cid)` |
| UploaderCard | 2 个状态（isFollowing/isLoading） | `interact('user', 'follow', uid)` |

**乐观更新 + 回滚模式**：

```jsx
const handleLike = () => {
  const prev = isLiked
  setIsLiked(!isLiked)           // 乐观更新
  setLikeCount(c => isLiked ? c - 1 : c + 1)
  interact('video', isLiked ? 'unlike' : 'like', vid)
    .catch(() => {
      setIsLiked(prev)           // 回滚
      setLikeCount(c => isLiked ? c - 1 : c + 1)
    })
}
```

乐观更新先改 UI 再发请求，失败时回滚。好处是"秒响应"。

**useInteraction Hook 暂缓原因**：三个组件都有乐观更新+回滚，但状态形状不一致（6/3/2 字段）。抽象时机：当多个组件有**完全相同**的状态形状和行为时才抽象，"差不多"不够，必须 shape 一致。

### 2.10 vid 变化不刷新（Bug 修复）

**问题**：点击侧边栏推荐视频跳转时，只有视频信息刷新，评论区+推荐视频不变。

**根因**：两个独立问题叠加。

① **useData 依赖缺陷**（影响推荐视频）：`useEffect` 依赖是 `[...params]`，request 箭头函数不在依赖里。vid 变化 → request 重新创建 → 但 params 内容没变 → useEffect 不触发。

② **评论接口不带 vid**（影响评论区）：getCommentList 没有 vid 参数，无法按视频过滤。

**修复**：vid 打入 params 数组。

```jsx
<DataList
  request={getCommentList}
  params={[video?.vid]}   // vid 变化 → params 变化 → useData 重新请求
  ...
/>
```

**为什么不用 key**：key 强制 remount 是粗粒度的，丢失内部状态（页码）。params 触发 useEffect 更精细。

**为什么 `video?.vid` 不是 `video.vid`**：首次渲染 video 是 undefined，`video.vid` 会崩溃。`video?.vid` 返回 undefined，等加载完才用真实 vid 过滤。

## 三、架构决策

### 3.1 受控组件模式

SortDropdown 内部 sort 和父组件 sort 是两个独立 state，通过 useEffect 单向同步。子组件不知道父组件拿 sort 干嘛——解耦。

### 3.2 状态归属

VideoPage 本身不持有任何交互 state，只负责数据流（video/sort）和布局。各子组件自身管理交互状态。

### 3.3 乐观更新 vs 等待成功

Video 页面交互用乐观更新+回滚（追求秒响应），后续 Essay/Post/Tag 页面改为等待成功模式（逻辑更简单，无回滚代码）。两种模式的取舍取决于业务场景。

> 等待成功模式的详细说明见 [通用组件与Hook.md](./10.通用组件与Hook.md) 第三节。
