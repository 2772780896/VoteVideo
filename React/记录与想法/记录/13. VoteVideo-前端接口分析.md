# VoteVideo React 前端 — 后端接口需求分析

## 概述

本文档基于 VoteVideo 项目 React 前端源码的完整分析，列出所有需要后端实现的 API 接口，以及每个接口的请求方式、路径、发送数据和预期返回数据。

**认证方式：** 基于 Cookie 存储的 Bearer Token。前端通过 `js-cookie` 管理 `token` 和 `uid` 两个 Cookie，需要认证的请求在 `Authorization` 请求头中携带 `Bearer <token>`。

**通用响应格式（从前端消费方式推断）：**

```json
{
  "code": 200,
  "message": "描述信息",
  "data": { ... }
}
```

列表类接口的 `data` 结构推断为：

```json
{
  "items": [ ... ],
  "total": 100
}
```

## 一、账户与认证模块

### 1. 用户登录

| 项目 | 内容 |
|------|------|
| **路径** | `POST /api/login` |
| **认证** | 不需要 |
| **调用位置** | `LoginModal.jsx` |

**请求体：**

```json
{
  "username": "字符串",
  "password": "字符串"
}
```

**预期响应（code=200 表示成功）：**

```json
{
  "code": 200,
  "message": "...",
  "data": {
    "token": "JWT 令牌字符串",
    "uid": "用户 ID（数字或字符串）"
  }
}
```

前端成功逻辑：`res.data.code === 200` 时将 `token` 和 `uid` 写入 Cookie，跳转到 `/user/profile`。失败时显示 `res.data.message`。

### 5. 获取个人资料子数据（投稿/收藏/历史/关注/消息）

| 项目 | 内容 |
|------|------|
| **路径** | `GET /api/user/profile/{profileType}/{dataType}` |
| **认证** | 需要（Bearer Token） |
| **调用位置** | `ProfileTabContent.jsx`、`ProfileFollow.jsx`、`ProfileMessage.jsx` |

**路径参数：**

- `profileType`：`'uploads'`（投稿）| `'favourites'`（收藏）| `'history'`（历史）| `'follow'`（关注）| `'message'`（消息）
- `dataType`：
  - profileType=uploads/favourites/history 时：`'videos'` | `'posts'` | `'essays'` | `'comments'`
  - profileType=follow 时：`'followingList'`
  - profileType=message 时：`'dialogueList'` | `'notificationList'`

**查询参数：**

```
sort    — 排序方式（如 "-date"）
page    — 页码，默认 1
element — 每页数量，默认 16
```

**预期响应 data（列表类通用结构）：**

```json
{
  "items": [ ... ],
  "total": 100
}
```

根据 profileType/dataType 不同，`items` 中的元素结构不同：

**uploads/favourites/history → videos 的 item：** 同视频卡片数据结构（vid, title, coverUrl, viewCount, commentCount, uploader 等）

**uploads/favourites/history → posts 的 item：** 同图文卡片数据结构（pid, title, pictureList, videoList, viewCount 等）

**uploads/favourites/history → essays 的 item：** 同文章卡片数据结构（eid, title, thumbnailUrl, viewCount 等）

**uploads/favourites/history → comments 的 item：** 同评论卡片数据结构（cid, text, date, type, uploader, replyTo, subCommentCount, isLiked, likeCount 等）

**follow/followingList 的 item：**

```json
{
  "uid": "用户 ID",
  "profilePictureUrl": "头像 URL",
  "userName": "用户名",
  "followerCount": 123
}
```

**message/dialogueList 的 item：**

```json
{
  "mid": "对话 ID",
  "opponent": {
    "uid": "对方用户 ID",
    "userName": "对方用户名",
    "profilePictureUrl": "对方头像 URL"
  },
  "sentences": [
    { "text": "消息文本", "date": "发送时间", "sender": "发送者标识" }
  ]
}
```

**message/notificationList 的 item：**

```json
{
  "mid": "消息 ID",
  "sender": {
    "uid": "发送者 ID",
    "userName": "发送者用户名",
    "profilePictureUrl": "发送者头像"
  },
  "text": "通知文本内容",
  "date": "时间",
  "targetType": "目标类型（如 comment）",
  "targetId": "目标 ID",
  "parentType": "父级类型（如 video）",
  "parentId": "父级 ID"
}
```

### 8. 获取资源列表（通用）

| 项目 | 内容 |
|------|------|
| **路径** | `GET /api/{resourceType}` |
| **认证** | 不需要 |
| **调用位置** | `Search.jsx`、`User.jsx`、`Dynamic.jsx`、`Video.jsx`（评论列表）、`Essay.jsx`（评论列表）、`Tag.jsx`（评论列表） |

**路径参数：** `resourceType`：`'video'` | `'essay'` | `'post'` | `'comment'` | `'user'` | `'tag'`

**查询参数：**

```
sort     — 排序方式（如 "-date" 表示按日期降序）
page     — 页码
element  — 每页数量
q        — 搜索关键词（搜索页使用；为空或不传时后端按热度返回默认推荐数据）
uid      — 用户 ID（用户主页筛选时使用；与 following=true 配合使用时表示当前登录用户）
following— 布尔值，true 表示只返回 uid 所关注用户的内容（动态页 Feed 使用）
vid      — 视频 ID（获取视频评论时使用）
```

**`q` 参数为空时的行为（搜索页默认推荐）：**

当搜索页 `/search` 的 URL 中 `q` 参数为空字符串或未传入时，前端会以 `sort=-viewCount`（按浏览量降序）调用资源列表接口，后端应返回**按热度排序的默认推荐数据**（即忽略 `q` 过滤，仅按 `sort` 排序返回全量数据）。这样用户进入搜索页时即可浏览热门内容，而非空白页面。

前端调用示例：
```javascript
// q 为空时：GET /api/video?sort=-viewCount&page=1&element=16
// q 有值时：GET /api/video?sort=-date&page=1&element=16&q=关键词
fetchItemList(type, { sort, page, element, q: keyword || undefined })
```

**预期响应 data：**

```json
{
  "items": [ ... ],
  "total": 100
}
```

items 中的元素结构取决于 resourceType，分别对应 VideoCard / EssayCard / PostCard / CommentCard / UserCard / TagCard 所需的数据。

**评论 item 结构（resourceType=comment）：**

```json
{
  "cid": "评论 ID",
  "text": "评论内容",
  "date": "评论时间",
  "type": "'text' | 'picture'",
  "pictureList": ["图片URL1", "图片URL2"],
  "replyTo": { "userName": "被回复的用户名" },
  "uploader": {
    "uid": "评论者 ID",
    "userName": "评论者用户名",
    "profilePictureUrl": "头像 URL"
  },
  "subCommentCount": 3,
  "subCommentList": [ ... ],
  "isLiked": false,
  "likeCount": 5,
  "isDisliked": false,
  "isFavourited": false,
  "favouriteCount": 0,
  "isReshared": false,
  "reshareCount": 0
}
```

注意：`uploader` 字段（而非 `commenter`）与其他资源类型保持一致。`subCommentList` 为嵌套的子评论数组，结构与一级评论相同，前端通过 `depth` + `maxDepth=2` 限制递归深度。`type='picture'` 时 `pictureList` 有值，前端会渲染图片附件。`replyTo` 存在时前端显示"回复 @XXX"标签。

### 12. 获取关注用户动态 Feed

| 项目 | 内容 |
|------|------|
| **路径** | `GET /api/{resourceType}` （复用第 8 接口，附加 `following` 参数） |
| **认证** | 需要（Bearer Token，因为需要知道当前用户的关注列表） |
| **调用位置** | `Dynamic.jsx` |

**路径参数：** `resourceType`：`'video'` | `'essay'` | `'post'` | `'comment'` | `'tag'`

**查询参数：**

```
uid      — 当前登录用户 ID（从 Cookie 读取）
following— true，固定值，表示只返回该用户关注的人发布的内容
sort     — 排序方式（默认 "-date"，最新优先）
page     — 页码
element  — 每页数量（默认 10）
```

**后端处理逻辑：**

1. 根据 `uid` 查询该用户关注的所有用户 ID 列表（从 `follow` 关系表中获取）
2. 筛选这些用户发布的、对应 `resourceType` 类型的资源
3. 按 `sort` 排序后分页返回
4. 如果用户没有关注任何人，返回空列表 `{ items: [], total: 0 }`

**前端调用示例：**
```javascript
// 获取关注用户的动态（post）
// GET /api/post?uid=123&following=true&sort=-date&page=1&element=10
fetchItemList('post', { sort: '-date', page: 1, element: 10, uid: '123', following: true })

// 获取关注用户的视频
// GET /api/video?uid=123&following=true&sort=-date&page=1&element=10
fetchItemList('video', { sort: '-date', page: 1, element: 10, uid: '123', following: true })
```

**预期响应 data：**

```json
{
  "items": [ ... ],
  "total": 100
}
```

`items` 中的元素结构取决于 `resourceType`，与第 8 接口的各类型 item 结构一致（VideoCard / EssayCard / PostCard / CommentCard / TagCard 所需的数据）。

## 四、社交互动模块（统一 interact 接口）

前端通过一个统一的 `interact(mediaType, action, mediaId, data)` 函数处理所有社交互动。**所有媒体类型（video/essay/post/tag/comment）都支持相同的交互操作**：点赞、点踩、收藏、转发、回复。`mediaType` 可选值为 `'video'` | `'essay'` | `'post'` | `'tag'` | `'comment'` | `'user'`。需要认证。

**前端交互模式（等待成功）：** InteractionBar 采用"先调 API，成功后再更新 UI"的模式（非乐观更新），无回滚代码。失败时显示 2 秒红色错误提示自动消失。每种互动操作前都会先检查登录状态（通过 Zustand `authStore`），未登录则打开全局登录弹窗而非发送请求。

**统一的交互接口（所有媒体类型都支持）：**

### 16. 点赞 / 取消点赞

### 22. 历史记录

| 项目 | 内容 |
|------|------|
| **历史记录路径** | `POST /api/{mediaType}/{mediaId}/history` |
| **调用位置** | `Video.jsx`、`Essay.jsx`、`Post.jsx`（媒体详情页面） |

**功能说明：**
- 当用户查看视频/文章/图文时，前端会自动调用此接口记录浏览历史
- 此接口应该在媒体详情加载完成后调用（即 `useData` 返回数据且 `loading` 为 `false` 时）
- 评论（`comment`）不需要历史记录功能
- 前端采用静默失败策略：历史记录失败不影响用户体验

**前端实现逻辑：**
```javascript
useEffect(() => {
  if (video && !loading) {
    interact('video', 'history', vid).catch(() => {
      // 静默失败
    })
  }
}, [video, loading, vid])
```

## 七、补充说明

**超时配置：** Axios 实例超时时间为 5000ms（5 秒）。

**响应数据解包：** `useData` Hook 统一从 `response.data.data` 中提取业务数据，因此后端返回需严格遵循 `{ code, message, data }` 三层结构。

**错误码处理：** `useData` Hook 对 HTTP 错误做了分类映射：400→"请求参数错误"、401→"未登录或登录已过期"、403→"没有权限访问"、404→"请求的资源不存在"、500→"服务器内部错误"、无响应→"网络连接失败"（code=0）、配置错误（code=-1）。Profile 页面收到 401 时自动跳转到 `/main`。

**认证状态管理：** 前端使用 Zustand `authStore` 管理全局认证状态。状态包括 `isLoggedIn`（布尔值）、`user`（缓存的用户信息）、`loginModalOpen`（登录弹窗可见性）。`initAuth()` 在应用启动时（`main.jsx`）调用 `getMyProfile()` 验证 Cookie 中的 token 有效性。token 和 uid 存储在 Cookie 中，有效期 7 天。

**交互登录检查：** 所有交互操作（点赞/踩/收藏/转发/回复/关注）在执行前都会通过 `useAuthStore` 检查登录状态，未登录则打开全局登录弹窗（`openLoginModal()`），而非直接发送请求。

**排序参数约定：** 前端使用 `-字段名` 表示降序、`字段名` 表示升序，如 `-date` 表示按日期降序。

**分页参数约定：** `page` 从 1 开始，`element` 表示每页数量（不同页面默认值不同，常见为 10 或 16）。

**needToken 标记说明：** API 层的 `needToken: true` 是语义标记，实际的 token 注入由 `request.js` 的请求拦截器完成——只要 Cookie 中存在 `token`，所有请求（无论是否标记 `needToken`）都会自动携带 `Authorization: Bearer {token}`。这使得公开接口也能返回个性化数据（如 `isLiked`、`isFavourited`）。
