## 二、开发历程

### 2.1 URL 驱动的 Tab 通信模式

```
外部触发（导航头"稿件"）：
  navigate('/user/profile?tab=upload')
    → useSearchParams() 读到 tab=upload
      → activeKey='upload' → Tab 高亮

内部切换（用户点"收藏"）：
  handleTab('favourite')
    → navigate('/user/profile?tab=favourite')
      → URL 变化 → useSearchParams 返回新值 → 切换内容
```

**为什么用 URL 而非 useState？** 外部页面可直接指定 Tab，浏览器前进/后退支持，两个组件松耦合。

**为什么不用 Context/Redux？** 只有导航头需要 isLogin（一行 `!!Cookies.get('token')` 足够），没有其他跨组件共享状态需求。

**参数语义化**：`search=1/4/5` → `tab=upload/favourite/history`。

**navigate 不会重新挂载组件**：同一 Route 匹配 → useSearchParams 感知变化 → useData 的 depsKey 不变 → 不重新请求。

### 2.2 认证体系

**完整登录链**：

```
① LoginModal 输入 → login(username, password) → POST /api/login
② mock 匹配 userName+password → 生成 token=Mock.Random.guid() → { code:200, data:{ token, uid } }
③ LoginModal → Cookies.set('token',...) + Cookies.set('uid',...) → navigate('/user/profile')
④ Profile → getMyProfile() → request.get('/api/user/profile/', { needToken:true })
⑤ 拦截器 → Cookies.get('token') → 改 GET 为 POST，body={ authorization:token }
⑥ mock verifyAuthorization → 匹配 user.token → { code:200, data:user } 或 { code:401 }
```

**为什么不用 JWT/Context/Redux？** JWT 在 mock 中无意义（无后端签名验证）；Context 只有导航头需要 isLogin；Cookie 存 token 跨页面不丢、拦截器自动注入。

**注册自动登录**：注册成功 → 直接调 login → Cookie → 跳转，不再 alert 后手动切回登录 tab。

**修复 createUser 缺 password 字段**：所有用户统一密码 `'123'`。

### 2.3 LoginModal 三合一重构

LoginModal + LoginForm + RegisterForm → 合为一个文件，纯 Tailwind。

| 纬度 | 旧 | 新 |
|------|-----|-----|
| 弹窗 | antd Modal | Tailwind 固定遮罩 + 卡片 |
| 表单 | antd Form + Input | Tailwind input 双向绑定 |
| 提示 | alert() | 内联绿色/红色提示 |

**修复 navigate 在循环里的 Bug**：

```js
// 旧：navigate 在循环里，第一轮就跳走，uid 没存上
for (const [key, value] of Object.entries(res.data.data)) {
    Cookies.set(key, value)
    navigate('/user/profile')      // ← bug
}

// 新：循环完再跳
for (const [key, value] of Object.entries(res.data.data)) {
    Cookies.set(key, value)
}
navigate('/user/profile')
```

**CSS 继承陷阱**：TopMenu 的 `text-white` 继承到弹窗 → 白色文字+白色背景=看不见。修复：弹窗加 `text-gray-900` 覆盖。排查方法：DevTools → Computed → color 属性追溯来源。

### 2.4 ProfileTabContent 三合一 + RESTful API

**问题**：ProfileUpload/ProfileFavourite/ProfileHistory 三个文件 99% 相同，唯一差异是 profileType 和 dataType。

**方案**：DATA_MAP 映射表 + profileType prop 驱动：

```js
const DATA_MAP = {
  videos: { label: '视频', Card: VideoCard, idKey: 'vid' },
  posts:  { label: '动态', Card: PostCard,  idKey: 'pid' },
  essays: { label: '文章', Card: EssayCard, idKey: 'eid' },
}
```

一个 ProfileTabContent 组件处理所有组合。

**API 路径 RESTful 化**：

| 旧 | 新 |
|----|----|
| `/api/user/profile/upload/videoList/` | `/api/user/profile/uploads/videos` |
| `/api/user/profile/favourite/postList/` | `/api/user/profile/favourites/posts` |

去尾斜杠、去 List 后缀、单数改复数。

### 2.5 关注系统

**设计目标**：展示关注的用户列表，每个卡片有取消关注/私信按钮。取消关注后按钮变"关注"（用户不离开列表，可恢复）。

**按钮状态管理：数组 vs Set vs 对象**

```
方案 1）Set — 删即移除     → ❌ 用户从列表消失，无法恢复
方案 2）对象 { [uid]:bool } → value 没用过，多余
方案 3）数组 [uid, uid]    → ✅ 简洁，includes 查 O(n) 够用（8个用户）
```

```js
const [unfollowed, setUnfollowed] = useState([])
setUnfollowed(prev => [...prev, uid])        // 取消关注
setUnfollowed(prev => prev.filter(id => id !== uid))  // 重新关注
const isUnfollowed = unfollowed.includes(user.uid)
```

**为什么 `[...prev, uid]` 而非 `prev.push(uid)`**：useState 靠引用比较，push 不改引用 → 不重渲染。

**为什么不在 DataList params 里传 unfollowed**：params 变了 → depsKey 变了 → 重新请求。unfollowed 是纯前端 overlay 状态，不应触发请求。正确范式：**数据获取和 UI 状态分离**。

### 2.6 消息系统

**原始设计**：5 个 antd 文件（ProfileMessage + MessageDialogue + MessageAted + MessagePraised + MessageSystem），三个通知列表 100% 相同。

**三通知合并**：用户不关心"这个通知是 @ 还是系统发的"，只想看"最近谁找过我"。

**通知跳转流程**：

```
buildUrl(msg) 根据 targetType 拼路径：
  video → /video/:vid
  comment → /video/:parentId?comment=:targetId

CommentCard: <div id={`comment-${comment.cid}`}>   ← DOM 锚点
Video.jsx: useSearchParams().get('comment') → scrollIntoView   ← 评论锚点滚动
```

**为什么用 id 不用 key？** React 的 key 是虚拟 DOM 内部用的，不输出到 HTML。`document.getElementById` 只认 HTML id 属性。

**对话系统**：

```
ProfileMessage（状态管理者）→ ChatApp（纯展示组件）

对话列表 → 点击张三 → enterChat(dialogue)
  → chatDialogue = dialogue
  → chatMessages = [...dialogue.sentences]  // 拷贝，不污染 mock 数据

ChatApp 输入 → handleSend
  → 乐观更新：chatMessages 立刻追加（消息瞬间出现）
  → sendMessage(mid, text)

← 返回 → exitChat()
  → refreshKey++ → DataList key 变化 → 强制重新请求最新数据
```

**为什么退出要强制刷新**：params 不变 → useData 不重新请求。key 变化 → React 销毁重建 → 拿到最新数据。

**handleSend 乐观更新**：消息立刻出现（像微信），失败时静默。

### 2.7 私信跨组件通信

**问题**：ProfileFollow 和 User 页都有"私信"按钮，但 chatDialogue 状态只在 ProfileMessage 里。三者是兄弟关系。

**方案：React Router state**

```js
// 发送方
navigate('/user/profile?tab=message', {
  state: { targetUser: { uid, userName, profilePictureUrl } }
})

// 接收方
const targetUser = useLocation().state?.targetUser

// 消费方（ProfileMessage）
useEffect(() => {
  if (targetUser && !chatDialogue) {
    enterChat({ mid: null, opponent: targetUser, sentences: [] })
  }
}, [targetUser])
```

**state vs searchParams**：`?tab=message` 在 URL 里（可见、可书签），`state` 在浏览器内存里（不可见，适合结构化数据）。

## 三、架构决策

### 3.1 URL 驱动 Tab vs useState

URL 让外部导航可直接指定 Tab，支持浏览器前进/后退，组件间松耦合。

### 3.2 Cookie vs JWT/Context/Redux

当前项目只需 Cookie：token 跨页面不丢、拦截器自动注入。JWT 在 mock 中无意义，Context/Redux 无跨组件共享需求。

### 3.3 数组 vs Set vs 对象（关注系统）

8 个用户用数组 + includes（O(n)）足够。100+ 个考虑 Set/Map。

### 3.4 navigate state vs URL params（私信）

tab 语义适合放 URL，用户数据（targetUser）适合走 state。

## 六、JS 基础踩坑

**对象键名**：`{ key: 1 }` 键名是字符串 "key"，`{ [key]: 1 }` 才用变量值。

**JSX `...` 双重语义**：属性位置 `<Card {...props}/>` 是拆成独立 prop，内容位置 `{...obj}` 是普通 JS 展开。

**JSX `{}` 只能放表达式**：`{user.name}` 可以，`{const a = 1}` 不行。
