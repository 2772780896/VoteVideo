## 一、DataList — 通用列表组件

首次出现于 Video 页面评论区。

### 1.1 Props 接口

| Prop | 类型 | 说明 |
|------|------|------|
| `request` | `Function` | API 请求函数 |
| `renderItem` | `Function` | `(item, index) => ReactNode` |
| `sort` | `number \| string` | 排序参数 |
| `params` | `Array` | 额外参数，同时用于 depsKey 依赖检测 |
| `pageSize` | `number` | 每页条数，默认 16 |
| `gridClassName` | `string` | 容器 className，默认 `flex flex-col gap-4` |

### 1.2 关键逻辑

**sort 变化时重置页码**：

```jsx
useEffect(() => { setPage(1) }, [sort, ...params])
```

不加这个：用户在第 3 页切换排序，新数据只有 1 页，第 3 页显示空。

**数据层**：

```jsx
const { data: listData, loading, error, refresh } = useData(request, sort, page, pageSize, ...params)
const items = listData?.data ?? []
```

**骨架屏**：`Array(pageSize).fill(0).map()` 占位，animate-pulse 闪烁，数量与 pageSize 一致保持 CLS=0。

**错误处理 + refresh**：使用 useData 的 error 和 refresh，不依赖 data.code（避免 data=null 时误触发）。

**函数式更新 `setState(prev => ...)`**：新值依赖旧值时必须用函数式。`setPage(page - 1)` 两次快速点击只减一次，`setPage(prev => prev - 1)` 利用 React 状态队列保证每次 prev 正确。

### 1.3 调用示例

```jsx
// 评论区（竖向列表）
<DataList request={getCommentList} sort={sort} params={[video?.vid]}
  renderItem={(item) => <CommentCard key={item.cid} comment={item} />} />

// 相关视频（竖向，少量）
<DataList request={fn} pageSize={5} gridClassName="flex flex-col gap-3"
  renderItem={(v) => <VideoCard video={v} />} />

// 搜索结果（网格，需 key 防 tab 切换崩溃）
<DataList key={activeKey} request={adapter} params={[type, keyword]}
  renderItem={(item) => <Card {...item} />} />
```

## 二、SortDropdown — 受控下拉组件

首次出现于 Video 页面评论区排序。

### 2.1 设计原则

**开闭原则**：组件只管"下拉菜单怎么渲染"，不关心"有哪些选项"。`sortList` 作为 prop 传入，换一个 sortList 就能用在不同场景。

### 2.2 受控组件模式

两个 sort 的职责分离：

| sort | 位置 | 职责 |
|------|------|------|
| SortDropdown 内部 state | 组件内 | 控制显示哪个 label |
| 父组件 state | VideoPage 等 | 传给下游触发重新请求 |

子组件通过 `useEffect(() => pushSort(sort), [sort])` 单向同步给父组件。

### 2.3 Props 接口

| prop | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `pushSort` | `function` | 是 | - | 回调，通知父组件 |
| `defaultSort` | `string` | 否 | `'-viewCount'` | 默认排序 key |
| `sortList` | `Array<{label, key}>` | 是 | - | 排序选项（必填，强制显式传参） |

### 2.4 Tailwind 实现要点

| 功能 | 实现 |
|------|------|
| 点击外部关闭 | `useRef` + `mousedown`（展开时才注册，收起时卸载） |
| 用 mousedown 不用 click | click 在 mousedown 之后触发，先关菜单会导致目标收不到事件 |
| 三角旋转 | `transition-transform duration-200 rotate-180` |
| 选中高亮 | 条件 class `bg-gray-50 text-blue-500` |
| 选中关闭 | `onSelect` 里 `setOpen(false)` |

### 2.5 完整代码

```jsx
const App = ({ pushSort, defaultSort = '-viewCount', sortList }) => {
  const [sort, setSort] = useState(defaultSort)
  const [open, setOpen] = useState(false)
  const ref = useRef(null)

  useEffect(() => { pushSort(sort) }, [sort, pushSort])

  useEffect(() => {
    const handleClickOutside = (e) => {
      if (ref.current && !ref.current.contains(e.target)) setOpen(false)
    }
    if (open) document.addEventListener('mousedown', handleClickOutside)
    return () => document.removeEventListener('mousedown', handleClickOutside)
  }, [open])

  return (
    <div ref={ref} className="relative">
      <button onClick={() => setOpen(!open)}>
        {sortList.find(i => i.key === sort)?.label}
        <span className={`transition-transform duration-200 ${open ? 'rotate-180' : ''}`}>▾</span>
      </button>
      {open && (
        <div className="absolute right-0 mt-2 w-40 bg-white rounded-xl shadow-lg z-50">
          {sortList.map((item) => (
            <button key={item.key}
              onClick={() => { setSort(item.key); setOpen(false) }}
              className={item.key === sort ? 'bg-gray-50 text-blue-500' : ''}>
              {item.label}
            </button>
          ))}
        </div>
      )}
    </div>
  )
}
```

## 三、InteractionBar — 通用交互栏

首次出现于 Video 页面交互重构，后推广到所有媒体类型。

### 3.1 ID_KEY_MAP 映射表

```js
const ID_KEY_MAP = { video:'vid', essay:'eid', post:'pid', tag:'tid', user:'uid', comment:'cid' }
const mediaId = item?.[ID_KEY_MAP[mediaType]]
```

新增资源类型只需加一行。

### 3.2 等待成功模式（vs 乐观更新）

Video 页面的交互用乐观更新+回滚（追求秒响应），但代码量翻倍。InteractionBar 统一用等待成功模式：

```js
const next = !isLiked
try {
  await interact(mediaType, next ? 'like' : 'unlike', mediaId)
  setIsLiked(next)
  setLikeCount(prev => next ? prev + 1 : prev - 1)
} catch {
  showError('操作失败，请重试')
}
```

逻辑直观，无回滚代码。

### 3.3 失败提示：useRef + setTimeout

```js
const timerRef = useRef(null)
const showError = (msg) => {
  setErrorMsg(msg)
  clearTimeout(timerRef.current)    // 清除旧定时器，防提前消失
  timerRef.current = setTimeout(() => setErrorMsg(null), 2000)
}
```

**为什么用 useRef 而非普通变量**：React 每次渲染重新执行函数体，`let timer` 会是新 undefined。useRef 跨渲染持久。

### 3.4 调用示例

```jsx
<InteractionBar mediaType="video" item={video} />
<InteractionBar mediaType="essay" item={essay} />
<InteractionBar mediaType="tag"   item={tag} showReshare={false} />
```

### 3.5 设计决策

**为什么不用 useInteraction Hook 统一抽象**：三个使用场景的状态形状不一致（6/3/2 字段），抽象复杂度超过收益。抽象时机：当多个组件有**完全相同**的状态形状和行为时才做。

## 四、useData — 数据请求 Hook

首次设计于 Main 页面，后在 Video/Essay/Search 页面持续演进。

### 4.1 设计动机

每个页面都写 `useEffect + useState(data/loading/error)` 是重复劳动。useData 封装成黑盒：

```js
const { data, loading, error, refresh } = useData(requestFunc, ...params)
```

### 4.2 三状态语义

| 状态 | data | loading | error | UI |
|------|------|---------|-------|-----|
| 加载中 | `null` | `true` | `null` | 骨架屏 |
| 成功 | 真实数据 | `false` | `null` | 渲染内容 |
| 失败 | `null` | `false` | Error | 错误提示+重试 |

**初始 data 用 null 而非 []**：`null`="还没请求"→骨架屏，`[]`="请求成功但为空"→空状态。语义不同。

### 4.3 核心实现

```js
const useData = (requestFunc, ...params) => {
  const [data, setData] = useState(null)
  const [loading, setLoading] = useState(true)
  const [error, setError] = useState(null)

  const refresh = () => getData()

  async function getData() {
    setLoading(true)
    setError(null)  // 重试时清旧 error，防 UI 残留
    try {
      const response = await requestFunc(...params)
      setData(response.data)
    } catch (err) {
      setError(err)
    } finally {
      setLoading(false)
    }
  }

  const depsKey = JSON.stringify(params)
  useEffect(() => { getData() }, [depsKey])

  return { data, loading, error, refresh }
}
```

### 4.4 演进历程

**v1（Main 页面）**：三状态 `{ data, loading, error }`，依赖 `[...params]`。

**v2（Video 页面）**：新增 `refresh`，替代 `navigate(0)` 整页刷新。

**v3（Essay 页面 depsKey 修复）**：`[...params]` 每次渲染创建新数组 → `[] !== []` → 死循环。改为 `JSON.stringify(params)` 做值比较。

**v4（Search 页面竞态修复）**：连续搜索时后发请求可能先返回，被先发请求覆盖。用 `useRef` 做 requestId，过期响应直接丢弃。

```js
const requestIdRef = useRef(0)
async function getData() {
    const currentId = ++requestIdRef.current
    // ... await ...
    if (currentId !== requestIdRef.current) return  // 过期，丢弃
    setData(response.data)
}
```

**useRef 原理**：`{ current: 0 }` 存在于 React 状态系统外部，改它不触发渲染，渲染也不重置。就像组件外面挂了个小盒子。

### 4.5 设计反思

当前 params 混合了"传参"和"依赖"两个职责——keyword 不需要传给函数（闭包拿了），但必须写进 params（让 depsKey 变）。理想设计：`useData(requestFunc, deps, ...args)` 分开。

## 六、接口请求统一（content.js）

### 6.1 SUBTYPE_MAP + fetchItem + fetchItemList

```js
const SUBTYPE_MAP = {
  video: { main: '/api/video/main', related: '/api/video/related' },
  picture: { carousel: '/api/picture/carousel' },
}

const fetchItem = (type, id) => request.get(`/api/${type}/${id}`)
const fetchItemList = (type, params = {}) => {
  const { subType, ...rest } = params
  const url = SUBTYPE_MAP[type]?.[subType] ?? `/api/${type}`
  return request.get(url, { params: rest })
}
```

### 6.2 interact 统一交互

> 完整 ACTION_MAP 见 [前端接口分析.md](./13.%20VoteVideo-前端接口分析.md) 第四节。

```js
const interact = (mediaType, action, mediaId) => { ... }
// mediaType + action → HTTP method + 路径
// 新增 action 只需改 ACTION_MAP
```

对外只暴露 `{ fetchItem, fetchItemList, uploadContent, interact }`。

### 6.3 设计决策

- **fetchItem vs fetchItemList 分开**：单个资源 `/api/{type}/{id}`，列表 `/api/{type}`，逻辑不同
- **subType 用字符串**：可拓展，新增接口只需 SUBTYPE_MAP 加一行
- **取消语义化包装**：6 action × 3 mediaType = 18 个函数太多，interact 自文档化
