## 一、 Node.js 基础与模块化

### 1. 运行环境
- **Node.js**：JS 的后端运行环境，脱离浏览器运行。
- **限制**：无法使用 DOM 和 BOM。
- **运行**：终端输入 `node xxx.js`。

### 2. 导入导出 (模块化)
#### CommonJS (CJS) - Node 默认
- **导出**：`module.exports = { name: 'zs' }`
- **导入**：`const m = require('./xxx.js')`
#### ECMAScript (ESM) - 现代标准
- **配置**：`package.json` 写入 `{"type": "module"}`。
- **导出**：`export default { ... }` 或 `export const a = 1`。
- **导入**：`import m from './xxx.js'` 或 `import { a } from './xxx.js'`。

## 二、 核心内置模块 (详解与实战)

### 1. path 模块：路径处理
- **__dirname**：当前文件的父文件夹绝对路径。
- **path.join()**：跨平台拼接路径（自动处理斜杠）。
  ```javascript
  const path = require('path');
  const fullPath = path.join(__dirname, 'data', '1.txt');
  ```

### 2. fs 模块：文件读写
- **同步/异步写法**：建议使用 `fs.promises` 以支持 `async/await`。
  ```javascript
  const fs = require('fs').promises;
  // 写入
  await fs.writeFile('./test.txt', '内容', 'utf-8');
  // 读取
  const data = await fs.readFile('./test.txt', 'utf-8'); 
  // 注意：如果不加 'utf-8'，读出来的是 Buffer (十六进制)，需 data.toString() 转换。
  ```

### 3. http 模块：创建 Web 服务 (了解原理)
```javascript
const http = require('http');
const server = http.createServer((req, res) => {
  // 设置响应头，防止中文乱码
  res.setHeader('Content-Type', 'text/html; charset=utf-8');
  if (req.url === '/index') {
    res.end('首页');
  }
});
server.listen(3000, () => console.log('启动在 3000'));
```

## 六、 进阶实战：你还需要学什么？

### 1. 数据库交互 (基础：MySQL 原生连接)
- **核心包**：`npm i mysql2`。
- **为什么要用连接池？**：频繁创建和销毁连接非常耗性能。连接池可以复用连接，提高效率。
- **实战配置 (db.js)**：
  ```javascript
  const mysql = require('mysql2/promise');

  // 创建连接池
  const pool = mysql.createPool({
    host: 'localhost',
    user: 'root',
    password: '密码',
    database: '数据库名',
    waitForConnections: true,
    connectionLimit: 10 // 最大连接数
  });

  module.exports = pool;
  ```
- **在 Express 中使用**：
  ```javascript
  const db = require('./db');

  app.get('/users', async (req, res) => {
    try {
      // 使用 execute 防止 SQL 注入
      const [rows] = await db.execute('SELECT id, username FROM users WHERE age > ?', [18]);
      res.send({ code: 200, data: rows });
    } catch (err) {
      res.status(500).send('服务器错误');
    }
  });
  ```

### 2. 数据库进阶：Prisma (ORM)
- **Node.js、Express 与 Prisma 的关系**：
  - **Node.js**：运行环境（地基）。
  - **Prisma**：数据库操作工具（挖掘机）。它运行在 Node.js 环境中，不依赖于 Express，但常被用在 Express 的路由函数里。
  - **Express**：Web 框架（房子）。它负责接收前端请求，然后调用 Prisma 去数据库拿数据。

- **Prisma 的核心优势**：
  - **强类型**：配合 TypeScript，写代码时能自动提示数据库所有字段名。
  - **模型驱动**：你只需要定义一个 `schema.prisma` 文件，它会自动帮你建表。

- **Prisma 快速上手流程**：
  1. **初始化**：`npx prisma init` (生成 schema 文件)。
  2. **定义模型**：
     ```prisma
     // schema.prisma
     model User {
       id    Int     @id @default(autoincrement()) // @id 主键，@default(autoincrement()) 自增
       name  String
       email String  @unique // @unique 唯一索引
       posts Post[]  // 关联模型
     }
     ```
  3. **同步数据库**：`npx prisma db push`。
  4. **CRUD 核心语法**：
     - **增**：`prisma.user.create({ data: { name: 'zs', email: 'zs@qq.com' } })`
     - **删**：`prisma.user.delete({ where: { id: 1 } })`
     - **改**：`prisma.user.update({ where: { id: 1 }, data: { name: 'ls' } })`
     - **查 (所有)**：`prisma.user.findMany()`
     - **查 (条件)**：`prisma.user.findMany({ where: { name: { contains: 'z' } } })` // 包含 'z' 的
     - **查 (排序/分页)**：`prisma.user.findMany({ skip: 0, take: 10, orderBy: { id: 'desc' } })`

- **select 与 in 详解**：

  > `select` 控制“返回哪些字段”，`in` 控制“匹配一组值”。两者都是查询时的常用操作。

  **❶ select —— 只查你要的字段**

  默认 Prisma 会把所有字段都返回，`select` 让你只拿你要的：
  ```js
  // 不用 select → 返回全部字段
  const likes = await prisma.userLike.findMany({ where: { uid: 42 } })
  // → [{ uid: 42, type: 'post', item_id: 5, addDate: '...' }, ...]

  // 用 select → 只要 item_id
  const likes = await prisma.userLike.findMany({
    where: { uid: 42 },
    select: { item_id: true }   // ← "只要 item_id 这个字段"
  })
  // → [{ item_id: 5 }, { item_id: 8 }, { item_id: 12 }]
  ```
  **对比 include 和 select：**

  | | include | select |
  |---|---------|--------|
  | 作用 | 把关联表的数据**带进来** | 只取当前表的**部分字段** |
  | 用法 | `include: { uploader: true }` | `select: { item_id: true }` |
  | 返回 | 全字段 + 关联数据 | 只返回你选中的字段 |
  | 场景 | 需要上传者信息 | 只需要 ID 做判断 |

  **❷ in —— “在…之中”（批量匹配）**

  普通的 `where` 只能匹配**一个值**，`in` 可以匹配**一组值**：
  ```js
  // 普通 where → 只能查一个
  const like = await prisma.userLike.findMany({
    where: { item_id: 5 }   // 只查 item_id = 5 的
  })

  // 用 in → 批量查
  const postIds = [5, 8, 12, 17]
  const likes = await prisma.userLike.findMany({
    where: {
      item_id: { in: postIds }   // ← "item_id 在 [5, 8, 12, 17] 之中"
    }
  })
  // 等价于 SQL: WHERE item_id IN (5, 8, 12, 17)
  ```

  **❸ 实战：批量查交互状态（项目里的典型用法）**
  ```js
  // 第一步：当前页有 10 条动态，拿到它们的 ID
  const postIds = [5, 8, 12, 17, ...]

  // 第二步：一次性查出“当前用户点赞过哪些”
  const likes = await prisma.userLike.findMany({
    where: {
      uid: currentUid,           // 当前用户
      type: 'post',              // 动态类型
      item_id: { in: postIds }   // item_id 在当前页的 ID 列表中
    },
    select: { item_id: true }    // 只要 item_id，不要其他字段
  })
  // → [{ item_id: 5 }, { item_id: 12 }]  （用户只点赞了 5 和 12）

  // 第三步：用 Set 快速判断每条动态有没有被点赞
  const likeSet = new Set(likes.map(like => like.item_id))
  // likeSet = { 5, 12 }

  // 第四步：给每条动态标记交互状态
  postIds.forEach(pid => {
    interactionMap[pid] = {
      liked: likeSet.has(pid),       // 5 → true, 8 → false
      favourited: favouritesSet.has(pid),
      reshared: reshareSet.has(pid)
    }
  })
  ```
  **为什么用 in 而不是一条条查？**

  | 写法 | 查询次数 |
  |------|--------|
  | 不用 in（循环查 10 条） | 10 次数据库查询 |
  | 用 in（一次批量查） | **1 次**数据库查询 |

  性能差 10 倍，数据量越大差距越大。

- **@relation 与 include 详解**：

  > **核心概念**：`@relation` 在 schema 里声明关系（只写一次），`include` 在查询时触发连表（每次查询可选）。

  **❶ 不写 @relation，不用 include —— 各查各的**

  数据库里 Video 表只有一个 `uploader_uid` 字段存数字 42，Prisma 不知道它和 User 有关。
  ```prisma
  model Video {
    vid           Int     @id
    title         String
    uploader_uid  Int     // 就是个普通数字字段
  }
  ```
  ```js
  // 查视频，只拿到数字 42
  const video = await prisma.video.findUnique({ where: { vid: 1 } })
  // → { vid: 1, title: '猫片', uploader_uid: 42 }

  // 想看用户名，得自己再查一次
  const user = await prisma.user.findUnique({ where: { uid: 42 } })
  ```

  **❷ 写 @relation，不用 include —— 声明关系但不查**
  ```prisma
  model Video {
    vid           Int     @id
    title         String
    uploader_uid  Int                                            // 真实字段（数据库里存的数字）
    uploader      User    @relation(fields: [uploader_uid], references: [uid]) // 虚拟字段（不占数据库空间）
  }

  model User {
    uid               Int     @id
    username          String
    videos            Video[]  // “一个用户有多个视频”
  }
  ```
  `@relation` 三个参数各是什么意思：
  ```
  @relation(fields: [uploader_uid], references: [uid])
              ↑                          ↑
          "我这边哪个字段"          "连你那边哪个字段"
          （Video.uploader_uid）    （User.uid）
  ```
  ```js
  // 不加 include，跟没写 @relation 一样
  const video = await prisma.video.findUnique({ where: { vid: 1 } })
  // → { vid: 1, title: '猫片', uploader_uid: 42 }  // uploader 不会出现
  ```

  **❸ @relation + include —— 自动连表查**
  ```js
  const video = await prisma.video.findUnique({
    where: { vid: 1 },
    include: { uploader: true }   // ← “帮我把上传者信息也查出来”
  })
  // → {
  //     vid: 1, title: '猫片', uploader_uid: 42,
  //     uploader: { uid: 42, username: '张三', profilePictureUrl: '...' }
  //   }
  ```
  **include 背后 Prisma 做了三步：**
  ```
  1. 查 Video 表 WHERE vid = 1 → 得到 uploader_uid = 42
  2. 翻 @relation 说明书：用 uploader_uid 去 User 表找 uid
  3. 查 User 表 WHERE uid = 42 → 把结果塞进 uploader 字段
  ```
  底层等价于两条 SQL（Prisma 在 JS 层拼接）：
  ```sql
  SELECT * FROM Video WHERE vid = 1;
  SELECT * FROM User WHERE uid = 42;
  ```

  **❹ 嵌套 include —— 多层连表**
  ```js
  // 查视频 + 上传者 + 评论（每条评论还要带评论者）
  const video = await prisma.video.findUnique({
    where: { vid: 1 },
    include: {
      uploader: true,
      comments: {
        include: { uploader: true }  // 评论也要带评论者信息
      }
    }
  })
  ```

  **❺ 命名关系 —— 同一张表被引用多次时必须起名**

  当一张表有**两个字段**都指向同一张表，Prisma 不知道哪个连哪个，必须给每条关系起名：
  ```prisma
  // Dialogue 表：一个对话有两个参与者，都是 User
  model Dialogue {
    participant1_uid  Int
    participant2_uid  Int
    participant1      User  @relation("P1", fields: [participant1_uid], references: [uid])
    participant2      User  @relation("P2", fields: [participant2_uid], references: [uid])
  }

  // User 表：用同样的名字对应
  model User {
    dialogues1  Dialogue[]  @relation("P1")  // 我是参与者1的对话
    dialogues2  Dialogue[]  @relation("P2")  // 我是参与者2的对话
  }
  ```

  **❻ @relation 规则速查表**

  | 情况 | 要起名吗 | 例子 |
  |------|----------|------|
  | A 表只有**一个字段**指向 B 表 | 不用 | Video → User（只有 `uploader_uid`） |
  | A 表有**两个字段**都指向 B 表 | 要 | Dialogue → User（`participant1_uid` + `participant2_uid`） |

  **❼ 不用 include 也行 —— 手动分步查**
  ```js
  // 完全可以不用 include，自己分两步查再拼装
  const video = await prisma.video.findUnique({ where: { vid: 1 } })
  const uploader = await prisma.user.findUnique({ where: { uid: video.uploader_uid } })
  const result = { ...video, uploader }
  ```
  对于简单查询这样写更直观，`include` 的优势在多层嵌套时才明显。

  **❽ 对比原生 SQL**

  | | 原生 SQL | Prisma |
  |---|---|---|
  | 关系定义 | 每次查询时写在 SQL 里 | 在 schema 里写一次 |
  | 查询写法 | `LEFT JOIN User ON v.uploader_uid = u.uid` | `include: { uploader: true }` |
  | 返回结构 | 扁平的一行，要自己组装嵌套 | 自动嵌套好 |
  | 灵活性 | 极高，想怎么 JOIN 怎么 JOIN | 受限于 schema 里声明的关系 |
  | 出错概率 | ON 写错字段名运行时才报错 | 字段名错了 Prisma 编译时直接报错 |

  > 所有主流 ORM（Python Django、Java Hibernate、JS Sequelize/TypeORM）都是同一个套路——**提前声明关系，查询时自动 JOIN**。只是各家语法和命名不同。

### 3. 身份鉴权 (基础：Session 模式)
- **核心包**：`npm i express-session`。
- **原理**：用户登录后，服务器在内存中开辟一块空间存储用户信息，并给浏览器发一个 `connect.sid` 的 Cookie。浏览器下次请求带上 Cookie，服务器就能认出用户。
- **实战配置**：
  ```javascript
  const session = require('express-session');

  app.use(session({
    secret: 'itheima', // 密钥，用于加密 Cookie
    resave: false,
    saveUninitialized: true
  }));

  // 1. 登录时存数据
  app.post('/api/login', (req, res) => {
    if (req.body.username === 'admin' && req.body.password === '123') {
      req.session.user = req.body; // 存入 Session
      req.session.isLogin = true;
      res.send('登录成功');
    }
  });

  // 2. 取数据 (判断是否登录)
  app.get('/api/username', (req, res) => {
    if (!req.session.isLogin) return res.status(401).send('未登录');
    res.send({ username: req.session.user.username });
  });
  ```
- **局限性**：无法跨域使用，且服务器重启后 Session 会丢失。

### 4. 身份鉴权 (进阶：JWT 模式)
- **核心包**：`npm i jsonwebtoken`。
- **JWT 原理 (三段式)**：
  1. **Header**：算法信息。
  2. **Payload**：用户信息（Base64 编码，不可放敏感数据）。
  3. **Signature**：签名（Header+Payload+密钥加密），防止篡改。
- **JWT 交互流程**：
  1. 登录成功 -> 后端 `jwt.sign()` 发放 Token。
  2. 前端存储 Token -> 之后请求放入 Header 的 `Authorization` 字段。
  3. 后端拦截器 -> `jwt.verify()` 校验密钥。
- **实战流程**：
  1. **登录成功签发 Token**：
     ```javascript
     const jwt = require('jsonwebtoken');
     const secretKey = 'itheima666';

     app.post('/api/login', (req, res) => {
       // ... 校验成功后
       const user = { id: 1, username: 'admin' };
       const token = jwt.sign(user, secretKey, { expiresIn: '10h' });
       res.send({ status: 200, token: 'Bearer ' + token });
     });
     ```
  2. **验证 Token (中间件模式)**：
     ```javascript
     const authMiddleware = (req, res, next) => {
       const token = req.headers['authorization']?.split(' ')[1];
       if (!token) return res.status(401).send('请登录');

       jwt.verify(token, secretKey, (err, decoded) => {
         if (err) return res.status(401).send('无效 Token');
         req.user = decoded; // 解析出的用户信息挂载到 req
         next();
       });
     };
     ```

### 5. RESTful API 设计规范
- **核心思想**：URL 代表资源，HTTP Method 代表动作。
- **实战标准**：
  - **GET `/users`**：获取用户列表。
  - **GET `/users/:id`**：获取指定 ID 的用户信息。
  - **POST `/users`**：新增一个用户（数据在请求体 `req.body`）。
  - **PUT `/users/:id`**：修改指定 ID 的用户信息（全量更新）。
  - **DELETE `/users/:id`**：删除指定 ID 的用户。
- **状态码规范**：
  - `200 OK`：操作成功。
  - `201 Created`：新增成功。
  - `400 Bad Request`：前端传参有误。
  - `401 Unauthorized`：未登录/Token无效。
  - `403 Forbidden`：没权限访问。
  - `404 Not Found`：资源不存在。
  - `500 Internal Server Error`：后端代码崩溃。

### 6. 环境变量与安全 (dotenv)
- **为什么需要它？**：
  - **安全性**：不要把数据库密码、JWT 密钥直接写在代码里，否则上传到 GitHub 所有人都能看到。
  - **工作原理**：`.env` 是个普通文本，Node 默认不认识。`dotenv` 模块负责读取该文件并将其内容注入到 `process.env` 内存对象中。
- **实战流程**：
  1. **安装**：`npm i dotenv`。
  2. **创建文件**：在项目根目录新建 `.env` 文件（**注意：要在 .gitignore 中忽略它**）。
     ```text
     PORT=3000
     DB_PASSWORD=root123456
     ```
  3. **在入口加载**：在 `app.js` 的第一行写上：`require('dotenv').config();`
  4. **代码中使用**：通过 `process.env.变量名` 访问。

- **协作与部署方案**：
  - **GitHub 协作**：创建一个 **`.env.example`** 文件上传，只写键名（如 `DB_PASS=`），不写具体值。别人下载后改名为 `.env` 填入自己的密码。
  - **线上部署**：线上环境通常不上传 `.env` 文件，而是直接在云平台的后台管理页面设置“环境变量”，Node 依然能通过 `process.env` 拿到。
