# Express 核心框架手册

> **一句话理解 Express**：它是在 Node.js 基础上封装的一套“Web 开发工具包”。它没有改变 Node.js 的本质，但让你写后端代码从“手动挡”变成了“自动挡”。

---

## 一、 Express 到底比原生 Node.js 多了什么？

| 功能 | 原生 Node.js (http 模块) | Express 框架 |
| :--- | :--- | :--- |
| **路由处理** | 需手动解析 `req.url` 并写大量的 `if/else` | `app.get()`, `app.post()` 语义化分发 |
| **参数解析** | 需监听 `data` 事件拼字符串，手动 `JSON.parse` | `req.query`, `req.body` 直接拿取对象 |
| **中间件** | 无法原生支持，逻辑全部堆在一起 | 插件化机制，逻辑解耦（如日志、校验、跨域） |
| **静态资源** | 需手动读取文件并设置 MIME 类型 | `app.use(express.static('public'))` 一行搞定 |
| **响应处理** | 只有 `res.end()`，需手动写 Header | `res.send()`, `res.json()`, `res.status()` 链式操作 |

---

## 二、 Express 核心概念：中间件 (Middleware)

中间件是 Express 的灵魂。它就是一个**函数**，在请求到达最终路由之前，先经过它。

### 1. 常用内置与第三方中间件
```javascript
const express = require('express');
const cors = require('cors');
const app = express();

app.use(cors());          // 解决跨域问题
app.use(express.json());  // 自动解析 POST 的 JSON 数据
app.use(express.static('public')); // 托管静态网页
```

### 2. `app.use()` 详解

**作用**：注册中间件（最核心的 Express API 之一）

**语法**：
```javascript
app.use([path,] callback [, callback...])
```

| 参数 | 说明 |
|------|------|
| `path`（可选） | 中间件生效的路径，默认为 `/`（所有请求） |
| `callback` | 中间件函数（可以是多个） |

**中间件函数的签名**：
```javascript
// 普通中间件（3 个参数）
function myMiddleware(req, res, next) {
  // req: 请求对象
  // res: 响应对象
  // next: 调用下一个中间件的函数（必须调用，否则请求会挂起）
}

// 错误处理中间件（4 个参数）
function errorMiddleware(err, req, res, next) {
  // err: 错误对象
}
```

**执行流程（中间件栈）**：

当请求到达 Express 服务器时，会按照 `app.use()` 的**注册顺序**执行中间件。

```javascript
// 中间件栈示例
app.use(cors())                    // 1. 所有请求先经过 CORS 处理
app.use(express.json())            // 2. 所有请求尝试解析 JSON 请求体
app.use('/api/auth', authRoutes)  // 3. 只有 /api/auth 开头的请求进入 authRoutes
```

**请求示例**：

**请求 1**：`POST /api/auth/login`
```
1. 经过 cors() 中间件
2. 经过 express.json() 中间件（解析 JSON 请求体到 req.body）
3. 匹配到 /api/auth，进入 authRoutes
4. authRoutes 匹配到 POST /login，执行 authController.login
```

**请求 2**：`GET /api/video`
```
1. 经过 cors() 中间件
2. 经过 express.json() 中间件（跳过，因为 GET 请求没有请求体）
3. 不匹配 /api/auth，继续往下找路由
4. 如果没有注册 /api/video 的路由，返回 404
```

### 3. `express.json()` 详解

**作用**：解析请求体中的 JSON 数据

**场景**：当前端发送 JSON 格式的数据时（比如登录请求）：
```javascript
// 前端发送的请求
POST /api/auth/login
Content-Type: application/json

{
  "username": "alice",
  "password": "secret"
}
```

**没有 `express.json()` 中间件**：
- Express 不会解析请求体
- `req.body` 等于 `undefined`
- 你无法获取前端发送的数据

**有 `express.json()` 中间件**：
- Express 会自动解析 JSON 格式的请求体
- 解析后的数据放在 `req.body` 中
- 你可以这样获取：`const { username, password } = req.body`

**底层原理**（简化版）：
```javascript
function jsonParser(req, res, next) {
  // 1. 检查 Content-Type 是否为 application/json
  if (req.headers['content-type'] !== 'application/json') {
    return next();  // 不是 JSON，跳过
  }
  
  // 2. 读取请求体（流式读取）
  let body = '';
  req.on('data', chunk => {
    body += chunk.toString();
  });
  
  // 3. 读取完成后，解析 JSON
  req.on('end', () => {
    try {
      req.body = JSON.parse(body);  // 解析后的 JSON 对象
      next();  // 调用下一个中间件
    } catch (err) {
      res.status(400).json({ error: 'Invalid JSON' });
    }
  });
}
```

### 4. 自定义中间件 (拦截器)
```javascript
app.use((req, res, next) => {
  console.log(`${new Date().toLocaleString()} - 有人来了：${req.url}`);
  // 必须调用 next()，否则请求会卡住，浏览器一直转圈
  next(); 
});
```

## 三、 路由与请求处理

### 1. `app.get()` vs `router.get()` 的区别

**核心区别**：`app.get()` 是全局路由，`router.get()` 是模块化路由（推荐）

| 对比项 | `app.get()` | `router.get()` |
|---------|--------------|----------------|
| **定义位置** | `app.js`（入口文件） | `routes/*.js`（路由文件） |
| **作用范围** | 全局 | 局部（挂载后才能访问） |
| **路径拼接** | 完整路径 | 相对路径（挂载时拼接） |
| **使用场景** | 健康检查、全局路由 | 模块化路由（推荐） |

**路径拼接示例**：

```javascript
// routes/authRoutes.js - 使用 router.get()（相对路径）
const router = express.Router()
router.get('/profile', (req, res) => { ... })  // 相对路径：/profile
router.post('/login', (req, res) => { ... })    // 相对路径：/login
module.exports = router

// app.js - 挂载路由（路径拼接）
app.use('/api/auth', authRoutes)
//          ↑ 前缀

// 实际路径：
// /api/auth/profile   → 获取用户信息
// /api/auth/login     → 登录
```

**为什么要用 `router.get()`（模块化）？**

如果把所有路由都写在 `app.js`：
- ❌ `app.js` 会变成 1000 行
- ❌ 无法多人协作（所有人改同一个文件）
- ❌ 难以维护

用 `router.get()` 模块化后：
```
Node/
├── app.js                  # 入口文件（只做中间件配置 + 路由挂载）
├── routes/
│   ├── index.js           # 路由索引（批量注册）
│   ├── authRoutes.js      # 认证路由（login, register, profile）
│   ├── videoRoutes.js     # 视频路由（list, detail, upload）
│   └── userRoutes.js     # 用户路由（profile, followers, following）
└── controllers/
    ├── authController.js  # 认证控制器
    ├── videoController.js # 视频控制器
    └── userController.js # 用户控制器
```

**优点**：
- ✅ 每个文件 < 100 行
- ✅ 多人协作不冲突
- ✅ 易维护

### 2. `req`（请求对象）详解

**作用**：获取客户端发送的所有数据（参数、请求头、请求体等）

**核心属性**：

| 属性 | 类型 | 说明 | 示例 |
|------|------|------|-------|
| `req.method` | `string` | HTTP 方法 | `'GET'`, `'POST'` |
| `req.url` | `string` | 请求路径（含 query） | `'/api/auth/login?uid=1'` |
| `req.path` | `string` | 请求路径（不含 query） | `'/api/auth/login'` |
| `req.query` | `object` | 查询参数（?后的部分） | `{ uid: '1' }` |
| `req.params` | `object` | 路径参数（`:id` 部分） | `{ id: '123' }` |
| `req.body` | `object` | 请求体（需 `express.json()` 解析） | `{ username: 'alice' }` |
| `req.headers` | `object` | 请求头 | `{ 'authorization': 'Bearer ...' }` |
| `req.ip` | `string` | 客户端 IP | `'::1'`（本地） |
| `req.cookies` | `object` | Cookie（需 `cookie-parser`） | `{ token: 'eyJ...' }` |
| `req.file` | `object` | 上传的单个文件（需 `multer`） | `{ fieldname: 'video', ... }` |
| `req.files` | `array` | 上传的多个文件（需 `multer`） | `[ {...}, {...} ]` |


**`req.headers` 的注意事项**：

```javascript
// 请求头中的字段名是小写（HTTP 标准）
req.headers['authorization']  // ✅ 正确
req.headers['Authorization']  // ❌ 错误（可能取不到）

// 如果是自定义请求头，保持原名
req.headers['x-custom-header']  // ✅ 正确
```

### 3. `res`（响应对象）详解

**作用**：向客户端返回响应（JSON、HTML、文件等）

**核心方法**：

| 方法 | 说明 | 示例 |
|------|------|-------|
| `res.json(body)` | 返回 JSON 响应（自动设置 `Content-Type: application/json`） | `res.json({ code: 200, data: user })` |
| `res.send(body)` | 返回任意类型响应（自动设置 `Content-Type`） | `res.send('Hello')`, `res.send({...})` |
| `res.status(code)` | 设置 HTTP 状态码（链式调用） | `res.status(201).json({...})` |
| `res.sendStatus(code)` | 设置状态码并发送（等价于 `res.status(code).send()`) | `res.sendStatus(404)` |
| `res.set(header, value)` | 设置响应头 | `res.set('X-Custom-Header', 'value')` |
| `res.redirect(path)` | 重定向 | `res.redirect('/login')` |
| `res.download(path)` | 下载文件（强制下载） | `res.download('./files/report.pdf')` |
| `res.sendFile(path)` | 发送文件（用于文件服务器） | `res.sendFile('./public/index.html')` |

**链式调用**（Express 的特色）：

```javascript
// 方式 1：分开写
res.status(201)              // 设置状态码
res.json({                   // 发送 JSON 响应
  code: 201,
  message: '创建成功',
  data: { uid: 1 }
})
```

// 方式 2：链式调用（推荐）
res
  .status(201)              // 设置状态码
  .set('X-Custom-Header', 'value')  // 设置响应头
  .json({                   // 发送 JSON 响应
    code: 201,
    message: '创建成功',
    data: { uid: 1 }
  })
```

**使用示例**（来自你的 `authController.js`）：

```javascript
// 成功响应（201 Created）
return res.status(201).json({
  code: 201,
  message: '注册成功',
  data: {
    token: token,
    uid: newUser.uid
  }
})

// 错误响应（400 Bad Request）
return res.status(400).json({
  code: 400,
  message: '用户名和密码不能为空',
  data: null
})

// 认证失败（401 Unauthorized）
return res.status(401).json({
  code: 401,
  message: 'Token 无效或已过期',
  data: null
})
```

### 4. `err`（错误对象）详解

**作用**：在 `catch(err)` 或错误处理中间件中获取错误信息

**标准 `Error` 对象格式**：

```javascript
{
  message: '错误信息',      // 错误描述（必填）
  stack: 'Error: 错误信息\n    at ...',  // 调用栈（用于调试）
  name: 'Error'            // 错误类型（可选）
}
```

**不同来源的错误格式**：

| 错误来源 | `err` 的格式 | 示例 |
|-----------|----------------|-------|
| **数据库错误**（Prisma） | Prisma 自定义错误对象 | `PrismaClientKnownRequestError` |
| **密码加密错误**（bcrypt） | 标准 `Error` 对象 | `Error: data and salt arguments required` |
| **Token 验证错误**（JWT） | 标准 `Error` 对象 | `JsonWebTokenError: invalid token` |
| **主动抛出错误** | 自定义 `Error` 对象 | `new Error('用户名已存在')` |

**Prisma 错误的详细格式**：

```javascript
try {
  const user = await prisma.user.create({
    data: { username: 'alice', password: '...' }
  })
} catch (error) {
  console.log(error instanceof Prisma.PrismaClientKnownRequestError)  // true
  console.log(error.code)    // P2002（唯一约束失败，如用户名重复）
  console.log(error.meta)    // { target: ['username'] }
  console.log(error.message) // "Unique constraint failed on the fields: (`username`)"
}
```

**常见 Prisma 错误码**：
- `P2002`：唯一约束失败（用户名重复）
- `P2025`：记录不存在（删除/更新时）
- `P2003`：外键约束失败

**JWT 错误的详细格式**：

```javascript
try {
  const decoded = jwt.verify(token, JWT_SECRET)
} catch (error) {
  console.log(error.name)    // "JsonWebTokenError" 或 "TokenExpiredError"
  console.log(error.message) // "invalid token" 或 "jwt expired"
}
```

**JWT 错误类型**：
- `JsonWebTokenError`：Token 格式错误、签名验证失败
- `TokenExpiredError`：Token 已过期
- `NotBeforeError`：Token 尚未生效

**自定义错误对象**（推荐）：

```javascript
// 自定义错误类（支持状态码）
class AppError extends Error {
  constructor(message, status = 500) {
    super(message)
    this.status = status
    this.name = this.constructor.name  // 类名（用于 instanceof 判断）
  }
}

// 使用自定义错误
const err = new AppError('用户名已存在', 400)
console.log(err.message)  // "用户名已存在"
console.log(err.status)   // 400
console.log(err instanceof AppError)  // true
```

**在控制器中使用**：

```javascript
// controllers/authController.js

const register = async (req, res, next) => {
  try {
    // ... 业务逻辑

    // 主动抛出错误（自定义）
    if (existingUser) {
      const err = new Error('用户名已存在')
      err.status = 400  // 自定义状态码
      throw err  // 进入 catch
    }

  } catch (error) {
    // error 的格式：
    // {
    //   message: '用户名已存在',
    //   status: 400,
    //   stack: 'Error: 用户名已存在\n    at ...'
    // }

    // 传递给错误处理中间件
    next(error)
  }
}
```

## 四、 综合实战：一个标准的后端入口 (app.js)

### 1. 基础版（所有路由写在 app.js）

```javascript
const express = require('express');
const cors = require('cors');
const app = express();

// --- 1. 配置全局中间件 ---
app.use(cors());
app.use(express.json());

// --- 2. 业务路由 ---
app.post('/api/register', (req, res) => {
  // 逻辑处理...
  res.status(201).json({ msg: '注册成功' });
});

// --- 3. 统一错误处理中间件 ---
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).send('服务器出错了！');
});

app.listen(3000, () => {
  console.log('Server is running at http://localhost:3000');
});
```

### 2. 进阶版（模块化路由 + 批量注册）

**目录结构**：

```
Node/
├── app.js                  # 入口文件
├── routes/
│   ├── index.js           # 路由索引（批量注册）
│   ├── authRoutes.js      # 认证路由
│   └── videoRoutes.js     # 视频路由
└── controllers/
    ├── authController.js  # 认证控制器
    └── videoController.js # 视频控制器
```

**`routes/index.js`**（路由索引）：

```javascript
// 引入所有路由模块
const authRoutes = require('./authRoutes')
const videoRoutes = require('./videoRoutes')

// 导出路由配置数组（按注册顺序）
module.exports = [
  { path: '/api/auth', router: authRoutes },
  { path: '/api/video', router: videoRoutes },
]
```

**`app.js`**（使用批量注册）：

```javascript
const express = require('express')
const cors = require('cors')
const routes = require('./routes')  // 引入路由索引

const app = express()

// --- 1. 配置全局中间件 ---
app.use(cors())
app.use(express.json())

// --- 2. 路由注册（批量注册） ---
routes.forEach(({ path, router }) => {
  app.use(path, router)
})

// --- 3. 健康检查接口 ---
app.get('/', (req, res) => {
  res.json({ code: 200, message: 'API is running' })
})

// --- 4. 统一错误处理中间件 ---
app.use((err, req, res, next) => {
  console.error('全局错误:', err.stack)
  res.status(500).json({
    code: 500,
    message: '服务器内部错误',
    data: null
  })
})

// --- 5. 启动服务器 ---
const PORT = process.env.PORT || 3000
app.listen(PORT, () => {
  console.log(`✅ 后端服务已启动: http://localhost:${PORT}`)
})
```

### 3. 错误处理中间件详解

**作用**：统一处理所有错误（避免在每个控制器中重复写错误处理代码）

**四个参数是怎么来的？**

Express 根据参数个数判断是否为错误处理中间件：

```javascript
// 普通中间件（3 个参数）
function myMiddleware(req, res, next) { ... }

// 错误处理中间件（4 个参数）← Express 会根据参数个数识别
function errorMiddleware(err, req, res, next) { ... }
```

**`err` 是谁传的？**

答案：是前面的中间件/路由调用 `next(err)` 传的。

**完整流程**：

```
请求 → 中间件 A → 中间件 B → 路由 → 错误处理中间件
         ↓           ↓           ↓
       正常        抛出异常     调用 next(err)
       next()      catch 捕获
                    ↓
                  next(error)
                    ↓
                  进入错误处理中间件（err 有值）
```

**`next()` 的三种用法**：

| 调用方式 | 作用 | 下一个中间件类型 |
|-----------|------|------------------|
| `next()` | 传递控制权（正常） | 普通中间件（3 个参数） |
| `next('route')` | 跳过当前路由，进入下一个路由 | 普通中间件 |
| `next(err)` | 传递错误 | 错误处理中间件（4 个参数） |

**错误处理中间件的执行顺序**：

Express 按照 `app.use()` 的注册顺序，找到**第一个 4 个参数的中间件**。

```javascript
// 错误处理中间件 A（4 个参数）
app.use((err, req, res, next) => {
  console.log('错误处理 A')
  next(err)  // 继续传递错误
})

// 错误处理中间件 B（4 个参数）
app.use((err, req, res, next) => {
  console.log('错误处理 B')
  res.status(500).json({ error: err.message })
})
```

**执行顺序**：

```
错误发生 → 错误处理 A → 调用 next(err) → 错误处理 B → 返回响应
```

### 4. 优化错误处理中间件（支持自定义状态码）

**基础版**（当前代码）：

```javascript
app.use((err, req, res, next) => {
  console.error('全局错误:', err.stack)
  res.status(500).json({
    code: 500,
    message: '服务器内部错误',
    data: null
  })
})
```

**问题**：
- ❌ 所有错误都返回 500（不够精细）
- ❌ 无法区分不同类型的错误（数据库错误、JWT 错误等）

---

**进阶版**（支持自定义状态码）：

```javascript
app.use((err, req, res, next) => {
  // 1. 记录错误日志（可以写入文件）
  console.error(`${new Date().toISOString()} - ${err.stack}`)

  // 2. 根据错误类型返回不同状态码
  let status = 500
  let message = '服务器内部错误'

  // Prisma 错误
  if (err instanceof Prisma.PrismaClientKnownRequestError) {
    status = 400
    message = '数据库操作失败'
  }

  // JWT 错误
  if (err.name === 'JsonWebTokenError' || err.name === 'TokenExpiredError') {
    status = 401
    message = 'Token 无效或已过期'
  }

  // 自定义错误（有 status 属性）
  if (err.status) {
    status = err.status
    message = err.message
  }

  // 3. 返回错误响应
  res.status(status).json({
    code: status,
    message: message,
    data: null
  })
})
```

**控制器中可以这样用**：

```javascript
// 主动传递错误（带状态码）
if (!username || !password) {
  const err = new Error('用户名和密码不能为空')
  err.status = 400  // ← 自定义状态码
  next(err)
  return
}
```

## 五、 响应处理与状态码 (Response & Status Codes)

### 1. 为什么要手动设置状态码？
- **语义化**：让前端通过数字（如 401, 404）直接判断结果，而不需要解析 JSON 内容。
- **自动化**：前端拦截器（如 Axios Interceptor）可以根据状态码统一处理（如 401 自动跳转登录）。

### 2. 常见场景举例
```javascript
// 场景 1：创建成功 (201 Created)
app.post('/api/todo', (req, res) => {
  res.status(201).json({ msg: '任务已创建' });
});

// 场景 2：前端传参有误 (400 Bad Request)
app.get('/api/user', (req, res) => {
  if (!req.query.id) {
    return res.status(400).json({ msg: '必须提供用户ID' });
  }
  res.json({ id: req.query.id, name: '张三' });
});

// 场景 3：未登录/Token无效 (401 Unauthorized)
app.get('/api/profile', (req, res) => {
  // 假设鉴权失败
  res.status(401).json({ msg: '令牌无效，请重新登录' });
});

// 场景 4：权限不足 (403 Forbidden)
app.delete('/api/admin/system', (req, res) => {
  // 假设不是管理员
  res.status(403).json({ msg: '权限不足，无法删除系统配置' });
});

// 场景 5：服务器崩溃 (500 Internal Server Error)
app.get('/api/error', (req, res) => {
  try {
    // 可能会报错的代码
    throw new Error('数据库连接断开');
  } catch (err) {
    res.status(500).json({ msg: '服务器内部错误' });
  }
});
```
