# 跨域（CORS）故障深度剖析与通关指南

在自建服务期间，项目曾遭遇了一场教科书级别的幽灵跨域（CORS）Bug。由于前置 AI 辅助工具的过度封装与硬编码，导致前后端调用链路发生偏离。以下是对该故障的复盘、CORS 底层原理的剖析，以及一劳永逸的工业级解决方案总结。

## 🛑 一、故障复盘：那个被硬编码偷走的"幻觉"

### 1. 故障现象

在 Ubuntu 云服务器上配置好 Nginx 并部署了 React 前端（dist）后，浏览器控制台疯狂抛出红色报错：

```text
Access to XMLHttpRequest at 'https://tcloudbase.com' 
from origin 'http://124.221.133.62' has been blocked by CORS policy: 
No 'Access-Control-Allow-Origin' header is present on the requested resource.
```

即使删除了本地 Node.js 后端的 cors 插件，报错依然存在。

### 2. 核心病因

**AI 助手的暗雷**：前置 Agent 为了图省事，悄悄在前端代码（Axios 配置文件）中将 baseURL 硬编码为了腾讯云微托管的线上域名（tcloudbase.com）。

**正常的幻觉**：因为微托管的Node使用cors中间件放行了所有跨域请求，导致 React 一直在越过本地服务器、直接把请求发给云托管的服务器，这导致了长时间的"断开访问一段时间后，后端无法访问，需要登录ssh唤醒后端"的错觉，尝试各种方式保活后端但都没有效果，因为云服务器上的前端实际上一直在访问云托管的后端，而该后端断开一段时间后就会休眠。

## 🔍 二、CORS 原理剖析：楚门的世界

CORS（跨域资源共享）根本不是后端服务器的限制，而是浏览器联手制定的一场安全保护机制。

### 1. 为什么要有跨域限制？

为了防范 CSRF（跨站请求伪造）攻击。如果没有"同源策略"，当你在浏览器登录银行网站 bank.com 的同时打开了钓鱼网站 bad.com，钓鱼网站的 JS 代码就能利用你的浏览器身份直接向银行发送转账请求。

### 2. 浏览器的"截胡"两部曲

当域名、协议、端口任何一个不一致时，浏览器会启动看门狗机制：

**第一步：暗中观察（普通请求）**：浏览器不会阻止 GET/POST 请求发出去，后端也会正常处理并返回数据。但在数据进门的一瞬间，浏览器发现响应头里没有 Access-Control-Allow-Origin，会当场将数据撕毁销毁，并对前端谎报 undefined / net::ERR_FAILED。

**第二步：先斩后奏（复杂请求）**：在发送真实请求前，浏览器会先发送一个 OPTIONS 预检请求。去询问后端："你允许我访问吗？"。如果后端没配 CORS，真实请求连发出去的机会都没有，直接胎死腹中。

### 3. 为什么 app.use(cors()) 插件不能随便用？

Node.js 的 cors 插件默认是一张全通票（Wildcard *）。它会在响应头中强行注入 `Access-Control-Allow-Origin: *`。这等同于全网不设防，在涉及腾讯云 ASR 语音识别（按量扣费、实打实的人民币）的商业项目中，盲目开启 * 会导致接口被黑客黑产恶意盗刷，面临巨额账单风险。

## 🛠️ 三、完美解决方案：Nginx 反向代理

CORS 只在浏览器里生效，服务器之间通信完全没有 CORS 的概念。利用这一特性，我们通过 Nginx 搭建"统一看门狗"，让前后端在浏览器眼里完全同源。

### 1. 架构演进

**【改造前：跨域翻车】**

```
React 前端 (124.221...80端口) ───(浏览器硬拦截 ❌)───> 腾讯云托管 / 3000端口
```

**【改造后：完美同源 ⭕】**

```
React 前端 (124.221...80端口) ───(/api 相对路径)───> Nginx (同源放行)
                                                        │ (后台悄悄转发)
                                                        ▼
                                                  本地 Node.js (127.0.0.1:3000)
```

### 2. 落地实施三步走

**第一步：修改前端 Axios 配置（切断硬编码）**

在前端源码中，将 baseURL 彻底改为相对路径 `/api`。

```javascript
// src/utils/request.js
axios.defaults.baseURL = '/api'; 
```

修改后必须重新执行 `npm run build` 打包并覆盖服务器的 dist 目录。

**第二步：配置 Nginx 核心转发（/etc/nginx/sites-enabled/default）**

在 Nginx 配置文件中加入反向代理，且不加任何跨域头：

```nginx
server {
    listen 80 default_server;
    root /var/www/myapp/dist; # 前端打包路径
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    # 🛠️ 精准拦截所有 /api/ 请求，转发给本地 Node.js 后端
    location /api/ {
        proxy_pass http://127.0.0.1:3000; # 末尾无斜杠，带 /api 传给后端
        
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
        
        # 透传客户端真实 IP
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

配置完成后执行 `sudo nginx -t && sudo systemctl reload nginx` 生效。

**第三步：清理后端代码**

前往 `/var/www/myapp/backend/app.js`，彻底删除或注释掉 `app.use(cors())`，让 Node.js 只专注于业务本身。

为了防止 SSH 断开后进程关闭，使用 PM2 长期守护后端：

```bash
pm2 start app.js --name "ai-subtitle-backend"
```
