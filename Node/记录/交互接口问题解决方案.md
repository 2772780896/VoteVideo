# 交互接口问题解决方案

## 一、问题背景

在实现用户交互功能（点赞、收藏）后，发现以下异常行为：

1. **操作后状态不回显**：用户点击点赞/收藏按钮后，前端立即收到"已收藏"等提示，但按钮状态未更新
2. **状态不同步**：切换到其他页面再返回时，按钮状态恢复为"未操作"状态
3. **Profile页面不显示**：用户的点赞/收藏内容未在个人资料页面显示

## 二、问题分析

### 根本原因

**后端返回内容数据时未携带当前用户的交互状态**

前端交互按钮的状态依赖于后端返回数据中的 `isLiked`、`isFavourited` 字段。但在修改前，后端在查询视频、文章、动态列表/详情时，只返回内容本身的基本信息，没有查询当前用户是否对已返回内容执行过点赞/收藏操作。

这导致：
- 前端无法正确渲染按钮的初始状态
- 用户操作后，前端乐观更新了状态，但重新请求数据时又收到了错误的状态
- Profile页面无法筛选出当前用户交互过的内容

### 技术细节

**修改前的数据流向：**
```
前端请求 → 控制器 → 服务层 → 数据库查询（无关联查询）
→ 数据转换（无isLiked/isFavourited字段）
→ 返回前端（缺少交互状态）
```

**问题表现：**
1. 前端首次渲染时，`isLiked` 和 `isFavourited` 为 `undefined`
2. 用户点击按钮 → 前端乐观更新状态 → 重新请求数据
3. 后端仍返回无状态数据 → 前端状态被错误覆盖
4. 出现"状态闪烁"或"状态丢失"现象

## 三、解决方案

### 核心思路

**全模块查询时关联当前用户的互动记录，数据转换时补充交互状态字段**

具体实现：
1. 查询时通过Prisma的 `include` 关联查询当前用户的 `likes` 和 `favourites` 记录
2. 数据转换时根据关联查询结果判断并返回 `isLiked`、`isFavourited` 状态
3. 控制器层从 `req.user?.uid` 获取当前用户ID，传递给服务层

### 修改范围

涉及3个模块（视频、文章、动态），共7个文件：

| 模块 | 服务层 | 控制器层 |
|------|---------|----------|
| 通用 | `baseService.js` | - |
| 视频 | `videoService.js` | `videoController.js` |
| 文章 | `essayService.js` | `essayController.js` |
| 动态 | `postService.js` | `postController.js` |

## 四、具体修改内容

### 1. 通用服务层 - `baseService.js`

#### 修改点1：MODULE_CONFIG 添加关联查询配置

为 `video`、`essay`、`post` 三个模块的 `includeConfig` 添加 `likes` 和 `favourites` 关联查询：

```javascript
// Node/services/baseService.js (第278-301行)
video: {
  // ... 其他配置
  includeConfig: {
    uploader: { /* ... */ },
    videoTags: { /* ... */ },
    // 新增：添加 likes 和 favourites 关联
    likes: {
      select: { uid: true }
    },
    favourites: {
      select: { uid: true }
    }
  }
}
```

同理修改 `essay` 和 `post` 模块的配置（`post` 只有 `likes`，因为动态不支持收藏）。

#### 修改点2：getListData 方法支持 currentUid 参数

```javascript
// Node/services/baseService.js (第118-173行)
async getListData(options = {}) {
  const {
    page = 1,
    element = 16,
    sort = `-${defaultSortField}`,
    q = '',
    where: extraWhere = {},
    include: overrideInclude = null,
    currentUid = null  // 新增参数
  } = options
  
  // ... 查询逻辑
  
  // 数据转换时传递 currentUid
  if (transformFunction) {
    transformedItems = items.map(item => {
      if (transformFunction.length >= 2) {
        return transformFunction(item, { currentUid })
      } else {
        return transformFunction(item)
      }
    })
  }
  
  return { data: transformedItems, total: total }
}
```

#### 修改点3：getItemData 方法支持 currentUid 参数

```javascript
// Node/services/baseService.js (第184-216行)
async getItemData(id, options = {}) {
  const {
    include: overrideInclude = null,
    throwIfNotFound = false,
    currentUid = null  // 新增参数
  } = options
  
  // ... 查询逻辑
  
  // 数据转换时传递 currentUid
  if (item && transformFunction) {
    if (transformFunction.length >= 2) {
      return transformFunction(item, { currentUid })
    } else {
      return transformFunction(item)
    }

```

**关键设计**：通过检查 `transformFunction.length >= 2` 来判断转换函数是否接受第二个参数（options），保证向后兼容。

### 2. 视频模块 - `videoService.js`

#### 修改点1：transformVideoData 函数支持 options 参数

```javascript
// Node/services/videoService.js (第20-67行)
const transformVideoData = (video, options = {}) => {
  const {
    includeVideoUrl = false,
    includeTags = true,
    currentUid = null  // 新增：当前用户ID
  } = options
  
  const videoData = {
    vid: video.vid,
    coverUrl: video.coverUrl,
    title: video.title,
    viewCount: formatCount(video.viewCount),
    commentCount: formatCount(video.commentCount),
    duration: video.duration,
    date: formatDate(video.date),
    // 新增：检查当前用户是否已点赞/收藏
    isLiked: currentUid ? video.likes?.some(like => like.uid === currentUid) || false : false,
    isFavourited: currentUid ? video.favourites?.some(fav => fav.uid === currentUid) || false : false
  }
  
  // ... 其他逻辑
  
  return videoData
}
```

#### 修改点2：getVideoDetailData 方法接收并传递 currentUid

```javascript
// Node/services/videoService.js (第116-137行)
const getVideoDetailData = async (vid, currentUid = null) => {
  // 增加播放量
  await prisma.video.update({
    where: { vid: parseInt(vid) },
    data: { viewCount: { increment: 1 } }
  })
  
  // 重新读取更新后的视频数据（传递 currentUid）

    throwIfNotFound: true,
    currentUid

    includeVideoUrl: true, 
    includeTags: true,
    currentUid
  })
  
  return videoItem
}
```

## 五、数据流图

### 修改后的数据流

```
前端请求
  ↓
控制器 (从 req.user?.uid 获取 currentUid)
  ↓
服务层 (接收 currentUid 参数)
  ↓
数据库查询 (include: { likes: { select: { uid: true } }, favourites: { select: { uid: true } } })
  ↓
数据转换 (根据 currentUid 判断 isLiked/isFavourited)
  ↓
返回前端 (包含交互状态字段)
```

## 七、注意事项

### 1. 性能考虑

- **关联查询开销**：为每个查询添加 `likes` 和 `favourites` 关联会增加数据库查询开销
- **优化建议**：如果用户量较大，可以考虑：
  - 使用Redis缓存用户交互状态
  - 只在详情页和Profile页面返回交互状态
  - 列表页通过前端本地存储维护交互状态

### 2. 向后兼容

- `transformFunction.length >= 2` 的检查保证了向后兼容
- 如果某个模块的转换函数不接受第二个参数，不会报错

### 3. 未登录用户处理

- `currentUid` 为 `null` 时，`isLiked` 和 `isFavourited` 返回 `false`
- 前端应根据这两个字段渲染正确的按钮状态

### 4. 数据库索引

- 确保 `likes` 和 `favourites` 表的 `uid` 字段有索引
- 确保 `video_likes`、`video_favourites` 等中间表有复合索引

## 八、总结

本次修改通过**在查询时关联当前用户的交互记录**，解决了交互状态不同步的问题。核心改动是：

1. **通用服务层**：支持 `currentUid` 参数，传递给数据转换函数
2. **模块服务层**：更新转换函数，根据 `currentUid` 判断交互状态
3. **控制器层**：从 `req.user?.uid` 获取当前用户ID，传递给服务层

该方案保证了前后端数据的一致性，用户体验更加流畅。
