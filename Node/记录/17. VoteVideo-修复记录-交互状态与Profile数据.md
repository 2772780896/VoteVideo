## 问题背景

上一轮修复（17 文件）解决了前后端接口格式不匹配的问题，但部署后用户反馈：

1. 收藏视频后回到视频页，收藏状态不显示（`isFavourited` 始终为 `false`）
2. 再次点击收藏报错 `400 Bad Request: 已经收藏过了`
3. Profile 页面的收藏、历史、对话等列表全部为空

## 根因分析

### 根因一：内容请求不携带 JWT Token

前端 `request.js` 拦截器仅在 `needToken: true` 时才附加 Authorization Header。而 `fetchItem` 和 `fetchItemList`（视频列表、详情、文章、动态等）都没有设置该标志，导致这些请求不携带 Token。

后端虽然已正确实现了 `req.user?.uid || null` 的可选认证逻辑，但因为请求中没有 Token，`req.user` 始终为 `undefined`，交互状态查询（`checkVideoInteractions` 等）永远不会被执行，所有 `isLiked` / `isFavourited` 返回 `false`。

用户收藏操作本身是成功的（数据库已写入），但页面刷新后看不到收藏状态，再次点击触发了重复插入（P2002 → "已经收藏过了"）。

### 根因二：Profile 查询的 type 字段不匹配

前端 `getProfileSubdata` 传 `dataType` 为复数形式（`videos` / `posts` / `essays`），但数据库中 `UserFavourite` 和 `UserHistory` 表的 `type` 字段存储的是单数形式（`video` / `post` / `essay`）。

```
前端请求：GET /api/user/profile/favourites/videos
控制器查询：prisma.userFavourite.findMany({ where: { type: 'videos' } })  ← 匹配 0 条
数据库实际：type = 'video'
```

## 修复清单

### 一、新增 optionalAuth 中间件

**文件：** `Node/middleware/authMiddleware.js`（+28 行）

新增 `optionalAuth` 中间件，与现有 `needToken` 并列导出：

```js
const optionalAuth = (req, res, next) => {
  const authHeader = req.headers.authorization
  if (!authHeader) return next()

  const token = authHeader.split(' ')[1]
  if (!token) return next()

  try {
    req.user = jwt.verify(token, JWT_SECRET)
  } catch (error) {
    // Token 无效或过期 → 静默忽略，当作未登录处理
  }
  next()
}
```

**设计意图：** 内容浏览类路由（视频、文章、动态等）对匿名访客和登录用户都开放，但登录用户可以获得交互状态（是否已点赞/收藏）。`optionalAuth` 只在 Token 存在且有效时设置 `req.user`，否则静默跳过。


## 数据流修复前后对比

### 修复前

```
用户点击收藏 → interact API(POST, 带Token) → DB写入成功 ✓
用户浏览视频页 → fetchItem(GET, 不带Token) → req.user=undefined → isFavourited=false ✗
用户看到未收藏状态 → 再次点击 → POST → P2002 "已经收藏过了" ✗
Profile 收藏列表 → dataType='videos' → type='videos' → 匹配 0 条 ✗
```

### 修复后

```
用户点击收藏 → interact API(POST, 带Token) → DB写入成功 ✓
用户浏览视频页 → fetchItem(GET, 带Token) → optionalAuth解码 → req.user={uid} → checkVideoInteractions → isFavourited=true ✓
用户看到已收藏状态 → 再次点击 → DELETE → 取消收藏成功 ✓
Profile 收藏列表 → dataType='videos' → dbType='video' → 正确匹配记录 ✓
```

## 部署注意事项

1. 重启后端服务（`node app.js` 或 PM2）使新的 `optionalAuth` 中间件生效
2. 前端需要重新构建（`npm run build`）使 `request.js` 的改动生效
3. 如果浏览器有旧 Token（如上次 db reset 后的残留），建议清除 Cookie 后重新登录
4. 数据库结构无需变更（`prisma db push` 在上一轮已同步）
