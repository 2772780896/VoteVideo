# VoteVideo — 通用服务层实施总结（函数式实现）

> 记录通用服务层架构的实施过程、代码修改、以及后续建议。
> **重要变更**：已从类实现改为函数式实现，移除所有类定义，改用服务工厂函数和纯函数。

通用服务层, createService, 函数式编程, 代码复用, 分层架构, 实施总结

## 涉及文件

## 一、实施目标

### 1.1 初始问题
1. **代码重复**：各控制器中重复定义工具函数、数据库操作、数据转换逻辑
2. **可维护性差**：数据库操作分散在控制器中，模型变更需要修改多个地方
3. **难以测试**：控制器包含太多职责，无法独立测试
4. **类实现复杂**：使用类导致大量`this`引用，代码难以理解

### 1.2 实施目标
1. **创建服务工厂函数**（`createService`）：封装通用的分页、排序、搜索、CRUD逻辑
2. **使用映射表配置**：通过`MODULE_CONFIG`定义各模块的字段映射和配置
3. **各模块服务层使用纯函数**：减少重复代码，提高复用性
4. **修改控制器层**：使其调用服务层函数，简化控制器逻辑
5. **函数式实现**：移除所有类定义，使用纯函数和工厂函数

## 二、实施内容

### 2.1 创建服务工厂函数

**文件**：`services/baseService.js`

**职责**：
- 封装通用的分页、排序、搜索逻辑
- 提供通用的CRUD操作（getListData, getItemData, count, create, update, delete）
- 定义工具函数（parseSortParam, parsePagination, formatCount, formatDate）
- 定义模块配置映射表（`MODULE_CONFIG`）
- 提供`createService`工厂函数，用于创建各模块的基础服务

**核心代码**：
```javascript
/**
 * 创建通用服务
 * @param {string} modelName - Prisma模型名称（如 'video', 'essay'）
 * @param {object} config - 配置对象
 * @returns {object} 包含通用服务方法的对象
 */
const createService = (modelName, config = {}) => {
  const model = prisma[modelName]
  
  if (!model) {
    throw new Error(`Prisma模型 "${modelName}" 不存在`)
  }
  
  // 解构配置
  const {
    idField = 'id',
    searchField = 'title',
    defaultSortField = 'createdAt',
    includeConfig = {},
    transformFunction = null
  } = config
  
  // 返回包含通用服务方法的对象
  return {
    async getListData(options = {}) { ... },
    async getItemData(id, options = {}) { ... },
    async count(where = {}) { ... },
    async create(data) { ... },
    async update(id, data) { ... },
    async delete(id) { ... }
  }
}
```

### 2.2 创建各模块服务层

#### Video服务层（`services/videoService.js`）
- 调用`createService`创建基础服务
- 定义数据转换函数（`transformVideoData`, `transformToCarouselItems`）
- 添加`getCarouselData`函数（轮播图数据）
- 添加`getVideoDetailData`函数（增加播放量逻辑）
- 添加`getRelatedVideosData`函数（相关视频推荐）

#### Essay服务层（`services/essayService.js`）
- 调用`createService`创建基础服务
- 定义数据转换函数（`transformEssayData`）
- 添加`getEssayDetailData`函数（可增加阅读量逻辑）

#### Post服务层（`services/postService.js`）
- 调用`createService`创建基础服务
- 定义数据转换函数（`transformPostData`）
- 添加`createPost`函数（处理JSON字段：pictureList, videoList）

#### Comment服务层（`services/commentService.js`）
- **特殊点**：需要处理递归子评论
- 定义递归数据转换函数（`transformCommentData`）
- 添加`getCommentListData`函数（只包含顶级评论，递归查询子评论）
- 添加`createComment`函数（处理回复逻辑，增加父评论的subCommentCount）

#### Tag服务层（`services/tagService.js`）
- **特殊点**：Tag没有上传者信息，数据结构较简单
- 调用`createService`创建基础服务
- 定义数据转换函数（`transformTagData`）
- 添加`getTagByName`函数（根据标签名称获取）
- 添加`getHotTags`函数（获取热门标签）

### 2.3 修改控制器层

#### 修改原则
1. **移除所有数据库操作**：控制器不再直接调用`prisma`
2. **移除所有数据转换逻辑**：控制器不再手动映射字段
3. **移除所有工具函数定义**：统一使用服务层提供的函数
4. **简化错误处理**：使用统一的错误处理函数
5. **调用服务层函数**：直接调用服务层导出的函数，无需实例化

#### 修改后的控制器结构
```javascript
// controllers/videoController.js（示例）
const { getCarouselData, getVideoListData, getVideoDetailData } = require('../services/videoService')

const handleControllerError = (res, error, customMessage, statusCode) => { ... }
const sendSuccessResponse = (res, data, message, statusCode) => { ... }

const getCarousel = async (req, res) => {
  try {
    const number = parseInt(req.query.number) || 5
    const carouselItems = await getCarouselData(number)  // ← 直接调用函数
    return sendSuccessResponse(res, carouselItems)
  } catch (error) {
    return handleControllerError(res, error, '获取轮播图错误')
  }
}

// ... 其他控制器函数
```

## 四、分层架构优点

### 4.1 代码复用
- **服务层函数可被多个控制器复用**（如`transformVideoData`可被`videoController`和`commentController`复用）
- **工具函数集中管理**，避免重复定义
- **工厂函数复用通用逻辑**，`createService`可被所有模块复用

### 4.2 易于维护
- **数据库模型变更**：只需修改服务层，控制器不受影响
- **数据格式变更**：只需修改服务层的数据转换函数
- **业务逻辑变更**：集中在服务层，便于定位和修改
- **函数式实现**：无`this`困扰，代码更易于理解和维护

### 4.3 易于测试
- **服务层函数可独立进行单元测试**（不需要HTTP请求）
- **控制器可mocking服务层**，进行集成测试
- **纯函数易于测试**：无副作用，输入确定，输出确定

### 4.4 代码清晰
- **职责分离**：每层只负责自己的职责
- **代码简洁**：控制器从平均150行减少到平均80行
- **易于理解**：函数式编程更直观，新成员可以快速理解代码结构
- **无类开销**：无需理解原型链、继承、`this`绑定等复杂概念

## 五、函数式实现 vs 类实现

### 5.1 函数式实现的优点
✅ **无`this`困扰**：所有函数直接使用`prisma.video`、`prisma.essay`等，直观清晰
✅ **代码更简洁**：减少了类定义、构造函数、`bind`等样板代码
✅ **易于理解**：函数式编程更直观，易于组合和测试
✅ **无需实例化**：直接调用函数，无需`new ClassName()`
✅ **易于组合**：函数可以自由组合，创建新的业务逻辑

### 5.2 类实现的缺点
❌ **`this`动态绑定**：容易出错，特别是在回调函数和异步函数中
❌ **需要绑定**：类方法需要`.bind(this)`或箭头函数
❌ **继承复杂**：深层继承链难以维护
❌ **实例化开销**：需要`new ClassName()`
❌ **不适合无状态服务**：服务层主要是无状态的函数，不需要类

### 5.3 代码示例对比

**类实现（之前）**：
```javascript
class VideoService extends BaseService {
  constructor() {
    super('video', MODULE_CONFIG.video)
    this.transformFunction = (video) => transformVideoData(video)
  }
  
  async getCarouselData(number) {
    const videos = await this.model.findMany(...)  // ← 需要this
    return videos.map(v => this.transformVideoData(v))  // ← 又是this
  }
}

// 需要实例化并绑定
const videoService = new VideoService()
module.exports = {
  getCarouselData: videoService.getCarouselData.bind(videoService)
}
```

**函数式实现（现在）**：
```javascript
// 调用工厂函数创建基础服务
const baseService = createService('video', config)

// 直接定义函数，无this
async function getCarouselData(number) {
  const videos = await prisma.video.findMany(...)  // ← 直接用prisma
  return videos.map(v => transformVideoData(v))  // ← 直接调用函数
}

// 直接导出函数
module.exports = {
  getCarouselData,
  getVideoListData: baseService.getListData
}
```

## 九、附录：从类到函数的重构步骤

### 9.1 重构步骤
1. **识别类中的状态**：如果类只是无状态的函数集合，可以改为纯函数模块
2. **识别类的实例化**：如果类只是被实例化一次，可以改为工厂函数或纯函数
3. **识别`this`的使用**：将`this.xxx`改为直接引用变量或函数
4. **识别继承关系**：如果继承只是复用代码，可以改为组合或工厂函数

### 9.2 重构示例

**步骤1：识别问题**
```javascript
// 类实现
class VideoService extends BaseService {
  constructor() {
    super('video', MODULE_CONFIG.video)
    this.transformFunction = (video) => transformVideoData(video)
  }
  
  async getCarouselData(number) {
    const videos = await this.model.findMany(...)  // ← this.model
    return videos.map(v => this.transformVideoData(v))  // ← this.transformVideoData
  }
}

// 需要实例化并绑定
const videoService = new VideoService()
module.exports = {
  getCarouselData: videoService.getCarouselData.bind(videoService)
}
```

**步骤2：创建工厂函数**
```javascript
// 工厂函数实现
const createService = (modelName, config) => {
  const model = prisma[modelName]
  
  return {
    async getListData(options) { ... },
    async getItemData(id, options) { ... }
  }
}
```

**步骤3：改为纯函数**
```javascript
// 纯函数实现
const baseService = createService('video', config)

async function getCarouselData(number) {
  const videos = await prisma.video.findMany(...)  // ← 直接用prisma
  return videos.map(v => transformVideoData(v))  // ← 直接调用函数
}

module.exports = {
  getCarouselData,
  getVideoListData: baseService.getListData
}
```

### 9.3 重构原则
1. **优先使用函数**：如果不需要状态，优先使用函数
2. **使用工厂函数复用代码**：如果多个模块需要相同的逻辑，使用工厂函数
3. **保持简单**：不要过度设计，函数式实现通常更简单
4. **易于测试**：纯函数易于测试，无副作用
