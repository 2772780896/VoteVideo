# 19. Service层重构总结（方案B：Transformer分离）

## 重构背景

原始 Service 层存在三个可读性问题：

1. **数据转换和业务逻辑混在一起** — `videoService.js` 306行中，近100行是 `transformVideoData`、`transformToCarouselItems` 等纯转换函数，和查询逻辑交织
2. **交互状态查询重复5次** — `checkVideoInteractions`、`checkEssayInteractions`、`checkPostInteractions`... 5个 Service 各自写了一套几乎相同的代码
3. **Controller 层也在重复** — `videoController`、`essayController`、`postController` 各自在列表接口里手动调 `checkXxxInteractions` 再合并

一句话概括：**同一个逻辑（查交互状态）在5个Service和3个Controller里各写了一遍，数据转换函数和数据库查询函数搅在一起。**

---

## 方案选型

当时提出了三种方案：

| 方案 | 思路 | 优点 | 缺点 |
|------|------|------|------|
| A. 原地整理 | 把转换函数移到文件底部，加注释分隔 | 改动最小 | 本质没变，只是视觉上整齐 |
| **B. Transformer层分离** | 数据转换抽到独立 `transformers/` 目录，交互查询抽到 `interactionService` | 职责清晰、可复用 | 新增文件和模块 |
| C. 中间件化 | 用 Express 中间件统一处理转换和交互 | 框架层面自动化 | 过度设计，调试困难 |

最终选择 **方案B**，核心原则：**一个文件只做一件事**。

---

## 重构架构

### 新增模块

```
services/
├── interactionService.js          ← 通用交互状态查询（配置驱动）
├── transformers/
│   ├── videoTransformer.js        ← 视频数据转换
│   ├── essayTransformer.js        ← 文章数据转换
│   ├── postTransformer.js         ← 动态数据转换
│   ├── commentTransformer.js      ← 评论数据转换（含递归子评论）
│   └── tagTransformer.js          ← 标签数据转换
├── baseService.js                 ← 增加了自动合并交互状态
├── videoService.js                ← 精简后
├── essayService.js                ← 精简后
├── postService.js                 ← 精简后
├── commentService.js              ← 精简后
└── tagService.js                  ← 精简后
```

### 数据流变化

**重构前**（以视频列表为例）：
```
Controller → videoService.getVideoListData()
               ├── 查数据库
               ├── 内联转换数据（50行转换代码）
               └── 返回
           → videoController 手动调 checkVideoInteractions()
           → videoController 手动合并交互状态到每条数据
```

**重构后**：
```
Controller → videoService.getVideoListData()（即 baseService.getListData）
               ├── 查数据库
               ├── 调 videoTransformer 转换数据
               └── 自动调 interactionService 合并交互状态
           → 直接返回完整数据
```

Controller 不再需要关心交互状态的查询和合并。

## 核心设计

### interactionService — 配置驱动

不再为每种媒体写一套查询逻辑，而是用配置表定义每种类型支持哪些交互：

```
video:   likes ✓  favourites ✓  reshares ✓  dislikes ✗
essay:   likes ✓  favourites ✓  reshares ✓  dislikes ✗
comment: likes ✓  favourites ✓  reshares ✗  dislikes ✓
tag:     likes ✓  favourites ✓  reshares ✗  dislikes ✓
```

查询时根据配置并行查对应的表（UserLike / UserFavourite / UserReshare / UserDislike），不支持的直接跳过。

对外只暴露两个函数：
- `checkInteractions(type, ids, uid)` — 返回交互状态映射
- `mergeInteractions(type, items, idField, uid)` — 原地合入交互状态

### baseService — 自动合并交互

`getListData` 在数据转换完成后，自动调用 `mergeInteractions`：

```
查数据库 → transformFunction 转换 → mergeInteractions 合并交互 → 返回
```

Service 层通过 `mediaType` 配置告诉 baseService 当前是什么类型，baseService 就知道该查哪些交互表。

### Transformer — 纯函数

每个 transformer 都是纯函数：**输入原始 Prisma 数据，输出前端需要的格式**。不依赖数据库、不查交互状态、不关心调用者是谁。

## 重构效果

### 代码量对比

| 文件 | 重构前 | 重构后 | 变化 |
|------|--------|--------|------|
| videoService.js | 306行 | 136行 | -56% |
| essayService.js | 207行 | 92行 | -56% |
| postService.js | 250行 | 115行 | -54% |
| commentService.js | 358行 | ~140行 | -61% |
| tagService.js | 178行 | ~96行 | -46% |

### 消除的重复代码

| 重复逻辑 | 重复次数 | 重构后 |
|----------|----------|--------|
| checkXxxInteractions（查询+拼装） | 5次（每个Service一份） | 1次（interactionService） |
| Controller手动合并交互状态 | 3次（video/essay/post） | 0次（baseService自动） |
| safeJsonParse 工具函数 | 2次（post/comment各一份） | 1次（baseService导出） |

### 职责分离

- **Controller**：只做 HTTP 翻译（解析参数、返回响应）
- **Service**：只做业务编排（调谁、按什么顺序、传什么参数）
- **Transformer**：只做数据格式转换（纯函数）
- **interactionService**：只做交互状态查询（配置驱动）
- **baseService**：提供通用的 CRUD + 自动合并交互

## 关键决策

1. **为什么 Controller 不再手动调 checkInteractions？**
   因为列表接口100%都需要交互状态，放在 baseService.getListData 里自动化更合理。详情页由各 Service 的 getXxxDetailData 单独处理。

2. **为什么 Transformer 是纯函数？**
   纯函数容易测试、容易复用、不产生副作用。详情页和列表页可以用同一个 transformer，只是 options 不同。

3. **为什么用配置驱动而不是继承？**
   5种媒体类型的差异仅在"支持哪些交互操作"，用一张配置表就能描述，不需要引入继承或多态。

## 后续修复

重构完成后发现两处残留重复，随后修复：

1. **commentService 复用 baseService**：原来 commentService 完全自己写了分页/排序/关注筛选/交互合并逻辑（~70行），后来改为通过 `where` + `include` 覆盖复用 `baseService.getListData`，删除了重复代码。

2. **配置表统一**：`interactService.js` 的 `RESOURCE_CONFIG` 和 `interactionService.js` 的 `INTERACTION_CONFIG` 描述的是同一件事，合并为一张表。`interactService` 现在从 `interactionService` 导入配置，不再各自维护。
