## 一、videoController 分析（以视频模块为例）

### 1.1 前端需求 vs 后端接口

#### ✅ 接口路径匹配

| 前端需求 | 后端实现 | 匹配度 |
|----------|----------|--------|
| `GET /api/picture/carousel?number=5` | `GET /api/picture/carousel` | ✅ 完全匹配 |
| `GET /api/video/main?page=1&element=16` | `GET /api/video/main` | ✅ 完全匹配 |
| `GET /api/video/:vid` | `GET /api/video/:vid` | ✅ 完全匹配 |
| `GET /api/video/related?vid=123&sort=-date&page=1&element=5` | `GET /api/video/related` | ✅ 完全匹配 |

#### ⚠️ 数据格式差异

**问题1：VideoItem 的 `videoUrl` 字段**
- 前端需求：列表页不需要 `videoUrl`，详情页才有
- 后端实现：列表页确实没返回 `videoUrl` ✅ 正确
- 但相关视频接口也没返回 `videoUrl` ✅ 正确

**问题2：VideoItem 的 `tagList` 字段**
- 前端需求：详情页才有 `tagList`
- 后端实现：
  - ✅ 详情页返回了 `tagList`
  - ❌ 列表页也返回了 `tagList`（前端不需要）
  - ❌ 相关视频没返回 `tagList`（前端可能也不需要）

**问题3：数据格式不一致**
- 前端需求：`viewCount` 和 `messageCount` 可以是 `number | string`（格式化后）
- 后端实现：返回了格式化后的字符串（如 "1.2k"）
- 建议：前端自己格式化，后端返回原始数字

### 1.3 数据库封装建议

#### 是否需要封装？

**建议**：✅ 需要封装，原因：
1. **代码复用**：多个接口需要相同的数据库查询逻辑
2. **统一管理**：数据库模型变更时，只需修改一处
3. **业务逻辑分离**：控制器只负责 HTTP 层，业务逻辑在 Service 层

#### 封装示例

**创建 `services/videoService.js`**：
```javascript
// services/videoService.js - 视频相关数据库操作封装

const { PrismaClient } = require('@prisma/client')
const prisma = new PrismaClient()

// 获取轮播图视频
const getCarouselVideos = async (number) => {
  return await prisma.video.findMany({
    take: number,
    orderBy: { viewCount: 'desc' },
    select: { vid: true, coverUrl: true, title: true }
  })
}

// 获取视频列表（带分页、排序、搜索）
const getVideoList = async (page, element, sort, q) => {
  // ... 分页逻辑、排序逻辑、查询逻辑
}

// 根据 ID 获取视频（包含关联数据）
const getVideoById = async (vid, includeTags = false) => {
  const include = {
    uploader: {
      select: { uid: true, username: true, profilePictureUrl: true }
    }
  }
  
  if (includeTags) {
    include.videoTags = { include: { tag: true } }
  }
  
  return await prisma.video.findUnique({
    where: { vid: vid },
    include: include
  })
}

module.exports = {
  getCarouselVideos,
  getVideoList,
  getVideoById
}
```

**控制器中调用**：
```javascript

  const vid = parseInt(req.params.vid)

  const video = await videoService.getVideoById(vid, true)
  
  if (!video) {
    return res.status(404).json({...})
  }
  
  // ... 返回响应
```

## 二、统一优化方案

### 2.1 创建工具函数模块

**文件**：`utils/formatter.js`

```javascript
// utils/formatter.js - 格式化工具函数

// 格式化数字（播放量、评论数等）
const formatCount = (count) => {
  if (count >= 1000000) {
    return (count / 1000000).toFixed(1) + 'm'
  } else if (count >= 1000) {
    return (count / 1000).toFixed(1) + 'k'
  }
  return count.toString()
}

// 格式化日期
const formatDate = (date) => {
  const d = new Date(date)
  const year = d.getFullYear()
  const month = String(d.getMonth() + 1).padStart(2, "0")
  const day = String(d.getDate()).padStart(2, "0")
  return `${year}-${month}-${day}`
}

// 格式化时长（秒 → "12:34"）
const formatDuration = (seconds) => {
  const mins = Math.floor(seconds / 60)
  const secs = seconds % 60
  return `${mins}:${secs.toString().padStart(2, "0")}`
}

module.exports = {
  formatCount,
  formatDate,
  formatDuration
}
```

### 2.3 统一数据格式

**原则**：
1. **列表接口**：返回简要信息（不含详情字段）
2. **详情接口**：返回完整信息
3. **统一字段名**：后端返回的字段名必须与前端 API 文档一致
4. **原始数据**：后端返回原始数据（如数字），格式化交给前端

**示例**：
```javascript
// 视频列表接口返回的数据（简要）
{
  vid: 123,
  coverUrl: '...',
  title: '...',
  viewCount: 1000,        // 原始数字，不格式化
  messageCount: 50,        // 原始数字，不格式化
  duration: '12:34',
  date: '2026-06-12',
  uploader: {...},
  // 不返回 tagList（详情页才有）
}

// 视频详情接口返回的数据（完整）
{
  vid: 123,
  coverUrl: '...',
  videoUrl: '...',         // 详情页才有
  title: '...',
  viewCount: 1000,
  messageCount: 50,
  duration: '12:34',
  date: '2026-06-12',
  uploader: {...},
  tagList: [...]            // 详情页才有
}
```

### 2.4 统一错误处理

**当前状态**：每个控制器都有重复的错误处理代码

**优化方案**：使用 Express 错误处理中间件（已在 `app.js` 中配置）

**控制器中**：
```javascript
const getVideoDetail = async (req, res, next) => {
  const vid = parseInt(req.params.vid)
  
  const video = await videoService.getVideoById(vid, true)
  
  if (!video) {
    // 直接抛出错误，交给错误处理中间件
    const error = new Error('视频不存在')
    error.status = 404
    return next(error)
  }
  
  // ... 返回响应
}
```

**错误处理中间件**（`app.js` 中已配置）：
```javascript
app.use((err, req, res, next) => {
  console.error('全局错误:', err.message)
  
  const status = err.status || 500
  const message = err.message || '服务器内部错误'
  
  return res.status(status).json({
    code: status,
    message: message,
    data: null
  })
})
```

## 五、总结

当前后端接口已基本完成，但存在以下问题：
1. **代码重复**：工具函数、数据库查询、错误处理都有重复
2. **数据格式不一致**：部分接口返回了前端不需要的字段
3. **可维护性差**：数据库操作分散在控制器中

**建议流程**：
1. 先统一工具函数（快速见效）
2. 再封装数据库操作（提高复用性）
3. 然后统一数据格式（确保前端正常）
4. 最后统一错误处理（提高可维护性）
