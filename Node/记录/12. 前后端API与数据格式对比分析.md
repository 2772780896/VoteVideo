## 三、前端数据消费存在的问题

### 3.1 问题1：交互API路径不匹配

**描述**：
- 前端`content.js`中的`ACTION_MAP`定义：`/api/:mediaType/:mediaId/like`
- 后端实际路由：`/api/interact/like/:type/:id`

**影响**：
- 前端发送请求时，路径错误
- 无法正确调用点赞、收藏、关注等功能

**解决方案**：
1. **修改后端路由**（推荐）：
   ```javascript
   // routes/interactRoutes.js
   router.post('/:type/:id/like', needToken, interactController.like)
   ```

2. **修改前端代码**：
   ```javascript
   // src/apis/content.js
   const ACTION_MAP = {
     like: { method: 'post', path: 'interact/like/:mediaType/:mediaId' },
   }
   ```

---

### 3.2 问题2：消息API路径不匹配

**描述**：
- 前端期望：`POST /api/user/message/send`
- 后端实际：`POST /api/message/send`

**影响**：
- 前端发送消息时，路径错误
- 无法正确调用发送消息功能

**解决方案**：
1. **修改后端路由**（推荐）：
   ```javascript
   // routes/messageRoutes.js
   router.post('/send', needToken, messageController.sendMessage)
   // 保持当前实现，更新前端代码
   ```

2. **修改前端代码**：
   ```javascript
   // src/apis/account.js
   export const sendMessage = (dialogueMid, text) =>
     request.post('/api/message/send', { dialogueMid, text }, { needToken: true })
   ```

---

### 3.3 问题3：Profile子数据需要多次请求

**描述**：
- 前端文档定义：`GET /api/user/profile/favourites/videos`
- 后端实际返回：只返回收藏记录（包含`item_id`），不包含具体数据
- 前端需要根据`item_id`再次查询具体数据

**影响**：
- 前端需要发送多次请求
- 增加服务器压力
- 降低用户体验

**解决方案**（推荐）：
- 后端在返回收藏记录时，同时返回具体数据（关联查询）
- 或者提供批量查询接口

**描述**：
- 后端已实现SSE：`GET /api/message/stream`
- 前端文档未提及SSE

**影响**：

## 四、最规范模式建议

### 4.1 API路径设计规范

**推荐方案**：`/api/{resource}/{id}/{action}`

**理由**：
1. **更符合RESTful规范**：资源在操作之前
2. **更直观**：`/api/video/123/like`比`/api/interact/like/video/123`更直观
3. **更易于扩展**：新增资源类型无需修改路由

**实施步骤**：

#### 步骤1：修改后端路由

```javascript
// routes/interactRoutes.js
// 原代码：
router.post('/:type/:id/like', needToken, interactController.like)

// 修改为：
router.post('/:type/:id/like', needToken, interactController.like)
router.delete('/:type/:id/like', needToken, interactController.like)
router.post('/:type/:id/favourite', needToken, interactController.favourite)
router.delete('/:type/:id/favourite', needToken, interactController.favourite)
router.post('/user/:id/follow', needToken, interactController.follow)
router.delete('/user/:id/follow', needToken, interactController.follow)
```

**注意**：当前后端路由已经是`/:type/:id/like`格式，但前端文档和代码期望的是`/:type/:id/like`格式，所以需要确认。

根据前端文档（第358-364行）：
```
| action | method | 路径 |
| like/unlike | POST/DELETE | /api/:type/:id/like |
```

所以前端期望的是`/api/video/123/like`，而后端实际是`/api/interact/like/video/123`。

**需要修改后端路由为**：
```javascript
// routes/interactRoutes.js
// 新的路由格式：/api/:type/:id/like
router.post('/:type/:id/like', needToken, interactController.like)
router.delete('/:type/:id/like', needToken, interactController.like)
```

但这样会与视频、文章等资源的路由冲突（因为`:type`会匹配`video`、`essay`等）。

**更好的方案**：使用独立的路由前缀`/api/interact/:type/:id/like`，并更新前端代码。

---

### 4.2 数据格式设计规范

**推荐方案**：统一字段名

| 字段类型 | 推荐字段名 | 说明 |
|----------|------------|------|
| 资源ID | `id` | 统一使用`id`，前端用映射表区分 |
| 评论数 | `commentCount` | 统一使用`commentCount` |
| 点赞数 | `likeCount` | 统一使用`likeCount` |
| 收藏数 | `favouriteCount` | 统一使用`favouriteCount` |
| 浏览数 | `viewCount` | 统一使用`viewCount` |

**理由**：
1. **更简洁**：无需记忆不同资源的ID字段名
2. **更统一**：同一含义的字段，在所有资源中名称相同
3. **更易于维护**：修改字段名时，只需修改一处

**实施步骤**：

#### 步骤1：更新后端服务层

```javascript
// services/videoService.js
const transformVideoData = (video, options = {}) => {
  return {
    id: video.vid,  // 改为 id
    coverUrl: video.coverUrl,
    title: video.title,
    viewCount: formatCount(video.viewCount),
    commentCount: formatCount(video.messageCount),  // 改为 commentCount
    // ...
  }
}
```

#### 步骤2：更新前端映射表

```javascript
// src/apis/content.js
const ID_KEY_MAP = {
  video: 'id',  // 改为 id
  essay: 'id',
  post: 'id',
  tag: 'id',
  user: 'id',
  comment: 'id'
}
```

### 4.3 前端数据消费优化建议

**推荐方案**：后端提供完整数据，减少前端请求次数

#### 优化1：Profile子数据返回完整信息

**当前实现**：
- 返回收藏记录（包含`item_id`）
- 前端需要根据`item_id`再次查询

**优化后**：
- 返回收藏记录 + 具体数据（关联查询）
- 前端无需再次查询

**实施示例**：

```javascript
// services/userService.js
async function getProfileSubdata(uid, profileType, dataType) {
  if (profileType === 'favourites') {
    // 关联查询：获取收藏记录 + 具体数据
    const favourites = await prisma.userFavourite.findMany({
      where: { uid, type: dataType },
      include: {
        video: true,  // 关联视频数据
        essay: true,   // 关联文章数据
        post: true     // 关联动态数据
      }
    })
    
    // 返回完整数据
    return favourites.map(f => ({
      id: f.id,
      item: f.video || f.essay || f.post  // 具体数据
    }))
  }
}
```
