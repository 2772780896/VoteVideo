> **核心目标**：消除 if-else 链条，使用映射表驱动代码，建立交互服务层。

Interact接口, 映射表, 配置驱动, 服务层, 代码复用, 重构总结

## 一、背景与问题

### 1.1 初始状态

**文件**：`controllers/interactController.js`（338行）

**问题1：if-else 链条严重**

点赞函数（`like`）中有 **50行** 的 if-else 链条，用于判断资源类型（`video`、`essay`、`post`、`comment`、`tag`）。

每个分支的代码结构完全相同：
1. 查询资源是否存在
2. 判断是点赞还是取消点赞（根据 HTTP 方法）
3. 更新对应表的 `likeCount` 字段

**问题2：代码重复度高**

收藏函数（`favourite`）中也有 **30行** 的 if-else 链条，用于更新 `favouriteCount` 字段。

**问题3：Token 验证重复**

3个函数（`like`、`favourite`、`follow`）都有相同的 Token 验证代码（各13行）。

**问题4：无服务层**

所有数据库操作都在控制器中完成，难以复用和测试。

### 1.2 具体问题

**问题代码示例**（第37-87行）：


**问题**：
- 5个分支结构完全相同

- 难以扩展（添加新资源类型需要修改所有函数）

## 二、解决方案：映射表 + 通用函数

### 2.1 设计思路

**核心思想**：用配置驱动代码，消除 if-else 链条

**步骤**：
1. 创建 **资源类型配置映射表**（`RESOURCE_CONFIG`）
2. 提取 **通用函数**（`findResource`、`toggleLike`、`toggleFavourite`、`toggleFollow`）
3. 用 **映射表配置** 替代 if-else 判断

### 2.2 映射表设计

**文件**：`services/interactService.js`

**映射表结构**：

```javascript
// 资源类型配置映射表
const RESOURCE_CONFIG = {
  video: {
    model: 'video',      // Prisma 模型名
    idField: 'vid',      // ID 字段名
    name: '视频',        // 中文名称（用于错误信息）
    hasLike: true,       // 是否有点赞功能
    hasFavourite: true   // 是否有收藏功能
  },
  essay: {
    model: 'essay',
    idField: 'eid',
    name: '文章',
    hasLike: true,
    hasFavourite: true
  },
  // ... 其他资源类型
}
```

**优点**：
- **清晰**：配置明确，易于理解
- **灵活**：可以为不同资源类型添加特殊配置
- **易于扩展**：添加新资源类型只需在映射表中添加一行

### 2.3 通用函数设计

**函数1：`findResource(type, id)`**

用途：查询资源是否存在，不存在则抛出错误

```javascript
async function findResource(type, id) {
  // 1. 获取配置
  const config = RESOURCE_CONFIG[type]
  if (!config) {
    throw new Error(`不支持的资源类型: ${type}`)
  }

  // 2. 查询资源
  const model = prisma[config.model]
  const resource = await model.findUnique({
    where: { [config.idField]: parseInt(id) }
  })

  // 3. 检查是否存在
  if (!resource) {
    throw new Error(`${config.name}不存在`)
  }

  return resource
}
```

**函数2：`toggleLike(type, id, method)`**

用途：更新资源的 `likeCount` 字段

```javascript
async function toggleLike(type, id, method) {
  // 1. 查询资源是否存在
  const resource = await findResource(type, id)

  // 2. 检查该资源类型是否支持点赞
  const config = RESOURCE_CONFIG[type]
  if (!config.hasLike) {
    throw new Error(`${config.name}不支持点赞`)
  }

  // 3. 计算增量（POST = 点赞 = +1, DELETE = 取消点赞 = -1）
  const increment = method === 'POST' ? 1 : -1

  // 4. 更新点赞数
  const model = prisma[config.model]
  const updated = await model.update({
    where: { [config.idField]: parseInt(id) },
    data: { likeCount: { increment: increment } }
  })

  return updated
}
```

**函数3：`toggleFavourite(uid, type, id, method)`**

用途：更新 `UserFavourite` 表和资源的 `favouriteCount` 字段

**函数4：`toggleFollow(uid, targetUid, method)`**

用途：更新 `UserFollowing` 表和用户的 `followerCount`、`followingCount` 字段

## 六、映射表设计总结

### 6.1 为什么用映射表？

**问题**：if-else 链条难以维护和扩展

**解决方案**：映射表 + 通用函数

**优点**：
1. **配置驱动**：用数据（映射表）驱动代码，而不是用 if-else 判断
2. **易于理解**：配置明确，易于阅读
3. **易于扩展**：添加新资源类型只需在映射表中添加一行
4. **减少错误**：避免 if-else 链条中的拼写错误

### 6.2 映射表结构设计

**基本原则**：
1. **每个资源类型一个配置项**
2. **配置项包含该资源类型的所有特征**（模型名、ID字段、中文名称、支持的功能等）
3. **用配置项替代 if-else 判断**

**示例**：

```javascript
const RESOURCE_CONFIG = {
  video: {
    model: 'video',      // Prisma 模型名
    idField: 'vid',      // ID 字段名
    name: '视频',        // 中文名称
    hasLike: true,       // 是否支持点赞
    hasFavourite: true   // 是否支持收藏
  },
  // ... 其他资源类型
}
```

### 6.3 通用函数设计

**基本原则**：
1. **从映射表读取配置**，而不是用 if-else 判断
2. **用动态字段名**（如 `config.model`、`config.idField`）
3. **错误处理统一**（如资源不存在、不支持的功能等）

**示例**：

```javascript
async function toggleLike(type, id, method) {
  // 1. 从映射表读取配置
  const config = RESOURCE_CONFIG[type]

  // 2. 用动态字段名
  const model = prisma[config.model]
  const updated = await model.update({
    where: { [config.idField]: parseInt(id) },
    data: { likeCount: { increment: increment } }
  })

  return updated
}
```
