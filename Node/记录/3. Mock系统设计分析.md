# VoteVideo Mock 系统设计分析与数据库迁移评估

> 本文档分析 `React/src/mock/` 文件夹的设计逻辑、规范性问题，并评估迁移为数据库设计的可行性。

---

## 一、Mock 系统架构概览

### 1.1 目录结构

```
src/mock/
├── index.js              # 统一导出所有 Mock 接口
├── publicState.js        # 全局状态管理（内存存储）
├── basicData/           # 数据生成层（Mock.js）
│   ├── createVideo.js
│   ├── createUser.js
│   ├── createEssay.js
│   ├── createPost.js
│   ├── createComment.js
│   ├── createTag.js

│   ├── getData/         # 获取单个资源

│   ├── Profile/        # 用户 Profile 相关
│   ├── login.js
│   ├── register.js

│   └── ...
```

### 1.2 数据流向

```
[前端请求]
    ↓
[Mock.mock() 拦截]
    ↓
[verifyAuthorization()] ← 验证 Token（从 body 读取）
    ↓
[从 publicState.js 获取数据] ← 内存中的全局变量
    ↓
[sortData()] ← 排序 + 分页
    ↓
[返回 Mock 响应] ← { code: 200, data: ... }
```

## 二、设计逻辑分析

### 2.1 数据生成层（basicData/）

**设计思路**：使用 Mock.js 随机生成数据，自动添加到全局状态。

**问题**：

#### 问题 1：数据生成逻辑错误

**createVideo.js**（第 15 行）：
```javascript
uploader: createUser()[0],  // ❌ 每个视频都创建一个新用户
```

**应该改为**：
```javascript
// 应该从已有用户中选择，而不是每个视频都创建新用户
uploader: getRandomUser(),  // 从 userList 随机选一个
```

**影响**：
- 用户和视频是一对一关系（应该是用户 1:N 视频）
- 数据库迁移时会产生大量重复用户

---

#### 问题 2：标签系统设计与数据库范式不符

**createVideo.js**（第 20 行）：
```javascript
tagList: createTag(Random.integer(5,15))  // ❌ 为每个视频动态生成新标签
```

**应该改为**：
```javascript
// 应该从标签库中选择，建立多对多关系
tagList: getRandomTags(5, 15),  // 从 tagList 中随机选
```

**影响**：
- 标签没有独立库，无法复用
- 数据库迁移时需要重建标签表和中间表

---

### 2.2 全局状态管理（publicState.js）

**设计思路**：使用 JavaScript 全局变量存储所有数据。

**问题**：

#### 问题 3：userExtension 结构不规范

**publicState.js**（第 2-49 行）：
```javascript
let userExtension = []

// 结构：
{
  uid: 1,
  uploads: {
    videos: [],   // ❌ 嵌套数组，不是独立表
    essays: [],
    posts: [],
  },
  favourites: {
    videos: [],   // ❌ 应该是中间表
    essays: [],
    posts: [],
  },
  history: {
    videos: [],   // ❌ 应该是中间表
    essays: [],
    posts: [],
  },
  follow: {
    followingList: [],  // ❌ 应该是中间表
  },
  message: {
    dialogueList: [],  // ❌ 应该是独立表
    notificationList: [],
  }
}
```

**应该改为**（数据库设计）：
```sql
-- 用户投稿
CREATE TABLE user_uploads (
  uid INT,
  type ENUM('video', 'essay', 'post'),
  item_id INT,
  addDate DATETIME,
  FOREIGN KEY (uid) REFERENCES users(uid)
);

-- 用户收藏
CREATE TABLE user_favourites (
  uid INT,
  type ENUM('video', 'essay', 'post'),
  item_id INT,
  addDate DATETIME,
  PRIMARY KEY (uid, type, item_id)
);

-- 用户历史
CREATE TABLE user_history (
  uid INT,
  type ENUM('video', 'essay', 'post'),
  item_id INT,
  addDate DATETIME,
  PRIMARY KEY (uid, type, item_id)
);

-- 用户关注
CREATE TABLE user_following (
  uid INT,
  following_uid INT,
  PRIMARY KEY (uid, following_uid)
);
```

---

#### 问题 4：数据关系通过对象引用，不是外键

**createVideo.js**（第 59 行）：
```javascript
addUserExtension(user.uid, 'uploads', 'videos', i)  // ❌ JavaScript 对象引用
```

**应该改为**（数据库外键）：
```sql
-- videos 表
CREATE TABLE videos (
  vid INT PRIMARY KEY AUTO_INCREMENT,
  title VARCHAR(100),
  uploader_uid INT,  -- ✅ 外键关联 users.uid
  FOREIGN KEY (uploader_uid) REFERENCES users(uid)
);
```

### 2.3 响应层（response/）

**设计思路**：使用 Mock.mock() 拦截请求，返回 Mock 数据。

**问题**：

#### 问题 5：Token 验证方式不标准

**verifyAuthorization.js**（第 11-12 行）：
```javascript
// ❌ 从 body 读取 token（因为 Mock.js 无法读取 Header）
const data = JSON.parse(options?.body)
const authorization = data?.authorization
```

**应该改为**（标准 HTTP 认证）：
```
Authorization: Bearer <token>  ← ✅ 放 HTTP Header
```

---

#### 问题 6：响应格式不符合 HTTP 标准

**getVideo.js**（第 26-30 行）：
```javascript
return Mock.mock({
  code: 200,          // ❌ 业务状态码（应该用 HTTP 状态码）
  message: 'ok',
  data: video
})
```

**应该改为**：
```javascript
// HTTP 状态码由响应头表示
res.status(200).json({
  data: video,
  message: 'ok'
});
```

---

### 2.4 工具函数（utils/）

#### sortData.js（可迁移）

**设计**：支持排序（`+` 升序，`-` 降序）和分页（`page`, `element`）。

**可以迁移为 SQL**：
```sql
-- 排序
ORDER BY date ASC     -- sort=+date 或 sort=date
ORDER BY date DESC    -- sort=-date

-- 分页
LIMIT 16 OFFSET 0    -- page=1, element=16
LIMIT 16 OFFSET 16   -- page=2, element=16
```

✅ **这部分逻辑可以直接迁移**。

## 三、数据库迁移可行性评估

### 3.1 ✅ 可以直接迁移的部分

#### 基本表结构

| Mock 数据 | 数据库表 | 迁移难度 |
|----------|---------|---------|
| `userList` | `users` | ✅ 低（直接映射） |
| `videoList` | `videos` | ✅ 低（直接映射） |
| `essayList` | `essays` | ✅ 低（直接映射） |
| `postList` | `posts` | ✅ 低（直接映射） |
| `tagList` | `tags` | ⚠️ 中（需要重建，因为 Mock 中标签是动态生成的） |
| `commentList` | `comments` | ✅ 低（直接映射） |

#### 一对多关系

| 关系 | Mock 实现 | 数据库实现 | 迁移难度 |
|------|----------|-----------|---------|
| 用户 → 视频 | `video.uploader.uid` | `videos.uploader_uid` 外键 | ✅ 低 |
| 视频 → 评论 | `comment.vid` | `comments.vid` 外键 | ✅ 低 |
| 评论 → 子评论 | `comment.replyTo` | `comments.replyTo_cid` 自引用 | ✅ 低 |

---

### 3.2 ❌ 需要重构的部分

#### 多对多关系

| 关系 | Mock 实现 | 数据库实现 | 迁移难度 |
|------|----------|-----------|---------|
| 视频 ↔ 标签 | `video.tagList` 嵌套数组 | `video_tags` 中间表 | ⚠️ 高（需要重建数据） |
| 用户 ↔ 关注 | `userExtension.follow.followingList` | `user_following` 中间表 | ⚠️ 高（需要拆分） |
| 用户 ↔ 收藏 | `userExtension.favourites` 嵌套对象 | `user_favourites` 中间表 | ⚠️ 高（需要拆分） |
| 用户 ↔ 历史 | `userExtension.history` 嵌套对象 | `user_history` 中间表 | ⚠️ 高（需要拆分） |

#### userExtension 拆分

```javascript
  history: { videos: [], essays: [], posts: [] },      // ❌ 应该拆成 user_history 表
  follow: { followingList: [] },                      // ❌ 应该拆成 user_following 表
```

### 3.3 ⚠️ 需要特殊处理的部分

#### 标签系统重构

**Mock 中的问题**：
- 标签是动态生成的（`createTag(Random.integer(5,15))`）
- 每个视频的标签都不同，没有复用
- 没有独立标签库

**迁移方案**：
1. 从所有视频的 `tagList` 中提取唯一标签
2. 创建独立的 `tags` 表
3. 创建 `video_tags` 中间表（vid, tid）

```sql
-- 标签表
CREATE TABLE tags (
  tid INT PRIMARY KEY AUTO_INCREMENT,
  tagName VARCHAR(50) UNIQUE,
  likeCount INT DEFAULT 0,
  favouriteCount INT DEFAULT 0,
  commentCount INT DEFAULT 0
);

-- 视频-标签中间表
CREATE TABLE video_tags (
  vid INT,
  tid INT,
  PRIMARY KEY (vid, tid),
  FOREIGN KEY (vid) REFERENCES videos(vid),
  FOREIGN KEY (tid) REFERENCES tags(tid)
);
```

---

#### 密码加密

**Mock 中的问题**：
```javascript
password: '123'  // ❌ 所有用户密码都是 123（明文）
```

**迁移方案**：
```javascript
// 注册时加密
const bcrypt = require('bcrypt');
const hashedPassword = await bcrypt.hash(password, 10);

// 登录时验证
const match = await bcrypt.compare(password, user.password);
```

---

#### Token 生成和验证

**Mock 中的问题**：
```javascript
token: Mock.Random.guid()  // ❌ 随机生成，存储在 user 对象中
```

**迁移方案**（JWT）：
```javascript
// 登录时签发
const token = jwt.sign({ uid: user.uid }, secretKey, { expiresIn: '10h' });

// 验证中间件
const token = req.headers.authorization?.split(' ')[1];
const decoded = jwt.verify(token, secretKey);
```

## 四、规范性问题总结

### 4.1 严重问题（影响数据库迁移）

| 问题 | 描述 | 影响 | 修复优先级 |
|------|------|------|-----------|
| **数据生成逻辑错误** | 每个视频都创建新用户 | 用户-视频关系错误 | P0 |
| **多对多关系设计错误** | 标签、关注、收藏等应该是中间表 | 无法映射为数据库表 | P0 |
| **userExtension 结构不规范** | 嵌套对象，不是独立表 | 需要拆分重构 | P0 |

### 4.2 中等问题（影响后端实现）

| 问题 | 描述 | 影响 | 修复优先级 |
|------|------|------|-----------|
| **没有数据持久化** | 刷新页面丢失所有数据 | 无法保存用户操作 | P1 |
| **Token 验证方式不标准** | 放 body 而不是 Header | 不符合 HTTP 标准 | P1 |
| **响应格式不标准** | 用 body.code 而不是 HTTP 状态码 | 不符合 RESTful 规范 | P1 |

### 4.3 轻微问题（命名和约定）

| 问题 | 描述 | 影响 | 修复优先级 |
|------|------|------|-----------|
| **分页参数命名不直观** | `element` 应该改为 `pageSize` | 可读性差 | P2 |
| **排序参数设计** | `sort=-date` 可以，但需要约定 | 需要文档说明 | P2 |

## 六、数据库表结构设计（参考）

### 6.1 基本表

```sql
-- 用户表
CREATE TABLE users (
  uid INT PRIMARY KEY AUTO_INCREMENT,
  username VARCHAR(50) UNIQUE NOT NULL,
  password VARCHAR(255) NOT NULL,  -- bcrypt 哈希
  profilePictureUrl VARCHAR(255),
  info VARCHAR(255),
  followerCount INT DEFAULT 0,
  followingCount INT DEFAULT 0,
  createdAt DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 视频表
CREATE TABLE videos (
  vid INT PRIMARY KEY AUTO_INCREMENT,
  title VARCHAR(100) NOT NULL,
  coverUrl VARCHAR(255),
  videoUrl VARCHAR(255),
  duration VARCHAR(10),
  viewCount INT DEFAULT 0,
  messageCount INT DEFAULT 0,
  likeCount INT DEFAULT 0,
  favouriteCount INT DEFAULT 0,
  uploader_uid INT,
  createdAt DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (uploader_uid) REFERENCES users(uid)
);

-- 文章表
CREATE TABLE essays (
  eid INT PRIMARY KEY AUTO_INCREMENT,
  title VARCHAR(100) NOT NULL,
  text TEXT,
  viewCount INT DEFAULT 0,
  commentCount INT DEFAULT 0,
  likeCount INT DEFAULT 0,
  favouriteCount INT DEFAULT 0,
  uploader_uid INT,
  createdAt DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (uploader_uid) REFERENCES users(uid)
);

-- 动态表
CREATE TABLE posts (
  pid INT PRIMARY KEY AUTO_INCREMENT,
  text TEXT,
  pictureList JSON,  -- 存储图片 URL 数组
  videoList JSON,     -- 存储视频 ID 数组
  viewCount INT DEFAULT 0,
  commentCount INT DEFAULT 0,
  likeCount INT DEFAULT 0,
  uploader_uid INT,
  createdAt DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (uploader_uid) REFERENCES users(uid)
);

-- 标签表
CREATE TABLE tags (
  tid INT PRIMARY KEY AUTO_INCREMENT,
  tagName VARCHAR(50) UNIQUE NOT NULL,
  likeCount INT DEFAULT 0,
  favouriteCount INT DEFAULT 0,
  commentCount INT DEFAULT 0
);

-- 评论表
CREATE TABLE comments (
  cid INT PRIMARY KEY AUTO_INCREMENT,
  vid INT,
  uploader_uid INT,
  text TEXT,
  type ENUM('text', 'picture') DEFAULT 'text',
  pictureList JSON,  -- 存储图片 URL 数组（type=picture 时）
  viewCount INT DEFAULT 0,
  likeCount INT DEFAULT 0,
  favouriteCount INT DEFAULT 0,
  subCommentCount INT DEFAULT 0,
  replyTo_cid INT NULL,  -- 自引用，回复哪条评论
  createdAt DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (vid) REFERENCES videos(vid),
  FOREIGN KEY (uploader_uid) REFERENCES users(uid),
  FOREIGN KEY (replyTo_cid) REFERENCES comments(cid)
);
```

## 七、总结

### 7.1 Mock 系统优点

✅ **快速原型开发**：使用 Mock.js 快速生成数据，无需后端  
✅ **前端独立开发**：前端可以不依赖后端进度  
✅ **灵活的数据生成**：可以模拟各种边界情况  

### 7.2 Mock 系统缺点

❌ **数据不持久化**：刷新页面丢失所有数据  
❌ **不符合数据库范式**：多对多关系设计错误  
❌ **Token 验证不标准**：放 body 而不是 Header  
❌ **响应格式不标准**：用 body.code 而不是 HTTP 状态码  

### 7.3 迁移可行性结论

| 部分 | 迁移可行性 | 工作量 | 说明 |
|------|-----------|--------|------|
| **基本表结构** | ✅ 高 | 低 | 可以直接映射 |
| **一对多关系** | ✅ 高 | 低 | 外键可以直接映射 |
| **多对多关系** | ⚠️ 中 | 高 | 需要重建数据和中间表 |
| **userExtension** | ⚠️ 中 | 高 | 需要拆分为多个表 |
| **标签系统** | ⚠️ 中 | 高 | 需要重建标签库和中间表 |
| **消息系统** | ✅ 高 | 中 | 可以直接映射 |

### 7.4 建议

1. **先修复 Mock 数据生成逻辑**（P0）：
   - 修复 `createVideo.js` 的用户关联
   - 重建标签系统（独立标签库）
   - 重构 `userExtension` 为独立对象

2. **再设计数据库表结构**（P0）：
   - 创建基本表和中间表
   - 设计索引和外键约束

3. **最后编写迁移脚本**（P1）：
   - 导出 Mock 数据为 JSON
   - 转换数据格式（处理多对多关系）
   - 导入数据库
