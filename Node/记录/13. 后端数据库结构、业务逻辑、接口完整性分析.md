## 一、数据库结构分析

### 1.1 Prisma Schema 概览

数据库：SQLite  
ORM：Prisma  
模型数量：11 个

| 模型 | 说明 | 主键 | 外键 |
|------|------|------|------|
| User | 用户表 | uid | - |
| Video | 视频表 | vid | uploader_uid → User(uid) |
| Essay | 文章表 | eid | uploader_uid → User(uid) |
| Post | 动态表 | pid | uploader_uid → User(uid) |
| Tag | 标签表 | tid | - |
| VideoTag | 视频-标签中间表 | (vid, tid) | vid → Video(vid), tid → Tag(tid) |
| Comment | 评论表 | cid | uploader_uid → User(uid), replyTo_cid → Comment(cid) |
| UserFavourite | 用户收藏中间表 | (uid, type, item_id) | uid → User(uid) |
| UserHistory | 用户历史中间表 | (uid, type, item_id) | uid → User(uid) |
| UserFollowing | 用户关注中间表 | (uid, following_uid) | uid → User(uid), following_uid → User(uid) |
| Dialogue | 对话表 | mid | participant1_uid → User(uid), participant2_uid → User(uid) |
| Message | 消息表 | id | mid → Dialogue(mid), sender_uid → User(uid) |
| Notification | 通知表 | nid | from_uid → User(uid) |

### 1.2 数据库设计评估

#### ✅ 设计合理的部分

1. **用户系统完整**：支持注册、登录、关注、粉丝计数
2. **内容模型清晰**：Video、Essay、Post 分离，符合实际业务
3. **评论支持递归**：通过 `replyTo_cid` 自关联实现楼中楼
4. **收藏/历史用中间表**：支持多类型内容（video/essay/post）
5. **标签系统**：Video-Tag 多对多关系，通过中间表实现
6. **私信系统**：Dialogue + Message 分离设计，支持 SSE 推送

#### ⚠️ 存在的问题

| 问题 | 位置 | 严重程度 | 说明 |
|------|------|----------|------|
| **字段名不一致** | Video.model | 🔴 高 | `messageCount` 应改为 `commentCount`（与前端和其他模型统一） |
| **Comment 关系错误** | Comment.model | 🔴 高 | `essay` 和 `post` 关系错误地引用了 `vid` 字段，应该是 `eid` 和 `pid` |
| **缺少 viewCount 递增** | Essay/Post | 🟡 中 | Video 详情有播放量+1，但 Essay 和 Post 没有阅读量递增逻辑 |
| **Notification 缺少接收者** | Notification.model | 🔴 高 | 只能存 `from_uid`，无法指定通知接收者（应有 `to_uid` 字段） |
| **Dialogue 只有 2 参与者** | Dialogue.model | 🟡 中 | 设计为固定 2 参与者，不支持群聊 |
| **图片/视频列表用 JSON** | Post.model | 🟢 低 | `pictureList` 和 `videoList` 存为 JSON 字符串，查询时不方便 |

### 1.3 数据格式与前端匹配度

| 数据类型 | 数据库字段 | 前端期望字段 | 匹配度 | 说明 |
|----------|-------------|-------------|--------|------|
| Video | vid, title, coverUrl, videoUrl, duration, viewCount, messageCount | vid, title, coverUrl, videoUrl, duration, viewCount, **commentCount** | ⚠️ 字段名不一致 | `messageCount` vs `commentCount` |
| Essay | eid, title, text, viewCount, commentCount | eid, title, text, viewCount, commentCount | ✅ 完全匹配 | - |
| Post | pid, text, pictureList, videoList | pid, text, pictureList, videoList | ✅ 完全匹配 | JSON 需解析 |
| Comment | cid, text, type, pictureList, replyTo_cid | cid, text, type, pictureList, replyTo | ⚠️ 字段名不一致 | `replyTo_cid` vs `replyTo` |
| User | uid, username, profilePictureUrl, info, followerCount, followingCount | uid, userName, profilePictureUrl, info, followerCount, followingCount | ⚠️ 字段名不一致 | `username` vs `userName` |
| Tag | tid, tagName, likeCount, favouriteCount, commentCount | tid, tagName, likeCount, favouriteCount, commentCount | ✅ 完全匹配 | - |

## 二、业务逻辑分析

### 2.1 服务层架构

后端采用 **Controller → Service → Prisma** 三层架构：

```
routes/*.js (路由定义)
    ↓
controllers/*.js (HTTP 请求处理)
    ↓
services/*.js (业务逻辑和数据转换)
    ↓
Prisma Client (数据库操作)
```

### 2.2 基础服务工厂 (baseService.js)

`baseService.js` 提供了通用服务工厂函数 `createService()`，封装了：

- **分页逻辑**：`parsePagination(page, element)`
- **排序逻辑**：`parseSortParam(sort, defaultField)`
- **搜索逻辑**：支持 `q` 参数模糊搜索
- **CRUD 操作**：`getListData()`, `getItemData()`, `create()`, `update()`, `delete()`
- **数据转换**：通过 `transformFunction` 配置

各模块服务通过调用 `createService()` 创建基础服务，然后扩展业务特定函数。

## 六、数据格式一致性问题

### 6.1 字段名映射问题

后端需要在响应数据时进行字段名转换：

| 数据库字段 | 前端期望字段 | 转换位置 |
|------------|-------------|----------|
| `username` | `userName` | userController.js |
| `messageCount` | `commentCount` | videoService.js → transformVideoData() |
| `followerCount` | `followerCount` | ✅ 已正确映射 |
| `followingCount` | `followingCount` | ✅ 已正确映射 |
| `pictureList` (JSON) | `pictureList` (Array) | postService.js → JSON.parse() |
| `videoList` (JSON) | `videoList` (Array) | postService.js → JSON.parse() |

### 6.2 响应格式一致性

后端的响应格式已基本统一：

```javascript
// 成功响应
{
  code: 200,
  message: '获取成功',
  data: {...}
}

// 列表响应
{
  code: 200,
  message: '获取成功',
  data: [...],
  total: 100
}

// 错误响应
{
  code: 404,
  message: '资源不存在',
  data: null
}
```

但仍有部分接口响应格式不一致：
- `interactController.favourite` 返回 `data: null`
- `interactController.follow` 返回 `data: null`
- `messageController.sendMessage` 返回 `data: { messageId, dialogueId, pushed }`
