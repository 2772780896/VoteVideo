# VoteVideo 前端 API 设计问题分析

> 本文档分析当前前端 API 设计（基于 `11.API与数据格式.md`）中不符合 RESTful 规范和标准实践的地方，以及为适配 Mock.js 所做的妥协。
> 
> **目标**：为后续重构前端 API 层、对接真实 Node 后端提供修正指南。

---

## 一、严重问题（影响后端实现和标准合规性）

### 问题 1：Token 传递方式违反 HTTP 标准

**当前设计**（`src/utils/request.js` 拦截器）：

```javascript
// 非 DELETE 请求：改 method 为 POST，token 塞进 body
config.data = { ...原data, authorization: token }
// DELETE 请求：保持 method 不变，token 放 data
config.data = { ...原data, authorization: token }
```

**标准做法**（RFC 7235 HTTP 认证标准）：

```
Authorization: Bearer <token>  （放在 HTTP Request Header）
```

**问题**：
- ❌ 违反 HTTP 认证标准（RFC 7235）
- ❌ GET 请求被强行改成 POST（GET 按规范不应有请求体）
- ❌ 后端必须对所有接口同时支持 GET/POST，增加不必要的复杂度
- ❌ 违反 RESTful 原则（HTTP Method 应表示动作，而不是全都用 POST）

**Mock.js 妥协原因**：
> Mock.js 对 Header 拦截支持不好，无法方便地读取自定义 Header，只能放 body。

**修正方向**：
```
// 前端：放 Header
request.interceptors.request.use(config => {
  if (config.needToken) {
    config.headers.Authorization = `Bearer ${Cookies.get('token')}`;
  }
  return config;
});

// 后端：从 Header 读取
const token = req.headers.authorization?.split(' ')[1];
```

### 问题 2：响应格式混淆 HTTP 状态码和业务状态码

**当前设计**：

```typescript
{
  code: 200,          // 业务状态码（放在响应体里）
  message: 'ok',
  data: T | T[],
  total?: number      // 列表接口附加，表示总条数
}
```

**标准做法**：

```typescript
// HTTP 状态码由响应头表示（200, 401, 404...）
// 响应体只放业务数据：
{
  data: T | T[],
  total?: number,
  message?: string    // 仅错误时附加
}

// 业务错误码用单独的字段（可选）：
{
  data: null,
  error: {
    code: 1001,      // 业务错误码（如"用户名已存在"）
    message: '用户名已存在'
  }
}
```

**问题**：
- ❌ `code: 200` 和 HTTP `200 OK` 语义重复
- ❌ 应该用 HTTP 状态码表示请求成功/失败，业务码表示更细粒度的错误类型
- ❌ 前端拦截器判断 `response.data.code !== 200`，而不是 `response.status !== 200`

**Mock.js 妥协原因**：
> Mock.js 生成简单格式方便，且统一用 `code` 字段判断更直观。

**修正方向**：
```javascript
// 前端拦截器：用 HTTP 状态码判断
request.interceptors.response.use(
  response => response,  // 2xx 自动进入 then
  error => {             // 4xx/5xx 进入 catch
    if (error.response?.status === 401) {
      // 跳转到登录页
    }
    return Promise.reject(error);
  }
);

// 后端：用 HTTP 状态码
res.status(200).json({ data: {...}, total: 100 });
res.status(401).json({ error: { code: 1001, message: 'Token 无效' } });
```

## 二、中等问题（RESTful 规范问题）

### 问题 3：资源子类型用路径而非查询参数

**当前设计**：

```
GET /api/video/main        # 首页推荐视频
GET /api/video/related    # 相关视频推荐
GET /api/picture/carousel # 轮播图
```

**标准做法**（RESTful：路径表示资源，查询参数表示过滤条件）：

```
GET /api/videos?category=main          # 首页推荐
GET /api/videos?relatedTo=123          # 相关视频（vid=123 的相关）
GET /api/pictures?type=carousel        # 轮播图
```

**问题**：
- ❌ 把"查询条件"放在路径里，违反 RESTful 原则
- ❌ 扩展困难（新增一个分类就要加一个路由）
- ❌ `SUBTYPE_MAP` 硬编码在前端，新增子类型需要改代码

**Mock.js 妥协原因**：
> Mock.js 对查询参数的动态匹配支持有限，用固定路径更容易 Mock。

**修正方向**：
```javascript
// 前端：用查询参数
fetchItemList('video', { category: 'main', page, element })
// → GET /api/videos?category=main&page=1&element=16

// 后端路由：只用一条
app.get('/api/videos', (req, res) => {
  const { category, relatedTo } = req.query;
  if (category === 'main') { ... }
  if (relatedTo) { ... }
});
```

### 问题 4：Profile 相关路由设计不直观

**当前设计**：

```
GET /api/user/profile/                   # 当前用户 profile
GET /api/user/profile/uploads/videos     # 用户投稿视频
GET /api/user/profile/favourites/posts   # 用户收藏动态
```

**标准做法**（RESTful 嵌套资源规范）：

```
GET /api/users/me                                   # 当前用户 profile
GET /api/users/me/uploads?type=video                # 投稿视频
GET /api/users/:id/favourites?type=post             # 用户收藏
```

**问题**：
- ❌ `/api/user/profile/` 语义不清晰（应该用 `/api/users/me` 或 `/api/profile`）
- ❌ 路径嵌套过深（`/profile/uploads/videos` 三级）
- ❌ `profileType/dataType` 应该用查询参数而非路径参数

**修正方向**：
```javascript
// 方案 A：用 /me 表示当前用户
GET /api/users/me                    # profile
GET /api/users/me/uploads?type=video # 投稿
GET /api/users/me/favourites?type=post # 收藏

// 方案 B：独立 profile 资源
GET /api/profile                      # 当前用户 profile
GET /api/profile/uploads?type=video  # 投稿
```

### 问题 5：交互 API 设计不统一

**当前设计**（`ACTION_MAP`）：

```javascript
| action | method | 路径 |
|--------|--------|------|
| like / unlike | POST/DELETE | /api/:type/:id/like |
| follow / unfollow | POST/DELETE | /api/:type/:id/follow |
```

**标准做法**（两种方案）：

```
方案 A（推荐）：独立资源模式
POST   /api/likes       { type: 'video', id: 123 }
DELETE /api/likes/:id   （likes 表主键）

方案 B：子资源模式（更符合当前设计）
POST   /api/videos/:id/likes
DELETE /api/videos/:id/likes
```

**问题**：
- ❌ `:type` 作为路径参数不直观（应该用请求体或查询参数）
- ❌ 取消操作用 HTTP DELETE，但前端用 `interact('video', 'unlike', 123)` 统一调用，容易混淆
- ❌ `like` / `unlike` 应该是同一个端点的 POST/DELETE，而不是不同 action

**修正方向**：
```javascript
// 前端：统一用 like 端点
const like = (type, id) => request.post(`/api/${type}s/${id}/likes`);
const unlike = (type, id) => request.delete(`/api/${type}s/${id}/likes`);

// 后端：
app.post('/api/videos/:id/likes', ...);    // 点赞
app.delete('/api/videos/:id/likes', ...);  // 取消点赞
```

### 问题 6：上传 API 设计不符合 REST

**当前设计**：

```
POST /api/upload/video    # 上传视频
POST /api/upload/essay    # 上传文章
```

**标准做法**（RESTful：上传是创建资源的一部分）：

```
POST /api/videos          # 创建视频（multipart/form-data）
POST /api/essays         # 创建文章
```

**问题**：
- ❌ `/api/upload` 不是资源，违反 RESTful（所有路径都应表示资源）
- ❌ 上传应该是"创建资源"的一部分，而不是独立接口
- ❌ 如果严格 REST，应该用 `POST /api/videos`（包含文件和元数据）

**修正方向**：
```javascript
// 前端：上传 = 创建资源
const uploadVideo = (formData) => request.post('/api/videos', formData, {
  headers: { 'Content-Type': 'multipart/form-data' }
});

// 后端：
app.post('/api/videos', upload.single('video'), (req, res) => {
  // req.file = 视频文件
  // req.body = { title, description, cover }
});
```

## 三、轻微问题（命名和约定）

### 问题 7：分页参数命名不直观

**当前设计**：`page` + `element`

**标准做法**：`page` + `pageSize` 或 `offset` + `limit`

**问题**：`element` 命名不直观，其他开发者看不懂。

**修正方向**：
```javascript
// 改 element → pageSize
GET /api/videos?page=1&pageSize=16
```

---

### 问题 8：排序参数设计可能增加后端复杂度

**当前设计**：`sort=-date` （减号表示降序）

**标准做法**：`sort=-date` 或 `orderBy=date&order=DESC`

**问题**：这个设计其实可以接受，但前端需要和后端约定好解析规则（`-` 开头 = 降序，默认 = 升序）。更直观的做法是 `orderBy` + `order`。

**修正方向**（可选，当前设计也可用）：
```
// 方案 A（当前）：简洁，但需约定
sort=-date    → 按 date 降序
sort=date     → 按 date 升序

// 方案 B（更直观）：
orderBy=date&order=DESC
```

---

### 问题 9：登录/注册接口路径不够规范

**当前设计**：

```
POST /api/login
POST /api/register
```

**标准做法**：

```
POST /api/auth/login     # 或 /api/sessions
POST /api/users         # 注册 = 创建用户
```

**问题**：不算严重，但 `/api/register` 应该合并到 `POST /api/users`（RESTful 创建资源）。

**修正方向**（可选）：
```
POST /api/auth/login    # 登录
POST /api/users         # 注册（创建用户）
```

## 四、Mock.js 导致的妥协总结

你提到的"考虑到 mockjs 的不便做了很多让步"，主要集中在：

| 妥协点 | Mock.js 限制 | 你的解决方案 | 标准做法 |
|--------|-------------|-------------|---------|
| **Token 传递** | Mock.js 对 Header 拦截支持差 | 放 `body.authorization` | 放 `Authorization` Header |
| **GET 改 POST** | Mock 无法动态读取 Header | 拦截器统一改 method | 保持 GET，Header 传 token |
| **响应格式** | Mock 生成简单格式方便 | `body.code` 判断 | HTTP 状态码判断 |
| **子类型路由** | Mock 对查询参数动态匹配有限 | 用固定路径 `/video/main` | 用查询参数 `?category=main` |

## 五、修正建议优先级

| 优先级 | 修正内容 | 工作量 | 影响范围 |
|--------|---------|--------|---------|
| **P0** | Token 传递改 Header | 中 | 前端拦截器 + 后端中间件 |
| **P0** | 响应格式改 HTTP 状态码 | 中 | 前端拦截器 + 后端统一响应 |
| **P1** | 上传 API 合并到资源创建 | 低 | 改路径即可 |
| **P1** | Profile 路由优化 | 低 | 改路径 |
| **P2** | 分页参数 `element` → `pageSize` | 低 | 全局替换 |
| **P2** | 子类型查询参数化（`/video/main` → `?category=main`） | 中 | 前端 + 后端路由 |
